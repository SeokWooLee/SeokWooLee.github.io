シングルトンのせいでテストが書けず、夜通し悩んだことはありませんか。

ロジック自体はちゃんとしているのに、いざテストを書こうとすると実際のDBを叩き、実際のAPIを呼び出してしまう状況です。

結論から先にお伝えします。

> シングルトンはコードの中でオブジェクトを直接呼び出して使うため、テスト時に偽物のオブジェクトへすり替えることができません。

一方、依存性注入は必要なオブジェクトを外から渡す方式なので、テストで思い通りに差し替えることができます。

今日はこの2つの違いを、実際のコードとともにじっくり見ていきます。

## シングルトンはなぜテストを阻むのか

シングルトンは、プログラム全体でただ1つだけ存在するオブジェクトです。

便利ではあります。どこからでも`Database.shared`の一行で取り出せますから。

問題はここから始まります。

関数の中で直接オブジェクトを呼び出してしまうと、その関数は本物のデータベースにべったり結びついてしまいます。

テスト時に「本物ではなく偽物のDBを使って」と指示する方法がありません。

```swift
// シングルトンを関数内で直接呼び出す
func saveOrder(_ order: Order) {
    // ここで本物のDBに直結してしまいます
    let db = Database.shared
    db.insert(order) // テストでこれを差し替える方法がない
}
```

このコードをテストするには、本物のDBが起動していなければなりません。

DBが落ちていれば、テストも一緒に失敗します。

ロジックが間違っているわけではないのに、インフラのせいで赤信号が灯るわけです。

---

## 依存性注入はどうやるのか

依存性注入は、思ったほど大げさなものではありません。

関数の中でオブジェクトを取り出すのではなく、外で作って渡してあげるだけでいいのです。

文字通り、必要なものを「注入(injection)」するということです。

```swift
// DBを外からパラメータとして受け取る
func saveOrder(_ order: Order, db: Database) {
    db.insert(order) // どのDBを使うかは呼び出し側が決める
}
```

違いはたった一行ですよね。

`Database.shared`で直接呼び出していたものを、パラメータとして受け取るように変えただけです。

しかし、この小さな変化がテストを一変させます。

実際のコードでは本物のDBを渡し、テストでは偽物のDBを渡せばいいのですから。

<figure>
  <img src="/assets/images/posts/bb76dac7-edf6-4e9c-b700-37fe51837a6f/4-1783847645964.png" alt="渡すものを変えただけなのに、これだけ結果が分かれます">
  <figcaption>渡すものを変えただけなのに、これだけ結果が分かれます</figcaption>
</figure>

コンストラクタで受け取る方式もよく使われます。

```swift
class OrderService {
    private let db: Database
    // オブジェクトを生成する際に外から受け取っておく
    init(db: Database) { self.db = db }
}
```

一度受け取ってしまえば、クラス内のどこからでも`self.db`として使えるので、ずっとすっきりします。

## テストで偽物のオブジェクトに差し替える

ここからが本当に面白い部分です。

依存性を外から受け取るように変えたので、テストでは本物のDBの代わりに偽物のDB(一般にモック・Mockと呼ばれます)を渡してあげればいいのです。

```swift
func test_주문을_저장하면_DB에_전달된다() {
    let fake = FakeDatabase() // 本物ではなく偽物
    saveOrder(Order(), db: fake)
    XCTAssertEqual(fake.savedCount, 1) // 保存が呼ばれたかだけ確認
}
```

本物のDBを起動する必要はありません。

ネットワークも外部サーバーも不要です。メモリ内で瞬時に終わります。

おかげでテストが速くなり、何より安定します。

外部の状況に左右されず、ロジックだけをピンポイントで検証できるからです。

## シングルトン vs 依存性注入、一目で比較

両者の違いを表にまとめてみました。

| 区分 | シングルトンを直接使用 | 依存性注入 |
|---|---|---|
| オブジェクトの取得場所 | 関数内で直接取り出す | 外から渡す |
| 偽物オブジェクトへの差し替え | 事実上不可能 | 自由に可能 |
| テスト速度 | 遅い(実際のリソースが必要) | 速い(メモリで処理) |
| テストの安定性 | 外部状況に左右される | ロジックのみ検証可能 |
| 結合度 | 高い | 低い |

シングルトン自体が必ずしも悪いわけではありません。

設定値のように、本当に1つだけあればいい場合には便利です。

ただ「どこでも直接呼び出して使う」という習慣が問題なのです。それがテストの手足を縛ってしまうからです。

## よくある質問(Q&A)

**Q. 依存性注入をするには必ずSwinjectのようなDIライブラリが必要ですか。**

いいえ。今日のコードのように、パラメータやコンストラクタで渡すだけで、それはもう依存性注入です。ライブラリはそれを自動で管理してくれるだけです。

**Q. パラメータが増えすぎると煩雑になりませんか。**

その通りです。そのため通常はコンストラクタ注入でまとめて受け取ったり、関連するものをグルーピングしたりします。パラメータが増え続けるなら、そのクラスがあまりに多くの役割を抱えているというサインでもあります。
