ChatGPTやClaudeを使っていると、もどかしい瞬間が訪れます。モデルは賢いのに、自分のデータベースも見られないし、社内Wikiも読めないからです。

そこでみんなAPIを直接つなぎ始めました。ところがモデルごと、ツールごとに連携方式がバラバラなので、組み合わせが増えるほど連携コードも幾何級数的に膨れ上がります。

MCP(Model Context Protocol)は、この問題を解決するために生まれた標準です。一言でまとめると、AIモデルに外部ツールやデータを差し込むための共通規格です。よく「AI界のUSB-C」と呼ばれます。

この記事では、MCPが正確に何を解決するのか、導入前後で連携作業がどう変わるのか、内部構造やfunction callingとの違い、そして今すぐつないでみたい代表的なサーバーまでまとめます。

まずは要点から見ていきましょう。

1. MCPはAIと外部ツールをつなぐオープン標準プロトコルです(2024年11月にAnthropicが公開)
2. 直接連携すれば数百行だった作業が、MCPサーバーがあれば設定数行で済みます
3. function callingを置き換えるものではなく、その上でツールを標準化・再利用する層です
4. OpenAIやGoogleも採用し事実上の業界標準となり、サーバーのエコシステムが急速に拡大しています

---

## MCPが解決する問題、M×N連携地獄

まずMCP以前の状況を見てみましょう。

AIアプリがM個、つなぎたいツールがN個あるなら、必要な連携コードはM×N個でした。Claude用のGitHub連携、ChatGPT用のGitHub連携、Cursor用のGitHub連携をそれぞれ別々に作らなければならなかったのです。

MCPはこの間に標準規格を挟み込みます。ツール側はMCPサーバーを一度だけ作り、AIアプリ側はMCPクライアントを一度だけ実装すれば終わりです。M×NがM+Nに減ります。

> USB-Cが登場する前、機器ごとに違う充電ケーブルを持ち歩いていた時代を思い浮かべると分かりやすいはずです。MCPはそのケーブル規格を統一したものです。

とはいえ、M+Nという数式だけを見てもいまいちピンとこないと思います。実際に同じ機能を両方の方式で作ってみると、その差がはっきり分かります。

## 同じ機能、導入前後の比較

「GitHubのイシューを調べて答えてくれるAI」を作るとしましょう。

MCP以前は、こうした作業がすべて自分の仕事でした。

1. ツールのスキーマ定義:「list_issuesはowner、repo、stateを受け取る」をJSONスキーマとして自分で作成
2. 実際の呼び出しコードの作成:GitHub REST APIクライアント、トークン管理、ページネーション処理
3. 実行ループの実装:モデルがツールを呼び出すと結果を受け取り、再びモデルに返す往復コード
4. 上記すべてをアプリごとに繰り返す:Claude連携用は別、社内チャットボット用は別

イシュー照会一つにボイラープレートを含め数百行、ツールが増えるたびに1〜3を繰り返すことになります。

MCPがあれば、こうして終わります。

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
    }
  }
}
```

スキーマ定義、API呼び出しコード、認証処理、エラー処理がすべてサーバーの中にすでに入っています。自分はサーバーを登録するだけで、「このリポジトリの最近のバグイシューを要約して」と言えばいいのです。

| 項目 | 直接連携 | MCPサーバー利用 |
| --- | --- | --- |
| 作成コード | スキーマ+呼び出し+ループで数百行 | 設定5行 |
| 認証・エラー処理 | 自前で実装 | サーバーに内蔵 |
| 他のAIアプリへの追加 | 最初からやり直し | 同じサーバーを再利用 |
| ツールの追加 | コード修正・再デプロイ | サーバーの更新のみ |

特に最後の行が重要です。サーバーに機能が追加されても自分側のコードは一行も変わらないこと、これこそが標準化の力です。

## 内部構造、ホスト-クライアント-サーバー

MCPの登場人物は3つです。

| 構成要素 | 役割 | 例 |
| --- | --- | --- |
| ホスト | ユーザーが使うAIアプリ | Claude Desktop、Cursor |
| クライアント | ホスト内でサーバーと1対1で通信 | ホストに内蔵 |
| サーバー | ツール・データを公開するプログラム | GitHubサーバー、DBサーバー |

ホスト1つが複数のクライアントを立ち上げ、クライアント1つがサーバー1つとつながります。通信規格はJSON-RPC 2.0で、接続方式はローカルプロセスならstdio、リモートならHTTPストリーミングを使います。

<figure>
  <img src="/assets/images/posts/45e71f91-f21d-492e-a6da-e0ffc8d4af18/2.png" alt="ホスト内のクライアントがサーバーと1対1でつながる構造です">
  <figcaption>ホスト内のクライアントがサーバーと1対1でつながる構造です</figcaption>
</figure>

サーバーが公開する機能は3種類に分かれます。

- ツール(tools):モデルが呼び出す関数。「イシュー作成」「クエリ実行」のような動作
- リソース(resources):モデルが読み取るデータ。ファイル内容、DBスキーマなどの読み取り資料
- プロンプト(prompts):サーバーがあらかじめ用意したプロンプトテンプレート

重要なのは、ツール一覧を実行時に問い合わせるという点です。クライアントがサーバーに接続すると「あなたは何ができるの?」と尋ね、サーバーがツール一覧とスキーマを返します。先ほど見た「サーバーが更新されても自分のコードはそのまま」が可能になる理由です。

## function callingと何が違うの?

ここまで読むと、こんな疑問が湧きます。「それってfunction callingですでにできていたことじゃないの?」

層が違います。function callingはモデルと対話するための規約です。「モデルよ、こういう関数があるから、必要なら呼び出し形式で答えてくれ」という約束です。関数をどう実装し、どこから持ってくるかは完全に開発者の仕事です。

MCPは、その関数を配布・再利用するための規約です。ツールの実装・認証・ドキュメントをサーバーというパッケージにまとめ、どのAIアプリでも差し込んで使えるようにしたものです。

> function callingが「関数呼び出しの文法」だとすれば、MCPは「関数が詰まったライブラリのエコシステム」です。実際、MCPクライアントは内部的にfunction callingをそのまま使っています。置き換えの関係ではなく、上下の層の関係なのです。

ですから「function callingとMCP、どちらを使うべきか?」という問い自体が成立しません。自分のアプリ一つだけで使う関数が数個であればfunction callingを直接実装すれば十分ですし、複数のアプリで再利用したり他人が作ったツールを使いたいならMCPが向いています。

## 今すぐつないでみたい代表的なサーバー

エコシステムが急速に拡大していて、すでに大抵のツールにはサーバーが登場しています。反応の良いものを5つだけ選びました。

| サーバー | できること |
| --- | --- |
| Playwright | AIがブラウザを直接操作。クリック・入力・スクリーンショット |
| Figma | デザイン案を読み取ってフロントエンドコードに変換 |
| Notion | 文書・議事録の検索と整理、ページ作成 |
| GitHub | イシュー・PRの照会と作成、コード検索 |
| Supabase(Postgres) | 自然言語でDBに問い合わせ、スキーマ把握 |

一つだけ挙げるならPlaywrightです。接続しておいて「このサイトに入って会員登録フォームを埋めて、スクリーンショットを撮って」と言ってみてください。AIがブラウザを開いて実際にクリックしてタイピングするのを見れば、ツールをつなぐということがなぜゲームチェンジャーなのか、あえて説明する必要がなくなります。E2Eテストの自動化や繰り返しの多いWeb作業にすぐ使えます。

Figmaもフロントエンド開発者なら実感が大きいはずです。デザイン案を見てマークアップを手で書き写していた作業が、デザイン案を直接読み取るAIに「このフレームをReactコンポーネントにして」と指示する作業に変わるからです。

## 注意点と限界

万能ではありません。実務で引っかかる点も押さえておきましょう。

まずセキュリティです。MCPサーバーはモデルに実行権限を与える通り道なので、信頼できないサーバーをつなぐとプロンプトインジェクションによってデータが漏れる可能性があります。公式レジストリや検証済みのサーバーだけを使うのが安全です。

コンテキストのコストもあります。サーバーを多くつなぐほどツール定義がコンテキストウィンドウを占領し、かえってモデルの性能が落ちるという逆説が生じます。上の表の中でも、今必要なものだけを選んでつなぐ方がよいでしょう。

## まとめ

MCPは「モデルをより賢く」する技術ではなく、「モデルが手を伸ばせる範囲」を標準化する技術です。直接連携では数百行書いていた作業が設定数行で済むようになったこと、そして他人が作ったツールをそのまま差し込んで使えるエコシステムができたこと。この2つが本質です。
