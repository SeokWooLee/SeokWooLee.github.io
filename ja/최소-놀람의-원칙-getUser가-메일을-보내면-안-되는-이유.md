コードレビュー中に、こんな関数に出くわしたことがあります。

`getUser()`という名前なのに、中を開けてみるとユーザーを取得しながら最終アクセス時刻を更新し、キャッシュを更新し、さらには休眠アカウントだと通知メールまで送っていました。

名前だけ見て「取得だから安全だろう」とループの中で呼び出していたら?ユーザーにメール爆弾が飛んでいたはずです。

今日取り上げる原則は、こういうコードをふるい落とす「驚き最小の原則」です。KISS・DRY・YAGNI・SOLIDと続けてきた開発原則シリーズですが、今回のは名前からして面白いですよね。

驚き最小の原則(Principle of Least Astonishment、略してPOLA)が言っているのはこういうことです。

> コードは読む人が予想した通りに動作すべきである。使う人を驚かせる設計があるなら、その設計を考え直すべきだ。

1960〜70年代のシステム設計から出てきた古い原則ですが、UIデザインやAPI設計、日常のコードにまで全部当てはまる汎用的な原則です。

---

## 驚きがなぜコストになるのか

プログラミングにおける「驚き」は、感情の問題ではなくコストの問題です。

開発者はコードを読むとき、名前や慣習を根拠に仮説を立てながら読み進めます。`getUser`は取得だけするはず、`isValid`はブール値を返すはず、というふうにです。コードが仮説通りに動けば、すらすら読めます。

ところが仮説が崩れた瞬間、読んでいた流れを止めて関数の中に入り、実際の動作を確認しなければなりません。こういう関数がコードベースに10個もあれば「このコードは名前を信用できない」という不信感が根付き、それ以降はすべての関数を開いて確認しながら読むことになります。コードを読む速度が半分になってしまうわけです。

最悪なのは、確認すらせずに仮説のまま使ってしまい、障害が起きるケースです。上のメール爆弾のようにですね。

## コードから驚きが飛び出す場面

実際に私が経験したパターンを中心にまとめてみます。

**1. 名前と振る舞いが違う関数**

```swift
// 名前は取得なのに中で状態を変えてしまう
func getUser(id: Int) -> User {
    let user = db.find(id)
    user.lastSeenAt = Date()  // 驚き1: 副作用
    db.save(user)
    if user.isDormant {
        mailer.send(to: user.email, message: "休眠解除のご案内")  // 驚き2: 外部呼び出し
    }
    return user
}
```

取得関数に隠れている書き込み処理が、驚きの代表格です。getは読み取り専用にとどめ、状態を変えるならupdateやtouchのような名前を使うべきです。

**2. 慣習を裏切る戻り値**

同じコードベースの中で、ある関数は見つからないとnilを返し、別の関数は例外を投げ、また別の関数は空のオブジェクトを返すとしたら、呼び出す側は毎回賭けをすることになります。どちらかに統一することが重要です。

**3. 沈黙する失敗**

```swift
func parseConfig(_ json: Data) -> Config {
    guard let config = try? JSONDecoder().decode(Config.self, from: json) else {
        return Config()  // 驚き: 設定が壊れているのに何事もなかったかのように空の設定を返す
    }
    return config
}
```

設定ファイルが壊れているのに黙ってデフォルト値に戻ってしまうと、ユーザーはしばらく経ってから「なぜ設定が反映されないんだろう?」と悩まされます。失敗は大きく知らせるほうが、結果的に驚きが少なくて済みます。

## 驚きの少ないコードを書くコツ

私が守ろうとしているのは次のようなことです。

| コツ | 内容 |
|------|------|
| 名前は契約である | 名前が約束したことだけを行い、それ以上のことをするなら名前を変える |
| 慣習に従う | 言語・フレームワーク・チームの既存のスタイルを理由なく外れない |
| 副作用は隠さない | 状態を変える関数は、名前からそれとわかるようにする |
| 驚かせるなら文書で | やむを得ず特殊な動作が必要なら、コメントとドキュメントで大きく知らせる |

この中で一番強力なのは2番目です。自分だけの気の利いたやり方より、退屈な標準のやり方のほうが、チーム全体で見れば常に勝ちます。前回のKISS編で「かっこいい一行」より普通のループのほうがいいと書いたのと、まさに同じ文脈です。

## おわりに

驚き最小の原則で覚えておきたいのは次の3つです。

- コードは名前や慣習が作る期待通りに動作すべきです。期待を裏切るコードがバグの温床になります。
- 取得関数に副作用を隠さず、失敗を黙って飲み込まないでください。
- 奇抜なコードより、退屈で予測可能なコードのほうがチームでは勝ちます。

良いコードの基準を一つだけ挙げろと言われたら、私はこれを挙げます。読む人を驚かせないコード。

最近コードを読んでいて「え、なんでこんな動き方するの?」と思った瞬間があったなら、それがまさにこの原則が破られている現場です。
