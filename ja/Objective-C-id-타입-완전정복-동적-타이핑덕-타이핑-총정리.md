Objective-Cを学び始めると、`id`という型の前で一度は立ち止まってしまいます。

「これは一体何の型なんだろう?なんでどんなオブジェクトを入れてもコンパイルが通るんだろう?」という疑問が湧いてきますよね。

結論から言うと、`id`は「どんなオブジェクトでも指せる万能ポインタ」であり、これこそがObjective-Cの動的タイピングとダックタイピングを支える核心です。

この記事では、`id`型が正確には何なのか、動的タイピングとダックタイピングが実際のコードでどう動くのか、そしてこれをいつ使い、いつ注意すべきかまで一つずつ整理してお伝えします。

## id型とは一体何なのか

一言で言うと`id`は「任意のObjective-Cオブジェクトを指すポインタ」です。

`NSString *`や`NSArray *`のように具体的なクラスを指定しなくても、どんなオブジェクトでも入れることができます。

実は`id`の正体は思ったよりシンプルです。内部的には次のように定義されています。

```objc
// objc.h 内部定義(要約)
typedef struct objc_object {
    Class isa;  // このオブジェクトがどのクラスかを指すポインタ
} *id;
```

つまり`id`は`isa`というポインタを一つ持つ構造体へのポインタなのです。

<figure>
  <img src="/assets/images/posts/985ef22b-6988-47cb-adf2-a9d803086c51/4-1783847906422.png" alt="idの正体、結局はisaを一つ持つポインタでした">
  <figcaption>idの正体、結局はisaを一つ持つポインタでした</figcaption>
</figure>

この`isa`が「自分はどのクラスのインスタンスなのか」を実行時に教えてくれる名札のような役割を果たします。

ここで面白い点が一つ。`id`はすでにポインタなので、`id obj`と書き、`id *obj`のようにアスタリスクは付けません。

`NSString *str`と比べると混同しやすい部分です。

## 動的タイピングはこう動く

動的タイピング(dynamic typing)とは、「オブジェクトの実際の型をコンパイル時ではなく実行時に判断する」という概念です。

`id`に入っているオブジェクトが実際に何のクラスなのかは、プログラムが動いている瞬間に`isa`ポインタを見て決まります。

そのため、こんなコードが可能になります。

```objc
id obj = @"Hello";           // 今はNSString
NSLog(@"%@", [obj class]);   // 出力: __NSCFConstantString
obj = @[@1, @2, @3];         // 今度はNSArrayに変わる
NSLog(@"%@", [obj class]);   // 出力: __NSArrayI
```

同じ変数`obj`が文字列だったのが配列に変わります。

コンパイラはこれを止めません。実際の型判断を実行時に先送りするからです。

メッセージを送るときも同様です。`[obj length]`を呼び出すと、コンパイル時点では`obj`が`length`に応答できるかどうかはわかりません。

ランタイムがその瞬間に`obj`のクラスを調べて該当メソッドを探し出し実行します。これをメッセージディスパッチと呼びます。

両者を比較するとこうなります。

- 静的タイピング: コンパイラが型を事前に確定して検査する
- 動的タイピング: 実行中に実際のオブジェクトを見て決定する

> Objective-Cにおける`id`と動的タイピングは、「とりあえず入れておいて、本当は何なのかは実行時に聞く」という哲学の産物です。

## ダックタイピングの本当の意味とは

ここで多くの方が誤解しているポイントがあります。ダックタイピング(duck typing)を単に「型検査をしないこと」とひとまとめにしてしまうケースです。

ダックタイピングの元々の表現はこうです。

「アヒルのように歩き、アヒルのように鳴くなら、それはアヒルだ」

オブジェクトがどのクラスに属しているかが重要なのではなく、「そのメッセージに応答できるかどうか」が重要だという意味です。

Objective-Cではこれを`respondsToSelector:`で直接確認できます。

```objc
// クラスが何であろうと、このメソッドに応答さえすれば呼び出す
if ([obj respondsToSelector:@selector(quack)]) {
    [obj quack];  // アヒルのように鳴けるならアヒルとして扱う
}
```

ここで`obj`が`Duck`クラスなのか`Robot`クラスなのかは何の関係もありません。

`quack`というメッセージに応答さえすれば、私たちのコードにとっては「アヒル」なのです。

これは継承ベースのポリモーフィズムとは性質が異なります。

同じ親クラスを継承していなくても、互いに何の関係もないクラスであっても、同じメソッド名さえ持っていれば同じように扱えるからです。

まさにこの柔軟さが、デリゲートパターンやターゲット・アクションといったCocoaの中核設計を可能にした土台なのです。

## 便利な分だけ危険なid、いつ注意すべきか

`id`は自由な分だけ代償が伴います。

コンパイラがほとんど型検査をしてくれないため、応答できないメッセージを送るとコンパイルは通ってしまい、実行中にアプリが落ちてしまいます。

まさにあの有名な`unrecognized selector sent to instance`クラッシュです。

そのため実務ではこう使います。

- 型が確実な場合は`id`の代わりに具体的なクラス(`NSString *`など)を明示する
- どうしても`id`を使う必要があるなら、呼び出し前に`respondsToSelector:`で防御する
- デリゲートプロトコルを定義して「このメッセージには応答する」ことを文書化する

最近では`instancetype`やジェネリクス(`NSArray<NSString *> *`)が導入され、無分別な`id`の使用はかなり減りました。

それでもフレームワークの深部やランタイムを扱うコードでは、依然として`id`が心臓のように動いています。

まとめると、`id`はObjective-Cの動的な性格を象徴する型であり、動的タイピングとダックタイピングは「クラスではなく振る舞いでオブジェクトを判断する」という一つの哲学でつながっています。

最初は見慣れず危険に見えても、この原理を理解すればCocoaフレームワークの設計がずっとはっきり見えてくるはずです。
