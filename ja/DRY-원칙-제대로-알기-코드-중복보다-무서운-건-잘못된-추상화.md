コードレビューをしていると、やけによく出くわす場面があります。

機能を一つ修正しようとして検索したら、ほぼ同じコードが3か所から出てくるんです。一つだけ直すと、残りの二つはそのままバグとして残ってしまうんですね。

今日の主役は、この問題を正面から狙い撃ちする原則、DRYです。前回扱ったKISS原則と、まるで相棒のようにセットで語られる存在です。

DRYは「Don't Repeat Yourself」の略です。同じ知識をシステムの中で繰り返すな、という原則ですね。

1999年に出た書籍『達人プログラマー』でアンディ・ハントとデイブ・トーマスがまとめた概念で、原文の定義がかなり的確です。

> すべての知識は、システムの中でただ一つの、曖昧さのない、権威ある表現を持たなければならない。

ここで注目すべき単語は「コード」ではなく「知識」です。これが今日の話の核心です。

---

## コードのコピペはなぜ問題なのか?

コピー&ペースト自体が罪というわけではありません。問題はその後に起こります。

同じロジックが3か所にあると、要件が変わるたびに3か所すべてを探して直す必要があります。一つでも見落とせば、それがバグになります。

```swift
// 配送料計算が3つのファイルに散らばっている場合
// Cart.swift
let fee = total >= 50000 ? 0 : 3000

// Checkout.swift
let shippingFee = totalPrice >= 50000 ? 0 : 3000

// OrderSummary.swift
let delivery = price >= 50000 ? 0 : 3000
```

ある日「送料無料の基準を3万ウォンに下げよう」という要望が来たら? 3つのファイルを全部探さなければなりません。検索ワードもそれぞれ違うので、一つくらいは必ず見落とします。

```swift
// 知識を一か所にまとめると
// Shipping.swift
enum ShippingPolicy {
    static let freeShippingThreshold = 50000
    static let fee = 3000

    static func shippingFee(for total: Int) -> Int {
        total >= freeShippingThreshold ? 0 : fee
    }
}
```

これでポリシーが変わっても、直すのは一か所だけで済みます。これがDRYです。

---

## コードではなく「知識」の重複です

ところが、ここで多くの人が一度つまずきます。DRYを「似た見た目のコードは全部まとめろ」と読んでしまうんです。

DRYが言う重複は、コードの見た目ではなく知識、つまりビジネスルールの重複を指します。

この区別がなぜ重要かというと、世の中には偶然似ているだけのコードもあるからです。

| 状況 | 重複か? |
|------|-----------|
| 配送料計算ロジックが3か所 | 本物の重複(同じ知識) |
| 会員登録の検証とイベント応募の検証がたまたま似ている | 偽の重複(異なる知識) |
| 定数3000が配送料とポイント積立の基準にそれぞれ登場 | 偽の重複(異なる意味) |

会員登録の検証とイベント応募の検証が、今は両方とも「名前必須、電話番号11桁」でコードが同じに見えても、この二つのルールは異なる理由で変わっていきます。これを一つにまとめてしまうと、イベントルールを変更した拍子に会員登録が壊れてしまいます。

## 早すぎる抽象化は重複より高くつきます

そのため最近のコミュニティでも、こんな言葉がよく語られます。「間違った抽象化より重複の方がまし(prefer duplication over the wrong abstraction)」だと。有名なRuby開発者、サンディ・メッツの言葉です。

似ているように見える二つのコードを無理やり一つにまとめると、こんなことが起こります。

1. 共通関数を作る
2. 片方の要件が変わってオプションパラメータを追加する
3. もう片方も変わってif分岐を追加する
4. いつの間にかパラメータ5個の、誰も理解できない関数になる

ここまで来ると、むしろコピペ二枚のままの方がましだったわけです。

<figure>
  <img src="/assets/images/posts/52ab8398-a81d-4f14-b59a-89c82327e577/4-1783847956412.png" alt="無理やりまとめた関数が崩れていく4段階">
  <figcaption>無理やりまとめた関数が崩れていく4段階</figcaption>
</figure>

そのため実務では「3の法則(Rule of Three)」がよく使われます。同じパターンが2回出てきたら一旦様子見、3回目に出てきたときに抽象化する、というものです。3回ほど繰り返されると、それが本当に同じ知識なのか、それとも偶然なのかが見えてきます。

## 私が使っている判断基準

重複を見つけたら、私はこの質問を一つ投げかけます。

> このコード二つは、同じ理由で変わるか?

同じ理由で変わるなら本物の重複なのでまとめます。異なる理由で変わるなら、見た目がどれだけ似ていてもそのままにしておきます。

## 今日の結論

DRY原則、要点だけまとめるとこうなります。

- 重複の単位は、コードの見た目ではなく知識(ビジネスルール)です。
- 同じ理由で変わるコードだけをまとめましょう。偶然似ているだけのコードはそのままにしておきましょう。
- 確信が持てないなら3の法則。3回目の繰り返しで抽象化しても遅くありません。

KISSが「シンプルに保て」だとすれば、DRYは「知識は一か所にだけ置け」です。どちらも結局は、変更コストを減らそうという話なんですね。

検索一発で3か所引っかかるロジック、一つくらい思い浮かぶのではないでしょうか。それが今日のリファクタリング候補です。
