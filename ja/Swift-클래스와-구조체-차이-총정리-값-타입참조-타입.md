Swiftで開発していると、必ず一度は手が止まる瞬間があります。

「これはclassで作るべきか、structで作るべきか」

最初にSwiftを学ぶときは、この2つをなんとなくの気分で選んでしまいがちです。そして、思わぬバグに出くわしてはじめて、その違いをしっかり学ぶことになります。

結論から言います。

> 特別な理由がなければ、まず構造体(struct)を選びましょう。
>
> クラスは「共有される一つの状態」がどうしても必要なときにだけ使います。

Apple公式のガイドラインも同じ立場です。今日はその理由を一つずつ解説していきます。

## まずは要点から(3つ)

忙しい方のために先にまとめました。

1. **構造体は値型**です。コピーすると完全に別の値になります。
2. **クラスは参照型**です。コピーしても同じインスタンスを一緒に指し続けます。
3. 迷ったら**まずstruct**、継承や状態の共有が必要なときだけclass。

この3行だけ覚えておけば、もう半分は終わったようなものです。

---

## 値型と参照型、何が違うのか

言葉だけではピンとこないですよね。コードで見るのが一番早いです。

まずは構造体です。コピーした瞬間、お互い無関係な存在になります。

```swift
struct Point { var x = 0 }

var a = Point()
var b = a      // 値がコピーされる
b.x = 10

print(a.x, b.x)
// 出力: 0 10
```

`b`を変更しても`a`はそのままです。それぞれが自分の値を持っているからです。

次はクラスです。同じようなコードに見えますが、結果は異なります。

```swift
class Box { var x = 0 }

let a = Box()
let b = a      // 参照がコピーされる(同じオブジェクト)
b.x = 10

print(a.x, b.x)
// 出力: 10 10
```

`b`だけを変更したのに、`a`まで10になってしまいました。

両者が同じインスタンスを指しているからです。ありがちなバグの原因は、まさにこれです。

## では、どちらをいつ使うのか(選択基準)

実務で判断するとき、頭の中に思い浮かべるのがこの表です。

| 状況 | 判断 |
|------|------|
| 単純にデータを保持するモデル | 構造体 |
| コピーされてもお互い独立していてほしい | 構造体 |
| 継承がどうしても必要 | クラス |
| 複数の箇所で同じインスタンスを共有する | クラス |
| Objective-Cと連携する必要がある | クラス |

もう少し詳しく整理すると、こうなります。

- 座標、色、サイズのように**値そのもの**を表すなら、構造体が自然です。
- 画面のコントローラーやネットワークマネージャーのように**一つの状態を複数の箇所が一緒に**見る必要があるなら、クラスが適しています。
- AppleのSwiftUIがViewをstructで作っているのも同じ理由です。軽量で予測しやすいからです。

ちなみに、Swift標準ライブラリの`Array`、`String`、`Dictionary`はすべて構造体です。それだけ値型が基本になっているということです。

## 構造体をまず勧める理由

値型は予測がしやすいです。

自分が渡した値が、他の場所でこっそり書き換えられる心配がありません。関数に渡しても、元の値は安全です。

マルチスレッド環境でも安心です。共有される状態がないぶん、データ競合が減るからです。

一方でクラスは強力ですが、慎重に扱う必要があります。あちこちで同じインスタンスを持っていると、値がどこで変わったのか追跡するのが難しくなります。

だからこそ「基本はstruct、必要なときだけclass」という習慣が、バグを大きく減らしてくれます。

---

## いつ使い、いつ避けるべきか

最後に判断基準を短くまとめます。

- **構造体を使う**: データ中心のモデル、独立したコピーが必要なとき、スレッドセーフが重要なとき
- **クラスを使う**: 継承が必要なとき、一つのインスタンスを共有する必要があるとき、識別子(identity)に意味があるとき
- **避けるべき**: 単純なモデルなのに習慣的にclassを使うこと、共有が必要なのにstructで無理やり回避すること

---

### 面接ではこう聞かれます

**Q. 構造体とクラスの一番大きな違いは何ですか?**

構造体は値型、クラスは参照型です。構造体はコピーすると完全に独立した値になりますが、クラスはコピーしても同じインスタンスを指し続けます。また、クラスだけが継承可能で、デイニシャライザ(deinit)を持つことができます。

**Q. Swiftはなぜ構造体の使用を推奨しているのですか?**

値型は共有状態がないため副作用が少なく、予測しやすいからです。元の値が意図せず書き換えられるリスクがなく、マルチスレッド環境でも安全なため、バグを減らすうえで有利です。
