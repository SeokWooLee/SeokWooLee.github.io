決済方法ごとにif-elseがどこまでも増えていくコード、経験したことはありませんか?

ソートのオプション、割引計算、フィルタのロジックが一つの巨大なswitch文の中で絡み合い始めると、触るのがだんだん怖くなってきますよね。

そんなときに取り出すべきツールが、まさにSwiftのストラテジーパターンです。

今日はプロトコルとクロージャを使って、アルゴリズムを部品のように差し替える方法を実務目線で解説していきます。

</figure>

まず核心から押さえていきましょう。

> ストラテジーパターンとは、「何をするか」と「どうやるか」を分離し、アルゴリズムを外部から差し替えられるようにする設計です。

Swiftでは、これをプロトコル(重厚な方式)またはクロージャ(軽量な方式)の2通りで実装できます。

以下で一つずつ見ていきましょう。

---

## ストラテジーパターンって何?3行でまとめると

複雑な理論より先に、まず骨組みを押さえておきましょう。

1. 差し替え可能なアルゴリズム群を一つの共通仕様(プロトコル)としてまとめる
2. 実際の処理はそれぞれのストラテジーオブジェクトやクロージャとして作る
3. 使う側は仕様だけを知っていればよく、具体的な中身は知らなくてもいい

例えるなら、ゲームキャラクターの武器交換のようなものです。

キャラクターは「攻撃する」ということだけ知っていればよく、剣であれ弓であれ魔法であれ、装備している武器が勝手に処理してくれます。

キャラクターのコードには手を加えず、武器だけを差し替えるわけです。

この「差し替え」がストラテジーパターンのすべてと言っても過言ではありません。

---

## プロトコルで実装する(王道の方式)

まずは最も教科書的なプロトコル方式から見ていきましょう。

割引計算を例にとってみます。一般会員、VIP会員、クーポン適用など、計算方法が複数あるケースです。

以下は、すべてのストラテジーが従うべき共通仕様と、実際のストラテジーを一つ定義したコードです。

```swift
// すべての割引ストラテジーが従うべき共通仕様
protocol DiscountStrategy {
    func discount(for price: Int) -> Int
}

// VIPストラテジー: 20%割引
struct VIPDiscount: DiscountStrategy {
    func discount(for price: Int) -> Int { price * 20 / 100 }
}
```

次に、このストラテジーを保持して使う側を見てみましょう。

使う側のオブジェクトは、どのストラテジーが渡されても気にしません。

<figure>
  <img src="/assets/images/posts/6191441f-b80b-4db0-8367-ca763cbfb2ce/3-1783847844763.png" alt="Checkoutは仕様だけを知っていて、中身は知りません">
  <figcaption>Checkoutは仕様だけを知っていて、中身は知りません</figcaption>
</figure>

```swift
struct Checkout {
    var strategy: DiscountStrategy   // ストラテジーを差し替えられる場所

    func finalPrice(_ price: Int) -> Int {
        price - strategy.discount(for: price)
    }
}

// 決済の瞬間にストラテジーだけを差し替える
let cart = Checkout(strategy: VIPDiscount())
```

新しい割引ポリシーが生まれても、`Checkout`には一切手を加えません。

`DiscountStrategy`に準拠した構造体を一つ新しく作るだけで済みます。

既存のコードを触らずに機能を拡張できること、これが実務で最も実感できるメリットでした。

---

## クロージャで実装する(Swiftらしい方式)

とはいえ、ストラテジーが単純な場合まで毎回構造体を作るのは少し面倒ですよね。

単純なロジックまですべてプロトコルで包んでしまうと、かえってコードが冗長になりがちです。

そんなときSwiftでは、クロージャの方がはるかに軽快で自然です。

```swift
struct Checkout {
    // ストラテジー自体を関数として受け取る
    var discount: (Int) -> Int

    func finalPrice(_ price: Int) -> Int {
        price - discount(price)
    }
}

// ストラテジーをその場で即席定義する
let cart = Checkout(discount: { $0 * 20 / 100 })
```

別途型を宣言することなく、渡すその場でロジックをそのまま書き込めます。

ソート基準、フィルタ条件、簡単な変換のような軽いアルゴリズムの差し替えには、この方式のほうがずっとすっきりします。

<figure>
  <img src="/assets/images/posts/6191441f-b80b-4db0-8367-ca763cbfb2ce/2.png" alt="sorted(by:)も、よく見れば私たちが毎日使っていたストラテジーパターンだったんですね">
  <figcaption>sorted(by:)も、よく見れば私たちが毎日使っていたストラテジーパターンだったんですね</figcaption>
</figure>

実は標準ライブラリの`sorted(by:)`がまさにこのクロージャ版ストラテジーパターンです。

ソートという骨組みは固定したまま、比較基準だけをクロージャで差し替える構造ですから。

私たちはすでに毎日これを使っていたわけです。

---

## プロトコルとクロージャ、いつどちらを使う?

実務で使い分けている基準を表にまとめてみました。

| 区分 | プロトコル方式 | クロージャ方式 |
|------|--------------|------------|
| 適した状況 | ストラテジーが複雑・状態を保持する | ストラテジーが短く単純 |
| 再利用性 | 複数箇所での再利用がしやすい | 一回限りの用途に有利 |
| コード量 | 型宣言が必要、多くなる | 即席定義、少ない |
| テスト | 個別の型として検証しやすい | ロジックだけを簡単に検証 |
| 可読性 | 名前で意図が明確になる | 短ければ明確、長いと読みにくい |

私の基準はこうです。

ストラテジーが内部状態を持つ場合、複数の画面で再利用される場合、あるいは名前で意図を明確に示す必要がある場合は、プロトコルが向いています。

逆に、一行二行のロジックをその都度差し替えるだけなら、クロージャが正解です。

もちろん両方を組み合わせて使っても全く問題ありません。私は大きなポリシーはプロトコルで、細かいオプションはクロージャでと分けて使っています。

## よくある質問(Q&A)

**Q. ストラテジーパターンと継承(オーバーライド)は何が違うのですか?**

継承は親クラスに縛られ、コンパイル時点で動作が固定されます。

一方ストラテジーパターンは、実行中にストラテジーを自由に差し替えることができ、クラス階層に縛られません。

**Q. enumのswitch文で処理するのとは何が違うのですか?**

switchは新しいケースが増えるたびに、既存のコードを開いて修正する必要があります。

ストラテジーパターンは新しいストラテジーを「追加」するだけで済むので、既存のコードに手を加える機会が減ります。

---

結局、ストラテジーパターンの核心は一つです。変わる部分を外に出して、差し替えやすくすること。

今日学んだプロトコルとクロージャ、この2つの方式を状況に応じて使い分ければ、あのうんざりするif-else地獄からきっと抜け出せるはずです。小さなコードから一度リファクタリングしてみることをおすすめします!
