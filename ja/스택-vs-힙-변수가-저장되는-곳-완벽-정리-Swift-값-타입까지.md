変数を宣言すると、その値はどこに保存されるのでしょうか。

「メモリです」と答えたら半分正解で半分足りません。メモリの中でもスタックに置かれる値とヒープに置かれる値では、運命がまったく違うからです。

スタックオーバーフローというエラー名、クラスと構造体の性能差、ARCがなぜ必要なのか——これらはすべてこの2つの領域を理解して初めて説明がつきます。

この記事では、スタックとヒープがそれぞれどう動作するのか、なぜ速度差が生まれるのか、Swiftでは何がどこに置かれるのかを整理します。

まず要点から見ていきましょう。

1. スタック:関数呼び出しとともに積まれ、リターンとともに消える自動管理領域。高速
2. ヒープ:実行時に必要なサイズを借りて使う動的領域。柔軟だが管理コストがかかる
3. 速度差の本質は「確保・解放の方式」と「後片付けを誰がするか」
4. Swiftでは構造体はおおむねスタック、クラスインスタンスはヒープに置かれる

---

## スタック:積み重ねたお皿

スタックは関数呼び出しの情報を積み上げる領域です。名前の通り、お皿を積み重ねるイメージです。最後に置いた皿を最初に取り出すLIFO(後入れ先出し)構造になっています。

関数を呼び出すと、その関数のローカル変数、引数、リターンアドレスが一塊(スタックフレーム)として積まれます。関数がリターンすると、そのフレームがまるごと消えます。

ここにスタックの強みがあります。

**確保が高速です**。スタックポインタを上にずらすだけで完了します。空き領域を探し回る必要がありません。

**解放もタダです**。関数が終わればポインタを元に戻すだけ。誰が片付けるべきか悩む必要がありません。

その代わり制約もあります。サイズはコンパイル時に決まっていなければならず、関数が終わると問答無用で消え、全体のサイズ上限も小さめです(通常は数MB程度)。再帰関数が終了条件なしに自分自身を呼び続けると、フレームが上限を超えてしまいます。そのエラーがまさにスタックオーバーフローです。

---

## ヒープ:広いが管理が必要な倉庫

ヒープは実行時に必要な分だけ借りて使う領域です。サイズを事前に知らなくてもよく、関数が終わっても生き残り、容量もずっと大きいです。

その代わり管理コストがついて回ります。

**確保は相対的に遅くなります**。要求されたサイズに合う空き領域を探す必要があり、複数のスレッドが同時に確保を要求すると調整も必要になります。

**解放は誰かが責任を持つ必要があります**。関数が終わっても消えないので、「もう使わない」を誰かが判断しなければなりません。Cではプログラマーが直接freeを呼び出し、Javaではガベージコレクタが定期的に掃除し、Swiftでは参照カウントを数えて0になった瞬間に解放するARCが担当します。

この判断を誤ると2種類の事故が起きます。解放が遅すぎればメモリリーク、早すぎれば解放済みメモリへのアクセス(ダングリングポインタ)です。

## Swiftでは何がどこに置かれるのか

Swiftの公式ドキュメントやWWDCセッションで繰り返し強調されている構図があります。値型(構造体・列挙型)はスタックに、参照型(クラス)はヒープに割り当てられるのが基本というものです。

```swift
struct PointStruct { var x, y: Double }
class PointClass { var x = 0.0, y = 0.0 }

func run() {
    let a = PointStruct(x: 1, y: 2) // スタックフレームの中にまるごと
    let b = PointClass()            // ヒープに割り当て、スタックには参照のみ
}
```

構造体aはスタックフレームに値がまるごと入ります。関数が終わればフレームと一緒に消えるので、参照カウントも不要です。

クラスインスタンスbはヒープに置かれ、スタックにはそのアドレスだけが置かれます。複数の場所から参照される可能性があるため、ARCがカウントを数える必要があります。

Appleが「構造体を優先的に検討せよ」と勧める性能面の根拠がこれです。ヒープ確保、参照カウント、ロックのコストがまるごと不要になるからです。

ただし「構造体=必ずスタック」というわけではありません。クラスのプロパティになった構造体は、そのクラスと一緒にヒープに置かれますし、StringやArrayのような型は見た目こそ構造体ですが、実際のデータバッファはヒープに持っています。「構造体はスタックに置かれる条件を備えている」程度に理解するのが正確です。

## 面接でよく聞かれる深掘り質問

**「なぜスタックはヒープより速いのですか?」**——スタックはポインタの移動だけで確保・解放が完了しますが、ヒープは空き領域の探索と解放管理(参照カウント・GC)が必要だからです。

**「スタックオーバーフローはなぜ起きるのですか?」**——スタックのサイズ上限は小さく、深い再帰や巨大なローカル変数によってフレームが上限を超えると発生します。

**「ローカル変数は必ずスタックに置かれるのですか?」**——いいえ。ローカル変数であっても参照型であれば、本体はヒープにあります。スタックに置かれるのは参照だけです。

## まとめ

- スタックは関数呼び出し単位で自動的に積まれ、消える領域。ポインタの移動だけで管理されるため高速
- ヒープは実行時にサイズを決めて借りる領域。柔軟だが確保時の探索と解放管理にコストがかかる
- 解放の責任:スタックは自動、ヒープは言語によって異なる(Cは手動、JavaはGC、SwiftはARC)
- スタックオーバーフローは深い再帰などでスタックの上限を超えたときに起きるエラー
- Swiftでは構造体はおおむねスタック、クラスはヒープ。これが値型を優先する性能上の根拠
- ただし、クラスの中の構造体はヒープに置かれ、String・Arrayは内部バッファがヒープにある
