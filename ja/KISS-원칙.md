開発をしていると、なぜかコードを難しく組んだほうが「デキる」気分になる瞬間ってありますよね。

デザインパターンをてんこ盛りにして、抽象化レイヤーを重ね、「あとで拡張するかもしれないから」とインターフェースを先回りして用意したりして。

でも数年経ってそのコードを開き直すと、結局苦労するのは未来の自分だったりします。

今日はその対極にある原則、KISSについて話してみます。

結論から言うと、KISSは「Keep It Simple, Stupid」の略です。直訳すると「シンプルにしろ、バカ」くらいのニュアンスですね。

システムは複雑にするときではなく、シンプルに保つときにこそ最もうまく動く、という設計思想です。

---

## KISSの原則、そもそもどこから来た言葉?

この言葉、実はもともとソフトウェア用語ではありませんでした。

1960年代、アメリカのロッキード社の航空エンジニア、ケリー・ジョンソン(Kelly Johnson)が生み出した言葉として知られています。あの有名なU-2、SR-71ブラックバード偵察機を設計した人物です。

ケリー・ジョンソンがチームに求めたのはこういうことだったそうです。

> 戦闘状況下で、平凡な整備士でも基本的な工具だけで直せる飛行機を作れ。

どれだけ優れた設計でも、現場で直せなければ意味がない、というわけです。

この精神がソフトウェアの世界にも持ち込まれました。どれだけ巧妙なコードでも、同僚が読んで直せなければ良いコードとは言えない、という話です。

ちなみに「Stupid」は開発者をバカ呼ばわりしているのではなく、「バカでも理解できるくらいシンプルに」というニュアンスに近いです。

---

## コードでKISSが崩れる瞬間

言葉で言うのは簡単ですが、実際のコードではどんな姿になるのでしょうか。よく見かけるパターンをいくつか紹介します。

**1. かっこよく見える一行**

```swift
// こう書くと賢く見えますが
let result = arr.reduce([Int: Item]()) { $0.merging([$1.id: $1]) { _, new in new } }

// こちらの方がずっと読みやすいです
var result = [Int: Item]()
for item in arr {
    result[item.id] = item
}
```

上のコードは短いですが、初めて見る人はしばらくじっと眺める必要があります。下のコードは誰が見ても3秒で理解できます。

**2. 使いもしない拡張性**

ボタン一つ作るのに、ファクトリーパターンからストラテジーパターンまで総動員するケースです。「あとでボタンの種類が増えたら?」という心配からですが、その「あとで」はたいてい来ません。

**3. 条件分岐の迷路**

ifの中にif、その中にまたif。3段階もネストすれば頭の中のスタックがパンクします。こういうときはearly returnで平坦にするだけでも、コードはぐっとシンプルになります。

## KISSを守る実践基準3つ

そこで私はコードを書くとき、この3つを基準にしています。

| 基準 | 質問 |
|------|------|
| 説明テスト | このコードを隣の席の同僚に1分以内で説明できるか? |
| 未来テスト | 6か月後の自分が見てもすぐ理解できるか? |
| 必要テスト | 今まさに必要だから入れたコードか、いつか使うかもしれないから入れたコードか? |

この3つのうち一つでも「いいえ」なら、もっとシンプルな方法がないか考え直します。

特に3つ目が重要で、これはYAGNI(You Aren't Gonna Need It)の原則にも通じる部分です。KISSとYAGNIは事実上セットで動きます。

## シンプルなものと雑に作ったものは違います

ここで誤解してはいけないことが一つあります。

KISSは「何も考えずに書け」という意味ではありません。むしろ逆です。

複雑にするのは簡単です。思いついたものを全部くっつければいいだけですから。シンプルにするのは難しい。何を削るか、真剣に悩まなければならないからです。

パスカルが手紙に残した有名な言葉があります。

> もっと短く書く時間がなかったので、長く書いた。

コードも同じです。シンプルなコードは雑に作った結果ではなく、複雑さを取り除く努力の結果です。

## まとめ

KISSの原則、まとめるとこうなります。

- コードは書く時間よりも読まれる時間の方がずっと長いです。読む人のためにシンプルに書きましょう。
- 今必要のない拡張性は入れないでください。その心配が現実になることは、思っているより稀です。
- シンプルさは実力不足ではなく、実力の証です。

私は最近、コードレビューで「これ、もっとシンプルにできませんか?」というコメントを一番よく残している気がします。不思議なことに、この一言だけでバグも減り、レビュー時間も短くなるんですよね。

皆さんのコードは、隣の席の同僚が1分以内に理解できるものでしょうか。今日一度、自分に問いかけてみてはいかがでしょうか。
