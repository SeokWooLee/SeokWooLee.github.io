開発をしていると、なぜかコードを難しく書いたほうが「できる人」っぽい気がすることがあります。

デザインパターンをあれこれ盛り込み、抽象化レイヤーを重ね、「あとで拡張するかもしれないから」とインターフェースをあらかじめ切っておいたり。

私も新人の頃はそうでした。でも数年経ってそのコードを開くと、結局苦労するのは未来の自分なんですよね。

今日はその対極にある原則、KISSについて話してみます。

結論から言うと、KISSは「Keep It Simple, Stupid」の略です。直訳すると「シンプルにしろ、バカ」くらいの意味になりますが。

システムは複雑にすればするほど良くなるわけではなく、シンプルに保ってこそ最もうまく動く、という設計思想です。

---

## KISS原則、どこから生まれた言葉?

この言葉、もともとソフトウェア用語ではありませんでした。

1960年代、アメリカのロッキード社の航空エンジニア、ケリー・ジョンソン(Kelly Johnson)が作った言葉として知られています。あの有名なU-2、SR-71ブラックバード偵察機を設計した人物です。

ケリー・ジョンソンがチームに求めたのはこういうことだったそうです。

> 戦闘状況下で、普通の整備士が基本工具だけで修理できる飛行機を作れ。

どれだけ優れた設計でも、現場で直せなければ意味がない、ということです。

この精神がソフトウェアにも引き継がれました。どれだけ気の利いたコードでも、同僚が読んで直せなければ良いコードとは言えない、という話です。

ちなみに「Stupid」は開発者をバカ呼ばわりしているわけではなく、「バカでも理解できるくらいシンプルに」というニュアンスに近いです。

---

## コードでKISSが崩れる瞬間

言葉にすると簡単ですが、実際のコードではどんな形になるでしょうか。私がよく見てきた(そしてやらかしてきた)パターンをいくつか紹介します。

**1. かっこよく見える一行**

```javascript
// こう書くと賢く見えますが
const result = arr.reduce((a, c) => ({ ...a, [c.id]: c }), {});

// こちらのほうがずっと読みやすいです
const result = {};
for (const item of arr) {
  result[item.id] = item;
}
```

上のコードは短いですが、初めて見る人はじっくり眺めないと理解できません。下のコードは誰が見ても3秒で分かります。

**2. 使いもしない拡張性**

ボタン一つ作るのに、ファクトリーパターンからストラテジーパターンまで動員するケースです。「あとでボタンの種類が増えたら?」という心配からくるものですが、その「あとで」はたいてい来ません。

**3. 条件分岐の迷路**

ifの中にif、その中にまたif。3段階もネストされれば、頭の中のスタックが破裂します。こういうときはearly returnで平らに展開するだけでも、コードがぐっとシンプルになります。

## KISSを守るための実践基準3つ

そこで私はコードを書くとき、この3つを基準にしています。

| 基準 | 問い |
|------|------|
| 説明テスト | このコードを隣の席の同僚に1分以内で説明できるか? |
| 未来テスト | 半年後の自分が見てもすぐ理解できるか? |
| 必要テスト | 今すぐ必要だから入れたコードか、いつか使いそうだから入れたコードか? |

この3つのうち1つでも「いいえ」が出たら、もっとシンプルな方法がないか考え直します。

特に3つ目が重要で、これはYAGNI(You Aren't Gonna Need It)原則にも通じる部分です。KISSとYAGNIは事実上セットで動きます。

## シンプルなものと雑に作ったものは違います

ここで誤解してはいけないことが一つあります。

KISSは「何も考えずに書け」という意味ではありません。むしろ逆です。

複雑にするのは簡単です。思いついたものを全部くっつければいいのですから。シンプルにするのは難しい。何を削るか、真剣に悩まなければならないからです。

パスカルが手紙に残した有名な言葉があります。

> もっと短く書く時間がなかったので、長く書きました。

コードも同じです。シンプルなコードは雑に作った結果ではなく、複雑さを取り除く努力の結果です。

## まとめ

KISS原則、まとめるとこうなります。

- コードは書く時間より読まれる時間のほうがずっと長いです。読む人のためにシンプルに書きましょう。
- 今必要ない拡張性は入れないこと。その心配が現実になることは、思ったより稀です。
- シンプルさは実力不足ではなく、実力の証です。

最近私はコードレビューで「これ、もっとシンプルにできませんか?」というコメントを一番多く残している気がします。不思議なことに、この質問一つでバグも減り、レビュー時間も減るんですよね。

あなたのコードは、隣の席の同僚が1分以内に理解できますか? 今日一度、自分に問いかけてみてください。
