# コンポジション vs 継承、「継承を使うな」という言葉の本当の意味

オブジェクト指向を勉強していると必ず出会う一文があります。「継承よりコンポジションを使え（Favor composition over inheritance）」。

結論から言うと、この言葉は「継承を絶対に使うな」という意味ではありません。「コード再利用のために継承を乱用するな」という意味です。継承は今でも有効な道具ですが、再利用が必要な場面のほとんどはコンポジションの方が安全な選択になります。

この記事では、両者の違いが実際のコードでどう分かれるのか、そしてどんな時にどちらを選ぶべきかを解説していきます。

## 継承とコンポジション、何が違うのか

まずはシンプルに整理してみましょう。

継承は「〜は〜である（is-a）」関係です。犬は動物である、というような関係ですね。親クラスの機能を子がそのまま受け継ぎます。

コンポジションは「〜は〜を持つ（has-a）」関係です。車はエンジンを持つ、というような形です。必要な機能を持つオブジェクトを内部に抱え込み、そのオブジェクトに仕事をさせる方式です。

コードで見るとよりイメージが湧きます。以下は継承で機能を受け継ぐ方式です。

```swift
// 継承: StackがNSMutableArrayのすべてのメソッドを受け継ぐ
class Stack: NSMutableArray {
    func push(_ o: Any) { add(o) }
    func pop() -> Any {
        let o = lastObject!
        removeLastObject()
        return o
    }
}
```

問題は、こうするとStackが望まないinsert(_:at:)、removeObject(at:)のようなメソッドまで全部外部に露出してしまうことです。

以下は同じものをコンポジションに変えた形です。

```swift
// コンポジション: 必要な機能だけを選んで委譲
class Stack {
    private var list: [Any] = []
    func push(_ o: Any) { list.append(o) }
    func pop() -> Any { list.removeLast() }
}
```

配列を内部に抱え込み、必要な動作だけを外に公開しました。Stackが本当にやりたいことだけが残ります。

## 「継承を使うな」という言葉の本当の意味

この言葉が生まれた背景には、継承の代表的な弱点が2つあります。

1つ目は、カプセル化が崩れることです。子は親の内部実装に依存するようになります。親のコードが変わると、問題なかったはずの子が急に壊れることもあります。

2つ目は、結合度が高くなりすぎることです。親と子がコンパイル時点でがっちり結びついてしまい、後から関係を変えるのが難しくなります。

> 継承はコードを再利用するための道具ではなく、型を定義するための道具です。

この一文が核心です。単にコードを再利用したいという理由だけで継承を使った瞬間、関係がこじれ始めます。

有名な例が正方形と長方形です。数学的には正方形は長方形なので、継承すればよさそうに思えます。ところが長方形の「幅と高さを別々に変える」機能を受け継いだ瞬間、正方形はもはや正方形ではなくなってしまいます。

is-a関係のように見えても、振る舞いまで完全に置き換えられないなら、継承は罠になります。

---

## では継承はいつ使ってもいいのか

継承が無条件に悪いわけではありません。以下の条件をすべて満たすなら、継承の方がむしろすっきりします。

1. 本当にis-a関係か：子は常に親の一種と言えるか
2. リスコフの置換原則を守っているか：親の場所に子を入れても何の問題もないか
3. 親が継承を前提に設計されているか：ドキュメント化されていて、拡張のために開かれているか

この3つがすべて当てはまるなら、継承を使っても構いません。UIViewControllerのようなフレームワーク基盤クラスを拡張する場合が代表例です。

逆にどれか一つでも曖昧なら、まずコンポジションを検討してください。

表で簡単に比較してみました。

| 状況 | おすすめ |
|------|------|
| 純粋なis-a関係、完全な置き換えが可能 | 継承 |
| 単純にコードだけ再利用したい場合 | コンポジション |
| 実行時に動作を切り替えたい場合 | コンポジション |
| 複数の機能を組み合わせる必要がある場合 | コンポジション |

ご覧の通り、実務で出会うほとんどの状況はコンポジション寄りになります。「favor composition」というアドバイスは伊達ではありません。

## 実務ではこう判断すべきです

新しいクラスを作る際は、習慣的に「これは親の一種なのか、それとも親の機能を借りたいだけなのか」をまず問うべきです。

借りたいだけなら、ほとんどの場合コンポジションを選ぶことになります。

デザインパターンを見ても方向性は明確です。ストラテジーパターン、デコレーターパターンといったものは、すべてコンポジションを基盤にしています。

特にストラテジーパターンは、動作をオブジェクトとして分離し、実行時に差し替えます。継承では真似しにくい柔軟性です。

もちろんコードが少し長くなるのは覚悟する必要があります。委譲メソッドを自分で書かなければならないからです。それでも、後で構造を変えるのがずっと楽になります。

---

「継承を使うな」という言葉は継承を禁止せよという意味ではなく、再利用目的の乱用を警戒せよというアドバイスです。is-aが確実なら継承、単に機能を借りたいだけならコンポジション。この基準一つを押さえるだけでも、コードは格段に堅牢になります。今日も良い設計を応援しています！
