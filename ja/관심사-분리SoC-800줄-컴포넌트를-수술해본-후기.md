800行のビューファイル1つに全部詰め込まれたコード、一度は見たことがあるはずです。

画面を描画するビューコードの中にAPI呼び出しがあり、その隣に割引率の計算ロジックがあり、その下に日付フォーマット関数があり、ところどころに分析イベント送信まで挟まっている。何か一つ直そうとしても800行全部を読まなければなりません。

こういうファイルが破っている原則こそ、関心の分離、英語でSeparation of Concernsです。開発原則シリーズ、最終回はこの原則です。

関心の分離(SoC)とは、プログラムを異なる関心事の単位に分割し、各部分が一つの関心事だけを扱うようにするという原則です。

「関心事」とは、プログラムが気にかけるべき事柄の種類のことです。画面をどう見せるか、データをどこから取ってくるか、ビジネスルールをどう適用するかは、それぞれ別の関心事です。

この用語を作ったのは計算機科学者のダイクストラ(Dijkstra)で、1974年の論文でこう述べています。

> 一度に一つの側面だけに集中して考えること。それが私の知る唯一の効果的な思考技法である。

人間の頭は一度に複数の関心事を扱えません。だからコードも一つの塊には一つの関心事だけを盛り込もうというわけです。

---

## 身近な例から: HTML、CSS、JS

Web開発をしたことがあるなら、すでに関心の分離を毎日実践してきたことになります。

| 技術 | 関心事 |
|------|--------|
| HTML | 構造 — 何があるか |
| CSS | 表現 — どう見えるか |
| JavaScript | 振る舞い — どう反応するか |

昔のようにHTMLタグに`style`属性と`onclick`をべたべた貼り付けると、ボタンの色一つ変えるのにもHTMLを漁らなければなりません。3つのファイルに分ければ、デザイナーはCSSだけ、マークアップ担当はHTMLだけを見ればよくなります。

バックエンドのコントローラー・サービス・リポジトリ構造、iOSでビューとビューモデル・データ層を分けることも、すべて同じ原理です。MVCだのレイヤードアーキテクチャだのと呼ばれるものは、結局関心の分離を定型化したパターンにすぎません。

## さっきの800行のビューを解剖してみる

先ほど見た800行のビューを関心事ごとに分けてみましょう。

```swift
// Before: すべての関心事が一つのファイルに
struct ProductPage: View {
    // 関心事1: データ取得(ネットワーク呼び出し、ローディング、エラー処理 150行)
    // 関心事2: ビジネスルール(割引率・在庫計算 200行)
    // 関心事3: 分析イベント(タップトラッキング 100行)
    // 関心事4: 画面描画(body 350行)
}
```

```swift
// After: 関心事ごとに居場所を用意した
ProductStore(id:)               // データ取得
calcDiscount(product, user)     // ビジネスルール(純粋関数)
Tracker.track("product")        // 分析イベント

struct ProductPage: View {      // 画面描画だけ
    @State private var store: ProductStore

    var body: some View {
        let price = calcDiscount(store.product, user)
        ...
    }
}
```

分けてみると良い点がはっきりします。割引ポリシーが変わったら`calcDiscount`だけ見ればよく、この関数は画面と無関係な純粋関数なのでテストも簡単です。APIの仕様が変わったら`ProductStore`だけ直せばいい。800行を全部読んでいた頃と比べると、修正範囲が10分の1に減ります。

## どこまで分けるべきか?

関心の分離で一番難しいのは「では境界線をどこに引くか」です。分けすぎると数十個のファイルを行き来しながら読む羽目になる迷路になってしまいます。

私が使っている基準は変更理由です。SOLID回で扱ったSRPと全く同じ問いです。

> このコードとあのコードは、同じ理由で、同じタイミングで変わるか?

割引ポリシーと画面レイアウトは違う理由で変わるので分離します。逆にボタンとそのボタンのタップハンドラーはほぼ常に一緒に変わるので、くっつけておきます。SwiftUIがビュー単位で構造・スタイル・振る舞いをむしろ一つのファイルにまとめているのも、この観点から見れば矛盾ではありません。技術の種類ではなく、一緒に変わる単位で関心事を引き直しただけなのです。

## シリーズを終えて

関心の分離で持ち帰ってほしい要点はこれです。

- 一つのコードの塊には一つの関心事だけを扱わせましょう。人間の頭はマルチタスクができません。
- 境界線は技術の種類ではなく「一緒に変わる単位」で引くのが実践的です。
- 過度な分離は迷路を生みます。分けること自体が目的になってはいけません。

これで開発原則シリーズが締めくくられました。KISS(シンプルに)、DRY(知識は一箇所に)、YAGNI(今必要なものだけ)、SOLID(変更の波及を狭く)、最小驚きの原則(予想通りに動くように)、そして関心の分離(一度に一つだけ)まで。

振り返ってみると、6つの原則すべてが同じところを指し示していました。コードはコンピューターではなく人間が読むものであり、良いコードとは変えやすいコードだということ。原則の名前は忘れても、この一文だけは覚えておいてください。
