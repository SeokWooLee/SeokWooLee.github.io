コードは確かに動いているのに、機能を一つ直そうとファイルを開くと、なぜあちこち手を入れなければならないのでしょうか。

ボタンの色を一つ変えただけなのに決済ロジックが壊れる経験、誰しも一度はあるはずです。

その原因をたった二語で説明する概念、それが結合度と凝集度です。

> 結合度は低いほど良く、凝集度は高いほど良い。良い設計の8割はこの一文から始まります。

この記事では、この二つの概念が正確には何なのか、なぜソフトウェア設計原則の根っこと呼ばれるのか、実際のコードにどう適用するのかまで掘り下げていきます。

まず要点から整理しましょう。

1. 結合度(Coupling):モジュール同士がどれだけ絡み合っているか → **低いほど**良い
2. 凝集度(Cohesion):一つのモジュール内のコードがどれだけ一つの仕事に集中しているか → **高いほど**良い
3. この二つがSOLIDやデザインパターンといった有名な原則の土台になっている
4. 目指すところは結局一つ、「直しやすく、差し替えやすいコード」

---

## 結合度と凝集度、一体何なのか

二つの概念をまず一文でまとめるとこうなります。

結合度はモジュールとモジュールの間の関係を指し、凝集度は一つのモジュール内部の結束を指します。

方向が正反対なので混同しやすいのですが、こう覚えると分かりやすいでしょう。

結合度は「外との距離」、凝集度は「内側のまとまり」だと。

結合度が高いと、Aを直すたびにB、Cまで芋づる式に影響が及びます。ドミノのようにです。

凝集度が低いと、一つのクラスの中に決済ロジック、メール送信、ログ記録が入り混じっています。名前だけを見ても何をするクラスなのか分かりません。

良い設計はこの二つを逆方向に押し進めます。外とは緩く、内は固くです。

---

## なぜこれが設計原則の「根っこ」なのか

結合度と凝集度は1970年代の構造化設計理論で最初に体系化された概念です。ラリー・コンスタンティンが提唱したものとして知られています。

半世紀近く経った今も通用し続けているのは、これが特定の言語や流行に左右されない本質だからです。

私たちが知っている有名な原則を分解してみると、結局ここに行き着きます。

| 原則・パターン | 結局言いたいこと |
|---|---|
| 単一責任の原則(SRP) | 凝集度を高めよ |
| 依存性逆転の原則(DIP) | 結合度を下げよ |
| インターフェース分離の原則(ISP) | 不要な結合を断ち切れ |
| 大半のデザインパターン | 低結合 + 高凝集 |

お分かりでしょうか。名前は違っても根っこは一つです。

ですから、新しい原則やパターンを学ぶときは、「これは結合度を下げようとしているのか、凝集度を高めようとしているのか」からまず問いかけてみましょう。それだけで半分は理解できます。

## 結合度を下げる方法、コードで見てみると?

言葉だけでは漠然としているので、短い例を見てみましょう。

以下は結合度が高いコードです。Orderクラスが特定の決済会社を直接知っています。

```swift
class Order {
    let pay = KakaoPay()  // KakaoPayにべったり依存してしまっている
    func checkout() {
        pay.send()        // 他の決済に変えるにはここを書き換える必要がある
    }
}
```

決済会社を別のものに変える瞬間、Orderを開いて修正しなければなりません。これが高い結合度の典型例です。

今度はプロトコルで間に距離を作ってみましょう。

```swift
class Order {
    let pay: Payment                // 「決済」という約束にだけ依存する
    init(pay: Payment) { self.pay = pay }
    func checkout() { pay.send() }  // 何が来ても問題ない
}
```

これでOrderはどの決済会社が来ても気にしなくなりました。差し替えるだけで済みます。

実務でこのように書き換えておくと、決済会社の追加要望が来ても既存コードをほとんど触らずに済むようになります。これが低い結合度の力です。

## 凝集度を高めるにはどうすればいいのか

凝集度は「このモジュールがたった一つのことだけをしているか」で判断します。

基準はシンプルです。クラス名を口にしたとき、その中のメソッドが全部自然に思い浮かべば凝集度が高いということです。

例えば、UserServiceの中にこんなものが混ざっていたら危険信号です。

- 会員登録処理(本業)
- マーケティングメール送信(他人の仕事)
- CSVレポート生成(これまた他人の仕事)

お互い関係のない仕事が一つの家に間借りしているようなものです。

こういうときは、メールはEmailSenderに、レポートはReportGeneratorにそれぞれ部屋を分けてあげればいいのです。

そうすれば、後でメールロジックを直すときにUserServiceを開く必要がなくなります。変更の波及が小さくなるわけです。

低い結合度と高い凝集度は、結局同じ方向を向いています。変更が広がらないようにすることです。

## よくある質問(Q&A)

**Q. 結合度と凝集度、どちらから気を配ればいいですか?**

凝集度から手をつけることをおすすめします。一つのモジュールが一つの仕事に集中するよう分割していくと、自然と結合度も整理されるケースが多いからです。

**Q. 結合度は無条件に0が良いのですか?**

いいえ。モジュール同士がまったく絡まなければプログラムは動きません。目指すのはゼロではなく「必要な分だけ、緩やかに」です。

**Q. 最初から完璧に設計すべきですか?**

その必要はありません。まずは動くように作った後、直すのがつらくなるたびに少しずつ結合を断ち、凝集を集めていけばいいのです。リファクタリングとはもともと繰り返すものです。

---

結合度と凝集度は派手な新技術ではありませんが、長く生き残るコードを書く人たちに共通する習慣です。

今日書いたコードで「これを直したらどこまで影響が及ぶだろう」と一度だけ思い浮かべてみてください。その問いが積み重なると、設計のセンスがぐっと育ちます。応援しています!
