シングルトンって本当にこのままでいいのかな、と思っていた方へ

iOS開発をしていると、シングルトン(Singleton)を使ったことがない人はいないと思います。私もそうでした。`SomeManager.shared`みたいなコード、どのプロジェクトにも必ず何個かありますよね。

でも、ある時からこの`.shared`がじわじわと気になり始めたんです。テストのときに状態が絡まったり、チームメンバーがあちこちから好き勝手にアクセスしたり。

そんな中で知ったのが、モノステート(Monostate)パターンです。今日はSwiftでモノステートとは何か、そして本当にシングルトンの代わりになれるのか、私が実際に使ってみた経験をシェアしたいと思います。

先に結論から言うと、モノステートは「状態は1つで共有しつつ、インスタンスは複数作れるようにする」パターンです。目的はシングルトンと同じですが、アプローチは正反対なんです。ただし万能というわけではなく、状況によって向き不向きが分かれます。以下で一つずつ見ていきますね。

## モノステートパターンって一体何?

シングルトンは「インスタンスをたった1つだけ作る」のが核心です。コンストラクタを塞いでおいて、`shared`だけを使わせるやり方ですね。

モノステートはその逆です。インスタンスはいくらでも作れます。その代わり、その中の状態(データ)を全部staticで共有してしまうんです。

つまりオブジェクトAを作ろうがBを作ろうが、両方が見ているデータは同じ場所です。見た目は普通のオブジェクトなのに、中身は1つにつながっているわけです。

Swiftで簡単に書くとこんな形になります。

```swift
struct Settings {
    private static var _volume = 50   // 状態をstaticで共有
    var volume: Int {                 // インスタンスのように使うが
        get { Settings._volume }
        set { Settings._volume = newValue }
    }
}
// 別々のインスタンスだが状態は1つ
var a = Settings(); a.volume = 80
print(Settings().volume)  // 80
```

`Settings()`を新しく作っても、volumeは80になります。インスタンスは複数だけど状態は1つ、これがモノステートのすべてです。

## シングルトンと何が違うの?

一番大きく違うのは、使う側のコードです。

シングルトンは、使う人が常に`.shared`を意識しなければなりません。「あ、これはシングルトンだった」という具合に。モノステートはただの普通のオブジェクトのように`Settings()`を作って使えばいいので、使う側が内部実装を知らなくても大丈夫です。

表にまとめるとこうなります。

| 区分 | シングルトン | モノステート |
|---|---|---|
| インスタンスの数 | 1つだけ | 複数可能 |
| 共有されるもの | インスタンス自体 | 状態(staticデータ) |
| 使う側 | `.shared`を明示 | 普通のオブジェクトのように |
| 継承 | 扱いにくい | 比較的自由 |

特に継承の部分が面白いんです。シングルトンは継承がなかなか厄介ですが、モノステートは通常の型なので拡張が比較的柔軟なんですよね。

---

## では、シングルトンの代わりになれるの?

正直に言うと、なれる場合もあれば、なれない場合もあります。

まずメリットから。使う側のコードがすっきりして、`.shared`というグローバルアクセスポイントがコードのあちこちに刺さらなくなります。インターフェースが普通のオブジェクトと同じなので、後で構造を変えるときの負担も少なくて済みます。

> モノステートは「グローバル状態を隠す」パターンです。便利に見えますが、隠されたグローバル状態はそれ自体が諸刃の剣です。

問題はここです。見た目は普通のオブジェクトなので、チームメンバーが「これはただのローカルオブジェクトだろう」と思って作ったのに、実は状態がグローバルで共有されていたとなると、かえって余計に混乱します。

そしてstatic共有である以上、マルチスレッド環境での同時実行の問題も同じように気を配る必要があります。この部分はシングルトンと変わりません。

私の経験上、最近のSwiftならモノステートよりも依存性注入(Dependency Injection)を先に検討することをおすすめします。テストもしやすいですし、依存関係も明確になりますから。

## じゃあ、いつ使えばいいの?

まとめると、こんなケースです。

1. 既存のシングルトンコードが多すぎて、段階的に取り除いていきたいとき
2. 使う側のインターフェースを普通のオブジェクトのようにすっきり保ちたいとき
3. 継承や拡張が必要なグローバル状態を扱わなければならないとき

逆に、新しいプロジェクトを最初から組むのであれば、モノステートよりも依存性注入や明示的な状態管理を先に検討したほうがいいと私は思います。

結局モノステートは「シングルトンの完璧な上位互換」ではなく、「性格の異なるもう一つの選択肢」です。道具箱に入れておいて、状況に合ったときに取り出して使えばいいんです。

今日の内容が、`.shared`の乱用に疲れていた方への小さなヒントになれば嬉しいです。むやみに乗り換えるより、自分のプロジェクトに何が合うのか、一度じっくり考えてみることを応援しています!
