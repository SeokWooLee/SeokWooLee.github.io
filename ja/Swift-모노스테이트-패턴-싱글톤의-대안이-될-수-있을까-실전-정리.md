シングルトン、本当にこのままで大丈夫かなと思ったことがある方へ

iOS開発をしていると、シングルトン(Singleton)を使ったことがない人はいないと思います。私もそうでした。`SomeManager.shared`みたいなコード、プロジェクトごとにいくつかは必ずありましたから。

でも、ある時からこの`.shared`がじわじわ気になり始めたんです。テストするたびに状態が絡み合うし、チームメンバーがあちこちから好き勝手にアクセスするし。

そんな中で知ったのがモノステート(Monostate)パターンです。今日はSwiftにおけるモノステートとは何か、そして本当にシングルトンの代替になり得るのか、私が実際に使ってみた経験をシェアしたいと思います。

先に結論から言うと、モノステートは「状態は一つに共有しつつ、インスタンスは複数作れるようにする」パターンです。シングルトンと目的は同じですが、アプローチは正反対なんですよね。ただし万能ではなく、状況によって向き不向きが分かれます。以下で一つずつ紐解いていきます。

## モノステートパターンとは一体何なのか

シングルトンは「インスタンスを一つだけ作る」のがポイントです。コンストラクタを塞いでおいて、`shared`一つだけを使わせるわけです。

モノステートはその逆です。インスタンスはいくつでも作れます。その代わり、中の状態(データ)を全部staticで共有してしまうんです。

つまりオブジェクトAを作ろうがBを作ろうが、両者が参照するデータは同じ場所にあります。見た目は普通のオブジェクトなのに、中身は一つにつながっているわけです。

Swiftで簡単に書くとこんな形になります。

```swift
struct Settings {
    private static var _volume = 50   // 状態をstaticで共有
    var volume: Int {                 // インスタンスのように使うが
        get { Settings._volume }
        set { Settings._volume = newValue }
    }
}
// 異なるインスタンスだが状態は一つ
var a = Settings(); a.volume = 80
print(Settings().volume)  // 80
```

`Settings()`を新しく作ってもvolumeは80のままです。インスタンスは複数あるのに状態は一つ、これがモノステートの全てです。

---

## シングルトンと何が違うのか

一番大きく違うのは、使う側のコードです。

シングルトンは使う人が常に`.shared`を意識しなければなりません。「あ、これはシングルトンだったな」と思いながら。モノステートは普通のオブジェクトのように`Settings()`を作って使えばいいので、使う側が内部実装を知らなくても済みます。

表にまとめるとこうなります。

| 区分 | シングルトン | モノステート |
|---|---|---|
| インスタンス数 | 1個のみ | 複数可能 |
| 共有されるもの | インスタンス自体 | 状態(staticデータ) |
| 使う側 | `.shared`を明示 | 一般オブジェクトのように |
| 継承 | 扱いにくい | 比較的自由 |

特に継承のところが面白いです。シングルトンは継承が結構ややこしいのですが、モノステートは通常の型なので拡張の自由度が高めです。

---

## では、シングルトンの代替になり得るのか

正直に言うと、なる時もあればならない時もあります。

まずメリットから。使う側のコードがすっきりして、`.shared`という大域的なアクセスポイントがコードのあちこちに刺さらずに済みます。インターフェースが普通のオブジェクトと同じなので、後で構造を変える時の負担も少ないです。

> モノステートは「グローバル状態を隠す」パターンです。便利そうに見えますが、隠されたグローバル状態はそれ自体が諸刃の剣です。

問題はここなんです。見た目は普通のオブジェクトなので、チームメンバーが「これはただのローカルオブジェクトだろう」と思って作ったら、実は状態がグローバルに共有されていた、なんてことになると、かえって余計に混乱します。

さらに、static共有であるためマルチスレッド環境での並行性の問題も同じように気をつけないといけません。この点はシングルトンと変わりません。

私の経験では、最近のSwiftならモノステートよりもDI(Dependency Injection、依存性注入)を先に検討することをおすすめします。テストもしやすいし、依存関係も明確になりますから。

## では、いつ使えばいいのか

まとめると、こういう場合です。

1. 既存のシングルトンコードが多すぎて、段階的に取り除いていきたい時
2. 使う側のインターフェースを普通のオブジェクトのようにすっきり保ちたい時
3. 継承や拡張が必要なグローバル状態を扱わなければならない時

逆に、新規プロジェクトを最初から組む場合なら、モノステートよりもDIや明示的な状態管理をまず検討することをおすすめします。

結局モノステートは「シングルトンの完璧な上位互換」ではなく、「性質の異なるもう一つの選択肢」です。道具箱に入れておいて、状況に合った時に取り出して使えばいいんです。

今日の内容が、`.shared`の乱用に疲れていた方への小さなヒントになればうれしいです。むやみに乗り換えるより、自分のプロジェクトに何が合うか一度考えてみることを応援します!
