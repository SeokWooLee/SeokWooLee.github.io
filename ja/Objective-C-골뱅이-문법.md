Objective-Cのコードを初めて開くと、まず目に留まるものがあります。

それが、あちこちに付いているアットマーク（@）です。

@interface、@propertyはもちろん、文字列まで@"Hello"のように書きますよね。初めて見ると「これ、タイプミスじゃないの？」と思うかもしれません。

でも、この@にはとても明確な理由があります。

> Objective-Cの@は、コンパイラに対して「ここからは純粋なCではなく、Objective-Cの拡張構文ですよ」と知らせるサインです。

今日は、なぜこのアットマークが@interfaceにも、@propertyにも、@"文字列"にも同じように付くのか、その仕組みをわかりやすく解説します。

## Objective-Cの@とは一体何なのか

まず結論からお伝えします。

@は「これはObjective-C独自の構文だ」とコンパイラに示す特別な記号です。

これを理解するには、Objective-Cの生い立ちを知る必要があります。

Objective-Cはまったく新しい言語ではありません。C言語の上にオブジェクト指向機能を乗せた言語です。

つまり、C言語をそのまま100%含みつつ、そこにクラスやメッセージといった機能だけを追加した構造なのです。

ところが、ここで一つ問題が生じます。

コンパイラの立場からすると、どこまでが元々のC構文で、どこからが新しく追加されたObjective-C構文なのかを区別する必要があるのです。

その境界線を引くための印が、まさに@だったわけです。

---

## なぜよりによってアットマーク（@）だったのか

アスタリスク（*）もあれば、シャープ（#）もあるのに、なぜわざわざ@だったのでしょうか。

答えは意外と実用的です。

C言語の標準構文において、@記号はどこにも使われていないからです。

Cで使われる文字を思い浮かべてみましょう。

- 変数・関数名：英字、数字、アンダースコア（_）
- 演算子：+、-、*、/、=、&など
- プリプロセッサ：#（例：#include）

このリストのどこにも@はありません。

そこでObjective-Cを作った人たちが@を採用したのです。既存のCコードと絶対に衝突しない、安全な記号だったからです。

もし@の代わりに単純にinterfaceという単語をキーワードにしていたら、interfaceという名前の変数を使っていた既存のCコードはすべて壊れていたでしょう。

@を前に付けることで、そうした衝突が根本から防がれるわけです。

わかりやすく例えるなら、@はObjective-C専用の構文をまとめて収める、専用の引き出しの取っ手のようなものです。

## @interface・@property・@"文字列"がすべて@である理由

ここからが本題です。性質が異なって見えるこの3つが、なぜ同じように@で始まるのか見ていきましょう。

結論から言うと、3つとも「Cにはない、Objective-C独自の機能」だからです。

一つずつ見ていきます。

**@interface** — クラスの骨組み（宣言）を開始する指示子です。Cにはクラスという概念自体が存在しないため、@で示す必要がありました。ペアとなる@implementation、@endも同じ理由です。

**@property** — オブジェクトのプロパティを自動的に生成してくれる機能です。これも純粋なCにはない構文なので@が付きます。

**@"文字列"** — これが一番面白いところです。単に引用符だけを使った"Hello"と@"Hello"は、まったく別物なのです。

表で比較してみましょう。

| 区分 | "Hello" | @"Hello" |
|------|---------|----------|
| 正体 | Cスタイルの文字配列 | NSStringオブジェクト |
| 所属 | 元々のC構文 | Objective-C拡張 |
| 機能 | 単純な文字の羅列 | lengthなどのメソッドが使える |

引用符の前に@を一つ付けるだけで、単なる文字の塊ではなく「オブジェクト」に変わります。

だからこそ@が必要なのです。「この文字列は単なるC文字列ではなく、NSStringオブジェクトとして作ってほしい」という指示のようなものですね。

まとめると、こうなります。

> @interfaceでも@propertyでも@"文字列"でも、すべて「Cにはなく、Objective-Cにだけある機能」だからこそ、同じ@マークが付いているのです。

## コードで一目で見る

言葉だけだと少し抽象的に感じるかもしれないので、短い例を用意しました。

以下はシンプルなクラス宣言の例です。@がどのようにObjective-C構文を包んでいるか見てみましょう。

```objectivec
@interface Person : NSObject   // クラス宣言の開始（Cにはない構文）
@property NSString *name;      // プロパティを自動生成
@end                           // 宣言の終わり

NSString *greeting = @"こんにちは"; // 引用符の前の@ → NSStringオブジェクト
```

ご覧の通り、@で始まる行はすべてObjective-Cが新たに追加した機能です。

逆に@が付いていない部分は、C言語からそのまま受け継いだ構文です。

このように@の有無を見るだけで、「あ、これはObjective-C拡張なんだな」とひと目で区別できるわけです。

## よくある質問（Q&A）

**Q. @はプリプロセッサの#と同じものですか？**

いいえ、違います。#includeの#は、コンパイル前に処理されるプリプロセッサ指示子です。一方、@はコンパイラが直接解釈するObjective-Cコンパイラ指示子（directive）なので、役割が異なります。

**Q. @の種類はこれ以外にもたくさんありますか？**

はい、かなりあります。@protocol、@selector、@try/@catch、@autoreleasepoolなど多彩です。最新の構文では@[]配列、@{}辞書、@42数値オブジェクトまで、@を使って簡単に作れます。

**Q. Swiftにも@があると聞きましたが？**

そうですね。ただし意味は少し異なります。Swiftの@は、@State、@IBOutletのように主にプロパティ（属性）を示すために使われます。ルーツは似ていますが、使われ方は分けて考えるとよいでしょう。

@さえ追いかければ、どこからがオブジェクト指向構文なのかがはっきり見えてくるからです。Objective-Cのコードを読むときは、この視点さえ覚えておけばぐっと読みやすくなるはずです。今日も楽しいコーディングを！
