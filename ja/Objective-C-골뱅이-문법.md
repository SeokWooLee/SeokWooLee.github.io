Objective-Cのコードを初めて開くと、真っ先に目に留まるものがあります。

そう、あちこちに付いているアットマーク（@）です。

@interface、@property、果ては文字列まで@"Hello"というふうに書かれています。私も最初は「これ、打ち間違いじゃないの？」と思いました。

でもこの@には、実にはっきりした理由があるんです。

> Objective-Cの@は、コンパイラに「ここからは純粋なCじゃなくて、Objective-C拡張構文ですよ」と知らせるサインです。

今日は、このアットマークが@interfaceにも@propertyにも@"文字列"にも同じように付いている理由を、その仕組みからゆるっと解説していきます。

## Objective-Cの@っていったい何？

結論から言いますね。

@は「これはObjective-C独自の構文だ」とコンパイラに示す特別な記号です。

これを理解するには、Objective-Cの成り立ちを知る必要があります。

Objective-Cはまったく新しい言語というわけではありません。C言語の上にオブジェクト指向機能を乗せた言語です。

つまり、C言語を100%そのまま含みながら、そこにクラスやメッセージといった機能だけを新たに付け加えた構造なんです。

ただ、ここで一つ問題が出てきます。

コンパイラの立場からすると、どこまでが元々のC構文で、どこからが新しく追加されたObjective-C構文なのかを区別しなければなりません。

その境界線を引く目印こそが@だったというわけです。

---

## なぜよりによってアットマーク（@）だったのか？

私もここが一番気になりました。アスタリスク（*）もシャープ（#）もあるのに、なぜ@だったんでしょうか。

答えは意外と実用的です。

C言語の標準構文では、@記号はどこにも使われていないんです。

Cで使う文字を一度思い浮かべてみましょう。

- 変数・関数名：英字、数字、アンダースコア（_）
- 演算子：+、-、*、/、=、&など
- プリプロセッサ：#（例：#include）

このリストのどこにも@はありません。

だからObjective-Cを作った人たちは@を採用したんです。既存のCコードと絶対に衝突しない、安全な記号だったから。

もし@の代わりにinterfaceという単語をそのままキーワードにしていたら、interfaceという名前の変数を使っていた既存のCコードが軒並み壊れていたはずです。

@を前に付けることで、そうした衝突を根本から防いでいるわけです。

分かりやすく例えるなら、@はObjective-C専用の構文をまとめておく、専用の引き出しの取っ手のようなものです。

## @interface・@property・@"文字列"が全部@である理由

さて、ここからが本題です。性質が違って見えるこの3つが、なぜ揃って@から始まるのかを見ていきましょう。

結論から言うと、3つとも「Cにはない、Objective-C独自の機能」だからです。

一つずつ見ていきますね。

**@interface** — クラスの骨組み（宣言）を開く指示子です。Cにはクラスという概念そのものが存在しないので、@で示す必要がありました。対になる@implementation、@endも同じ理由です。

**@property** — オブジェクトのプロパティを自動生成してくれる機能です。これも純粋なCにはない構文なので@が付きます。

**@"文字列"** — これが一番面白いところです。単なる引用符だけの"Hello"と@"Hello"は、まったく別物なんです。

表で比較してみましょう。

| 区分 | "Hello" | @"Hello" |
|------|---------|----------|
| 正体 | Cスタイルの文字配列 | NSStringオブジェクト |
| 所属 | 元々のC構文 | Objective-C拡張 |
| 機能 | 単なる文字の羅列 | lengthなどのメソッドが使える |

引用符の前に@を一つ付けるだけで、単なる文字の塊ではなく「オブジェクト」に変わります。

だからこそ@が必要なんです。「この文字列は単なるC文字列じゃなく、NSStringオブジェクトとして作ってくれ」という指示というわけですね。

まとめるとこうなります。

> @interfaceであれ@propertyであれ@"文字列"であれ、全部「Cにはなく、Objective-Cだけにある機能」だから、同じ@マークが付いているんです。

## コードで一目で見る

言葉だけだと少し抽象的に感じるかもしれないので、短い例を用意しました。

以下は簡単なクラス宣言の例です。@がどのようにObjective-C構文を包んでいるかが見て取れます。

```objectivec
@interface Person : NSObject   // クラス宣言の開始（Cにはない構文）
@property NSString *name;      // プロパティを自動生成
@end                           // 宣言の終わり

NSString *greeting = @"안녕하세요"; // 引用符の前の@ → NSStringオブジェクト
```

見てのとおり、@から始まる行はすべてObjective-Cが新たに追加した機能です。

逆に@が付いていない部分は、C言語からそのまま持ってきた構文です。

こうして@の有無を見るだけで、「あ、これはObjective-C拡張だな」とひと目で区別できるんです。

## よくある質問（Q&A）

**Q. @はプリプロセッサの#と同じものですか？**

いいえ、違います。#includeの#は、コンパイル前に処理されるプリプロセッサ指示子です。一方@は、コンパイラが直接解釈するObjective-Cコンパイラ指示子（ディレクティブ）なので、役割が異なります。

**Q. @の種類は他にもたくさんありますか？**

はい、かなりあります。@protocol、@selector、@try/@catch、@autoreleasepoolなど多岐にわたります。最近の構文では@[]配列、@{}辞書、@42数値オブジェクトまで、@を使って簡単に作れます。

**Q. Swiftにも@がありますよね？**

そうですね。ただし意味は少し違います。Swiftの@は、@State、@IBOutletのように主に属性（アトリビュート）を示すのに使われます。ルーツは似ていますが、使われ方は分けて考えるといいでしょう。
