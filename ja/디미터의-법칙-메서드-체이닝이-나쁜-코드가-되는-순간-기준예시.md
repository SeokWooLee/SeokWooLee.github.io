コードレビューをしていると、一度はこんな指摘を受けることがあります。

「この行、ドット(.)が多すぎませんか?」

私も最初はそれの何が問題なのかよく分かりませんでした。一行にずらっと書くと、むしろすっきりして見えたんですよね。

でも、この話のポイントがまさに**デミトリスの法則**なんです。

今日はデミトリスの法則が何なのか、そしてなぜあるメソッドチェーンは問題なくて、あるメソッドチェーンは悪いコードになるのかを、実例とともに解説していきます。

まず結論から言いますね。

> デミトリスの法則とは、「見知らぬオブジェクトに話しかけるな」というルールです。
>
> メソッドチェーン自体が悪いのではなく、チェーンによって他人の内部まで踏み込んでしまうときに悪いコードになります。

この一文さえ覚えておけば、今日の記事の半分は押さえたも同然です。

## デミトリスの法則とは

デミトリスの法則(Law of Demeter)は「最小知識の原則」とも呼ばれます。

簡単に言うとこうです。

あるオブジェクトは、自分が直接知っている「近い友人」とだけ会話すべきだということです。

友人の友人、そのまた友人まで引っ張り出して使い始めると問題が生じます。

メソッドの中で呼び出してよい対象は、一般的にこう整理されます。

- 自分自身(this)のメソッド
- メソッドの引数として渡されたオブジェクト
- メソッドの中で直接生成したオブジェクト
- 自分が保持しているフィールド(メンバー)オブジェクト

この範囲の中だけで会話しよう、ということです。

ここから「一行に一つのドットまで」という有名な経験則も生まれました。

もちろんドットの数はあくまでヒントであって、絶対的なルールではありません。

## メソッドチェーンはなぜ悪いコードになるのか?

私が実際に経験したコードでお見せします。

注文から顧客の都市名を取り出す場面でした。

```java
// 注文 → 顧客 → 住所 → 都市までずるずる踏み込む
String city = order.getCustomer()
                   .getAddress()
                   .getCity()
                   .getName();
```

一見すっきりして見えますよね。

ですが、この一行はOrderがCustomerの内部、その中のAddress、さらにその中のCityまですべて知っているということを意味します。

問題はここで起こります。

もしAddressの構造が変わったり、Cityがなくなったりしたら、このコードも一緒に崩れます。

他人の家の引き出しの中にまで手を突っ込んでいるので、その家の構造が変わると自分のコードが壊れてしまうわけです。

このようにオブジェクトグラフをたどって下りていくチェーンは、よく「列車事故(train wreck)」と呼ばれます。

## では、良いチェーンと悪いチェーンはどう区別するのか?

ここが今日の記事で一番重要な部分です。

メソッドチェーンだからといって、すべてが悪いわけではないからです。

判断基準は一つです。

**「チェーンしながら他人の内部オブジェクトを次々と取り出しているか?」**

先ほどの例のようにgetCustomer → getAddress → getCityと、互いに異なる他人のオブジェクトを次々と掘り下げていくと違反になります。

逆に、下のように同じ種類の対象を扱いながら自分自身を返し続けるチェーンは問題ありません。

```java
// ストリーム・ビルダーは毎回「同じ文脈」を返すので違反ではない
List<String> names = users.stream()
                          .filter(User::isActive)
                          .map(User::getName)
                          .toList();
```

ストリームやビルダーパターンは、ドットがいくら多くても他人の内部をかき回すことはありません。

毎回自分自身(同じ流れ)を返しながらつながっているだけです。

二つのコードを表で比較するとこう整理できます。

| 区分 | 悪いチェーン | 良いチェーン |
| --- | --- | --- |
| 対象 | 他人の内部オブジェクトを次々取り出す | 同じ流れ・型を返す |
| 例 | order.getCustomer().getAddress() | stream().filter().map() |
| 結合度 | 高い(構造変更に弱い) | 低い |
| 判定 | デミトリス違反 | 違反ではない |

## 違反コード、こうやって直します

解決法は思ったより単純です。

「聞くな、命じろ(Tell, Don't Ask)」を思い出せばいいんです。

内部を取り出してくるのではなく、その仕事をやってくれと頼むわけです。

先ほどの都市の例なら、こう変えられます。

```java
// Orderに必要な結果だけを尋ねる
String city = order.getShippingCity();
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="聞くな、命じろ。こうやって一行で済むんですよね">
  <figcaption>聞くな、命じろ。こうやって一行で済むんですよね</figcaption>
</figure>

Orderの内部で自らCustomerとAddressを経由して都市を返すように作るわけです。

こうすればAddressの構造が変わっても、外側のコードは無事です。

修正範囲がOrderの内側にきっちり閉じ込められるからです。

私も実際にこういう形でリファクタリングしてみたところ、あとで構造を変えるときに手を入れる箇所がぐっと減りました。

---

最後にまとめるとこうです。

デミトリスの法則はドットの数を数えろというルールではなく、他人の内部にむやみに手を出すなという話です。

次に長いチェーンに出会ったら、一つだけ問いかけてみてください。

「これ、他人の引き出しを開けているんじゃないか?」

この問い一つを投げかけてみるだけでも、コードはぐっとすっきりします。
