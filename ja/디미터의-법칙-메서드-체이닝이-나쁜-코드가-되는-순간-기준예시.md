コードレビューをしていると、こんな指摘を一度は受けることがあります。

「この行、ドット(.)が多すぎませんか?」

一行にずらっと書くとむしろすっきり見えるのに、何が問題なのでしょうか。

この話の核心が、まさに**デミテルの法則**です。

今日はデミテルの法則が何なのか、そしてなぜあるメソッドチェーンは問題なく、あるものは悪いコードになってしまうのかを、例とともに解説していきます。

まず結論からお伝えします。

> デミテルの法則とは「見知らぬオブジェクトに話しかけるな」というルールです。
>
> メソッドチェーン自体が悪いのではなく、チェーンによって他人の内部にまで踏み込むことが悪いコードになるのです。

この一文さえ覚えておけば、今日の記事の半分は押さえたことになります。

---

## デミテルの法則とは

デミテルの法則(Law of Demeter)は「最小知識の原則」とも呼ばれます。

分かりやすく言うとこうです。

あるオブジェクトは、自分が直接知っている「近しい友人」とだけ会話すべきだということです。

友人の友人、さらにその友人の友人まで引っ張り出して使い始めると、問題が生じます。

メソッドの中で呼び出してよい対象は、通常このように整理されます。

- 自分自身(self)のメソッド
- メソッドの引数として渡されたオブジェクト
- メソッド内で直接生成したオブジェクト
- 自分が保持しているプロパティ(メンバー)オブジェクト

この範囲の中だけで会話しよう、ということです。

ここから「一行にドットは一つまで」という有名な経験則も生まれました。

もちろんドットの数はあくまでヒントであって、絶対的なルールではありません。

---

## メソッドチェーンはなぜ悪いコードになるのか?

実際に私が経験したコードでお見せします。

注文から顧客の都市名を取り出す場面でした。

```swift
// 注文 → 顧客 → 住所 → 都市まで次々と掘り下げる
let city = order.customer
                .address
                .city
                .name
```

一見すっきりして見えますよね。

ですがこの一行は、OrderがCustomerの内部、さらにその中のAddress、そのまた中のCityまで全部知っているということを意味します。

問題はここで起こります。

もしAddressの構造が変わったり、Cityがなくなったりすると、このコードも一緒に壊れてしまいます。

他人の家の引き出しの中にまで手を突っ込んでいるので、その家の構造が変わると自分のコードが壊れてしまうわけです。

このようにオブジェクトグラフを辿って下りていくチェーンは、よく「列車事故(train wreck)」と呼ばれます。

## では良いチェーンと悪いチェーンはどう見分けるのか?

ここが今日の記事で一番重要な部分です。

メソッドチェーンだからといって、すべてが悪いわけではないからです。

判断基準は一つです。

**「チェーンしながら他人の内部オブジェクトを次々と取り出しているか?」**

先ほどの例のように、customer → address → cityと、それぞれ異なる他人のオブジェクトを次々と掘り下げていくなら違反です。

反対に、以下のように同じ種類の対象を扱いながら自分自身を返し続けるチェーンは問題ありません。

```swift
// filter・mapのチェーンは毎回「同じ文脈」を返すので違反ではない
let names = users
    .filter { $0.isActive }
    .map(\.name)
```

こうした高階関数のチェーンやビルダーパターンは、ドットがいくら多くても他人の内部をかき回すことはありません。

毎回自分自身(同じ流れ)を返しながらつながっているだけです。

両方のコードを表で比較するとこうなります。

| 区分 | 悪いチェーン | 良いチェーン |
| --- | --- | --- |
| 対象 | 他人の内部オブジェクトを次々と取り出す | 同じ流れ/型を返す |
| 例 | order.customer.address | filter { }.map { } |
| 結合度 | 高い(構造変更に弱い) | 低い |
| 判定 | デミテル違反 | 違反ではない |

## 違反コード、こう直します

解決法は思ったより簡単です。

「聞くな、命じよ(Tell, Don't Ask)」を思い浮かべればいいのです。

内部を取り出してくるのではなく、その仕事をやってほしいと頼むわけです。

先ほどの都市の例なら、こう変えられます。

```swift
// Orderに必要な結果だけを尋ねる
let city = order.shippingCity
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="聞くな、命じよ。こんな一行で十分でした">
  <figcaption>聞くな、命じよ。こんな一行で十分でした</figcaption>
</figure>

Orderの内部で自らCustomerとAddressを経由して都市を返すようにするわけです。

こうすればAddressの構造が変わっても、外側のコードは無事です。

修正範囲がOrderの内側だけにきっちり閉じ込められるからです。

実際にこの方法でリファクタリングしてみると、後で構造を変えるときに手を加える箇所がぐっと減ります。

---

まとめるとこうなります。

デミテルの法則は、ドットの数を数えろというルールではなく、他人の内部にむやみに手を出すなという話です。

次に長いチェーンに出会ったら、一つだけ問いかけてみてください。

「これ、他人の引き出しを開けているんじゃないか?」

この問い一つを投げかけてみるだけでも、コードがぐっとすっきりします。
