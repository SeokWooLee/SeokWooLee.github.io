Swiftでオブジェクトをコピーしていて、「これって本当にコピーされてるの？」と手が止まったことはありませんか。

クラスのインスタンスを代入一行でコピーすると、複製したほうを直したのに元のオブジェクトまで一緒に変わってしまう問題につながりがちです。

結論から言うと、Swiftにおけるコピーは大きく二つに分かれます。

値型（struct）は代入するだけで自動的にコピーされますが、クラス（参照型）は`NSCopying`を自分で実装しないと本当の意味でのコピーにはなりません。

この記事では、プロトタイプパターンとは何か、そして`NSCopying`と値型のコピーがどう違うのかをコードで一つずつ整理していきます。

## プロトタイプパターンとは？

プロトタイプパターンを一言で言うと、**既存のオブジェクトを複製して新しいオブジェクトを作るデザインパターン**です。

ゼロから新しく作るのではなく、すでに設定が済んだ元のオブジェクトを「型を取るように」コピーする方式です。

私はこれを、はんこに例えて理解しています。元になるはんこを一つ丁寧に彫っておけば、あとは押すだけで済みますよね。

オブジェクトの生成コストが大きかったり、初期設定が複雑だったりする場合に特に有用です。

例えば、ゲームキャラクターを作るとき、ステータスや装備、スキルを毎回設定する代わりに、基本キャラクターを一体複製して少しずつ変えていくようなイメージです。

重要なのは「複製したオブジェクトが元のオブジェクトと完全に切り離されていること」です。

> 複製を修正したときに元のオブジェクトまで一緒に変わってしまうなら、それはコピーではなく参照の共有にすぎません。

まさにこの点で、値型と参照型の違いが分かれてきます。

---

## 値型のコピーはなぜ楽なのか？

Swiftのstructとenumは値型です。

値型の最大のメリットは、代入や関数への引き渡しをするだけで**自動的にコピーされる**という点です。

下のコードを見ればイメージがつかめるはずです。元のオブジェクトをコピーしたあと値を変えても、お互いに影響しません。

```swift
struct Character {
    var name: String
    var level: Int
}

var origin = Character(name: "전사", level: 1)
var copy = origin      // ここで値がまるごとコピーされる
copy.level = 99
print(origin.level)    // 1（元の値はそのまま！）
```

`copy.level`を99に変えても、`origin.level`は依然として1のままです。

別途コピー用のコードを書かなくてもいいので、ミスの余地がぐっと減ります。

そのため、特別な理由がない限りはstructをまず検討するようにしています。

Swiftが値型のコピーを推奨しているのも、まさにこの安全性のためです。

## NSCopyingはどう使うのか？

問題はクラス（参照型）です。

クラスは代入してもコピーにはならず、**同じインスタンスを指すアドレスが共有される**だけです。

そのため、本当の意味でのコピーが必要な場合は`NSCopying`プロトコルを採用し、`copy(with:)`メソッドを自分で実装する必要があります。

```swift
class Character: NSCopying {
    var name: String
    init(name: String) { self.name = name }

    func copy(with zone: NSZone? = nil) -> Any {
        return Character(name: self.name)
    }
}

let origin = Character(name: "전사")
let clone = origin.copy() as! Character
```

`copy()`を呼び出すと新しいインスタンスが作られるので、`clone`を変更しても`origin`はそのままです。

ここで一つ注意しておきたい点があります。

`copy(with:)`の中で内部のプロパティまで新しく作ってはじめて、完全なコピー（ディープコピー）に近づきます。

内部のオブジェクトをそのまま代入するだけだと、表面だけコピーされて中身は依然として共有される、シャローコピーになってしまいます。

---

## 値型のコピー vs NSCopying、ひと目で比較

二つの方式の違いを表にまとめてみました。

| 区分 | 値型（struct）のコピー | NSCopying（class） |
|------|------|------|
| コピー方式 | 代入時に自動コピー | `copy()`を自分で呼び出す |
| 実装の要否 | 不要 | `copy(with:)`の実装が必要 |
| 基本動作 | 常に独立したコピー | 代入時はアドレスを共有 |
| シャロー/ディープコピー | 気にする場面が少ない | 自分で対応する必要あり |
| おすすめの場面 | ほとんどのデータモデル | 参照型がどうしても必要なとき |

まとめると、よほどの理由がない限り値型で設計するほうが楽で安全です。

ただし、継承が必要な場合やObjective-C APIと絡む場合、インスタンスの同一性が重要な場合には、クラスと`NSCopying`を使います。

こうしたケースでプロトタイプパターンを`NSCopying`で実装すると、きれいにはまります。

## よくある質問（Q&A）

**Q. structだけ使っていればNSCopyingは知らなくてもいいですか？**

基本的なコピーに関してはそうです。ただ、UIKitやObjective-CベースのAPIを扱っているとクラスをコピーする場面が出てくるので、概念は知っておいたほうがいいでしょう。

**Q. copy()とmutableCopy()は何が違いますか？**

`copy()`はイミュータブルな複製、`mutableCopy()`は変更可能な複製を作る用途に分かれます。後者を使うには`NSMutableCopying`を採用する必要があります。

**Q. ディープコピーが常に正解なのでしょうか？**

そうとは限りません。内部のオブジェクトを共有しても問題ない状況であれば、シャローコピーのほうがパフォーマンス面で有利なこともあります。

## おわりに

コピーで迷ったときは、「元のオブジェクトを触ったときに複製も一緒に変わるか」だけを思い浮かべてみてください。

値型はその悩みをSwiftが代わりに解決してくれますし、クラスは`NSCopying`で自分たちの手できちんと対応すればいいだけです。
