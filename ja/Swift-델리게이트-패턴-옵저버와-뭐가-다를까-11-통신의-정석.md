iOS開発をしていると、デリゲートパターンには本当に嫌というほど出会います。

UITableViewやUITextFieldを触るたびにdelegateを付けながら、こう思ったことが一度はあるはずです。「これ、なんでデリゲートなんだろう?普通にクロージャやNotificationを使えばいいんじゃない?」

特にオブザーバーパターンまで学んだ後だと、むしろ余計に混乱しやすくなります。どちらも「何か起きたら知らせる」構造なのに、一体何が違うのか、というわけです。

結論から言います。

> デリゲートは、ある1つのオブジェクトがたった1人に「これは君に任せるよ」と委任する1対1通信で、
>
> オブザーバーは、1つの出来事を複数の相手が同時に購読する1対N放送です。

この一行さえ押さえておけば、もう半分は理解できたようなものです。今日はこの違いをコードと一緒にきっちり整理していきます。

## デリゲートパターンって何?

デリゲート(delegate)は日本語で言うと「委任」です。

自分で処理するには微妙な仕事を、信頼できる別の1つのオブジェクトに「これは君が代わりに決めてくれ」と渡すことです。

日常生活に例えるとこうなります。

レストランを予約したいけれど、自分で電話するのが面倒なので、秘書1人に頼みます。「予約、代わりにやっておいて」。

ここで秘書がまさにデリゲートです。頼む人は1人、頼まれる人も1人。まさに1対1ですね。

Swiftではこれをプロトコルで約束します。

```swift
// 委任する仕事をプロトコルで定義
protocol OrderDelegate: AnyObject {
    func didFinishOrder(_ menu: String)
}

class Restaurant {
    weak var delegate: OrderDelegate?  // たった1人だけを指定
    func order(_ menu: String) {
        delegate?.didFinishOrder(menu)  // その1人だけに通知
    }
}
```

ここで注目すべきポイントは、`delegate`が配列ではなく**たった1つのプロパティ**だということです。

1人だけに任せるので、当然1つしか入りません。これがデリゲートが1対1である理由です。

そして`weak`が付いていますね。互いを強参照すると循環参照(retain cycle)でメモリリークが起きるため、デリゲートは慣例的に弱参照で保持します。

## オブザーバーパターンと何が違う?

オブザーバー(observer)は「観察者」です。

何か出来事が起きると、それを見守っていた**複数の相手に一斉に**通知が飛びます。

先ほどの秘書の例えを続けてみましょう。

今度はレストランのオープンのお知らせを、街中に放送でばら撒くとします。「本日夕方オープンします!」

この放送を聞く人は10人かもしれないし、100人かもしれません。誰が聞いているかは気にせず、ただ流すだけです。

これが1対N放送、オブザーバーパターンです。

iOSでは`NotificationCenter`が代表例ですね。

```swift
// イベント発生 → 購読している全員に放送
NotificationCenter.default.post(name: .storeOpen, object: nil)

// 関心のあるオブジェクトがそれぞれ購読
NotificationCenter.default.addObserver(
    self, selector: #selector(handleOpen),
    name: .storeOpen, object: nil)
```

送る側は、受け取る対象が誰なのか知りません。ただ名前(name)を付けて投げるだけです。

受け取る側も、複数がそれぞれ勝手に購読します。

お互いを知らなくてもいいので結合度は低くなりますが、代わりに今誰がこの通知を聞いているのか追跡しづらいという欠点があります。

## 一目でわかるデリゲート vs オブザーバー

言葉だけだと混乱しやすいので、表にまとめてみました。

<figure>
  <img src="/assets/images/posts/84b5ff78-dfd2-4f9b-aa60-f4cd50f3afe1/4-1783847561867.png" alt="1対1の委任と1対Nの放送、図で見れば一目瞭然です">
  <figcaption>1対1の委任と1対Nの放送、図で見れば一目瞭然です</figcaption>
</figure>

| 区分 | デリゲート | オブザーバー |
|------|-----------|--------|
| 通信方式 | 1対1(1人に委任) | 1対N(複数に放送) |
| 相手を知っているか | お互いはっきり知っている | お互い知らなくてもいい |
| 代表例 | UITableViewDelegate | NotificationCenter |
| 応答(戻り値) | 受け取りやすい | 受け取りにくい |
| 結合度 | 比較的高い | 低い |
| 追跡・デバッグ | 容易 | やや難しい |

ポイントは「応答を返してもらう必要があるかどうか」です。

デリゲートは「このセルの高さをいくつにする?」と聞くと、相手が値を返してくれます。会話のやり取りがある構造です。

オブザーバーはただ投げっぱなしです。誰がどう反応するかに、送る側は関心を持ちません。

## では、いつどちらを使えばいい?

実際にプロジェクトで使いながら立てた基準はこうです。

**デリゲートを使うとき**

1. たった1つのオブジェクトとだけやり取りすればいいとき
2. 相手から値を返してもらう必要があるとき(例:セルの数、高さ)
3. 画面遷移のように順序と流れが明確なとき

**オブザーバーを使うとき**

1. 1つの出来事に複数の画面が同時に反応する必要があるとき
2. ログイン状態の変更、ダークモード切り替えのようなアプリ全体のイベントのとき
3. 送る側と受け取る側が互いを知らなくてもいいとき

例えば、ログインに成功したときにホーム画面、マイページ、上部バナーが同時に変わる必要があるなら、デリゲートでは荷が重いです。

1人にしか委任できない構造なので、3箇所すべてに繋げようとするとコードが汚くなってしまいます。

こういうときはオブザーバーで「ログインしたよ!」と一度放送するほうがずっとすっきりします。

逆に「入力欄で確認ボタンを押した」のような1対1の状況では、デリゲートやクロージャが正解です。

## おわりに

似ているように見えても、「何人に知らせるのか」さえ思い浮かべれば選択は簡単になります。

1人ならデリゲート、複数ならオブザーバー。この基準1つだけ押さえておけば十分です。
