「後で必要になりそうだから、先に作っておこうか」

開発をしていると、この誘惑は本当によく訪れます。設定オプションもあらかじめ切り出しておいて、多言語対応の構造も先に組んでおいて、管理画面で変更できるようにテーブルも先に用意しておく、といった具合に。

私自身も以前、「いつか使うだろう」と思って作っておいた機能がありました。2年後にそのコードを消したのは、他の誰でもない自分でした。一度も使われることがなかったからです。

今日取り上げるYAGNIは、まさにこの誘惑に真っ向から「ノー」を突きつける原則です。KISS、DRYと並んで、開発三大原則と呼ばれています。

YAGNIは「You Aren't Gonna Need It」の略です。「それ、どうせ必要にならないよ」という意味です。

エクストリーム・プログラミング(XP)という開発手法から生まれた言葉で、ケント・ベックやロン・ジェフリーズといったXP陣営の人たちが口癖のように使っていた表現が、原則として定着したものです。

核心的な主張はひとつです。

> 機能は実際に必要になったときに実装せよ。必要になりそうだと予想したときではなく。

---

## 先に作っておくと何が問題なのか

「先に作っておけば後で楽じゃないか」という反論がすぐ出てくると思います。問題は、その予測がほとんどの場合外れるということです。

マーティン・ファウラーがこのテーマについてまとめた記事では、先回りして作った機能のコストを4つに分けています。

| コスト | 内容 |
|------|------|
| 実装コスト | 今すぐ使わない機能を作るのにかかった時間 |
| 遅延コスト | その時間で作るべきだった本当に必要な機能が後回しになる |
| 維持コスト | 使わないコードでもテスト・リファクタリング・バグ修正はずっとついてくる |
| 修正コスト | いざ必要になった時点では要件が変わっていて、結局作り直す羽目になる |

一番痛いのが最後です。「後で」が本当にやってきたとき、その「後で」の要件は、自分が想定していた形とほぼ必ず違っています。結局、先に作ったコードを作り直すコストまで二重に払うことになるわけです。

## 現場でYAGNIが破られる瞬間

口では誰もがうなずくのに、実際の現場ではこんな形でそっと入り込んできます。

**1. パラメータを先に用意しておく**

```javascript
// 今はメール通知だけなのに
function sendNotification(user, message, channel = 'email',
  retryCount = 3, priority = 'normal', template = null) {
  // channelが'email'のときだけ動くコード
}
```

呼び出し元は全部`sendNotification(user, msg)`だけです。残りのパラメータは「後でSlack通知を追加するときのために」作ったものですが、Slack通知の要件は結局最後まで来ませんでした。

**2. インターフェースを先に敷いておく**

実装がひとつしかないのに、先にインターフェースを作ってしまうケースです。`UserService`をひとつ作るのと同時に`IUserService`も作るパターンですが、2つ目の実装が登場するまで、そのインターフェースはファイル数を増やすだけの飾りになります。

**3. 「設定に切り出しておけばいいじゃないか」**

ハードコーディングが怖くて、あらゆる値を設定ファイルと管理画面に切り出してしまうケース。実際にはその設定を変更することが年に一度もなく、設定が増えるほど「この値を変えたらどこに影響が出るんだっけ」という疑問だけが増えていきます。

## YAGNIではないもの

誤解を防ぐために、線を引いておきます。YAGNIはこういう意味ではありません。

まず、設計をするなという話ではありません。コードを直しやすく書く努力(良い命名、小さな関数、テスト)は今すぐ価値のあることなので、YAGNIの対象外です。むしろコードが直しやすくなっていてこそ、「必要になったその時」に素早く作れるのです。

そして、取り返しのつかない決定まで先送りしろという話でもありません。データベースの選定やAPIの公開仕様のように、一度決めたら変えにくいものは、あらかじめ検討しておくのが正解です。YAGNIが狙っているのは、「後で簡単に追加できるのに、先に作ってしまう機能」です。

## 私が使っている判断基準

私は「先に作っておこうか」と思ったとき、2つの質問をします。

> この機能が必要だという根拠は、ロードマップにあるのか、それとも自分の想像の中にあるのか?

> 後で追加すると、今より格段にコストがかかるのか?

根拠が想像でしかなく、後で追加してもコストがそれほど変わらないなら、作りません。ほとんどのケースがこれに当てはまることに気づきました。

## まとめ

YAGNI原則から持ち帰るべきことは3つです。

- 機能は必要になったときに作りましょう。必要になりそうなときではなく。
- 先回りして作った機能は、実装・遅延・維持・修正のコストを四重に請求してきます。
- ただし、良い設計習慣と取り返しのつかない決定は、YAGNIの対象ではありません。

KISS、DRYに続いてYAGNIまで並べてみると、この3つの原則は結局ひとつの文章に集約されます。今日必要なものを、シンプルに、一箇所にだけ作れ。

「いつか使うだろう」と思って眠らせている機能がひとつでもあるなら、私のように2年後に自分の手で消すことになるかもしれません。
