「あとで必要になりそうだから、先に作っておこうか」

開発をしていると、この誘惑は本当によくやってきます。設定オプションもあらかじめ切り出しておいて、多言語対応の構造も先に用意しておいて、管理画面から変更できるようにテーブルも先に掘っておく……。

今日取り上げるYAGNIは、まさにこの誘惑に真正面から「NO」を突きつける原則です。KISS、DRYと並んで開発原則の御三家と呼ばれています。

YAGNIは「You Aren't Gonna Need It」の略です。「それ、どうせ必要にならないよ」という意味です。

エクストリーム・プログラミング(XP)という開発方法論から生まれた言葉で、ケント・ベックやロン・ジェフリーズといったXP陣営の人たちが口癖のように使っていた表現が、そのまま原則として定着したものです。

核心の主張はひとつです。

> 機能は実際に必要になったときに実装せよ。必要になりそうだと予想されたときではなく。

---

## 先に作っておくと何が問題なのか

「先に作っておけば、あとで楽じゃないですか?」という反論がすぐに出てきそうですが、問題はその予測がほとんどの場合外れることです。

マーティン・ファウラーがこのテーマについてまとめた記事では、先回りして作った機能のコストを4つに分けています。

| コスト | 内容 |
|------|------|
| 実装コスト | 今は使わない機能を作るのにかかった時間 |
| 遅延コスト | その時間で作るべきだった本当に必要な機能が遅れる |
| 維持コスト | 使わないコードでもテスト・リファクタリング・バグ修正はずっとついてくる |
| 修理コスト | いざ必要になった時点では要件が変わっていて、結局作り直すはめになる |

一番痛いのが最後です。「あとで」が本当にやってきたとき、その「あとで」の要件は自分が想定していた形とはほとんど常に違います。結局、先に作ったコードを作り直すコストまで二重に払うことになるわけです。

---

## 実務でYAGNIが破られる場面

口で言えば誰もがうなずくのに、実戦ではこんな形でこっそり入り込んできます。

**1. パラメータをあらかじめ切り出しておく**

```swift
// 今はメール通知だけなのに
func sendNotification(user: User, message: String, channel: String = "email",
                      retryCount: Int = 3, priority: String = "normal",
                      template: String? = nil) {
    // channelが"email"のときだけ動作するコード
}
```

呼び出し箇所はすべて`sendNotification(user: user, message: msg)`だけです。残りのパラメータは「あとでSlack通知を追加するときに使うつもり」で作ったものですが、Slack通知の要望はついに来ませんでした。

**2. プロトコルをあらかじめ用意しておく**

実装が1つしかないのに、先にプロトコルから作るケースです。`UserService`を1つ作るのと同時に`UserServiceProtocol`を作るパターンですが、2つ目の実装が登場するまで、そのプロトコルはファイル数を増やすだけの飾りにすぎません。

**3.「設定として切り出しておけば便利じゃないですか」**

ハードコーディングを恐れるあまり、すべての値を設定ファイルや管理画面に切り出すケース。実際にはその設定を変える機会は年に一度もなく、設定が増えるほど「この値を変えたらどこに影響するんですか?」という質問ばかりが増えていきます。

## YAGNIではないもの

誤解を防ぐために、線引きをしておきます。YAGNIはこういう意味ではありません。

まず、設計をするなという話ではありません。コードを直しやすく組む努力(良い命名、小さな関数、テスト)は今すぐ価値があることなので、YAGNIの対象ではありません。むしろコードが直しやすい状態にあってこそ、「必要になったその時」に素早く作れるのです。

そして、後戻りしにくい決定まで先送りしろという話でもありません。データベースの選定やAPIの公開仕様のように、一度決めると変更が難しいものは事前に検討しておくのが正解です。YAGNIが狙い撃ちしているのは、「あとで簡単に追加できるのに、先に作ってしまう機能」です。

---

## 私が使っている判断基準

私は「先に作っておこうか」と迷ったとき、2つのことを自問します。

> この機能が必要だという根拠は、ロードマップにあるのか、それとも自分の想像の中にあるのか?

> あとで追加すると、今作るよりもはるかにコストが高くなるのか?

根拠が想像でしかなく、あとで追加してもコストがさほど変わらないなら、作りません。ほとんどの場合、これに当てはまるんですよね。

## まとめ

YAGNI原則から持ち帰るべきことは3つです。

- 機能は必要になったときに作りましょう。必要になりそうなときではなく。
- 先回りして作った機能は、実装・遅延・維持・修理のコストを四重に請求してきます。
- ただし、良い設計習慣と後戻りしにくい決定はYAGNIの対象ではありません。

KISS、DRYに続いてYAGNIまで並べてみると、この3原則は結局ひとつの文章に集約されます。今日必要なものを、シンプルに、一箇所にだけ作れ。

「いつか使うかも」と思って眠らせたままの機能がひとつでもあるなら、2年後に自分の手で消すことになるかもしれません。
