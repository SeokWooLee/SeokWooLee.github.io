「後で必要になりそうだから、先に作っておきましょうか?」

開発をしていると、この誘惑は本当によく訪れます。設定オプションもあらかじめ切り出しておいて、多言語対応の構造も先に組んでおいて、管理画面で変更できるようテーブルも先に用意しておいて。

私にも昔、「いつか使うだろう」と思って作っておいた機能がありました。2年後にそのコードを削除したのは、他でもない私自身でした。一度も使われなかったんです。

今日取り上げるYAGNIは、まさにこの誘惑に真っ向から「ノー」と答える原則です。KISS、DRYと並んで開発原則の三本柱と呼ばれています。

YAGNIは「You Aren't Gonna Need It」の略です。「それ、どうせ必要にならないよ」という意味です。

エクストリーム・プログラミング(XP)という開発手法から生まれた言葉で、ケント・ベックやロン・ジェフリーズといったXP陣営の面々が口癖のように使っていた表現が、原則として定着したものです。

核心的な主張はひとつです。

> 機能は実際に必要になったときに実装せよ。必要になりそうだと予想したときではなく。

---

## 先に作っておくと何が問題なのか

「先に作っておけば後で楽じゃないですか?」という反論がすぐ出てきそうですが、問題はその予測がほとんど外れるということです。

マーティン・ファウラーがこのテーマについてまとめた記事では、先に作った機能のコストを4つに分けています。

| コスト | 内容 |
|------|------|
| 実装コスト | 今すぐ使わない機能を作るのにかかった時間 |
| 遅延コスト | その時間にあれば作れたはずの本当に必要な機能が遅れる |
| 維持コスト | 使わないコードでもテスト・リファクタリング・バグ修正はずっとついて回る |
| 修理コスト | いざ必要になった時点では要件が変わっていて、結局作り直しになる |

一番きついのが最後です。「いつか」が本当にやって来たとき、その「いつか」の要件は自分が想定していた形とほぼ必ず違います。結局、先に作ったコードを作り直すコストまで二重に払うことになるわけです。

## 実務でYAGNIが崩れる場面

言葉にすると誰もがうなずくのに、現場ではこんな形でこっそり忍び込んできます。

**1. パラメータをあらかじめ切り出しておく**

```javascript
// 今はメール通知だけなのに
function sendNotification(user, message, channel = 'email',
  retryCount = 3, priority = 'normal', template = null) {
  // channelが'email'のときだけ動作するコード
}
```

呼び出し元は全部 `sendNotification(user, msg)` だけです。残りのパラメータは「後でSlack通知を追加するときに使おうと思って」作ったものですが、Slack通知の要望はついに来ませんでした。

**2. インターフェースを先に敷いておく**

実装が1つしかないのに、インターフェースから作ってしまうケースです。`UserService`をひとつ作るのと同時に`IUserService`も作るパターンですが、2つ目の実装が登場するまで、そのインターフェースはファイル数を増やすだけの飾り物です。

**3. 「設定に切り出しておけばいいじゃないですか」**

ハードコーディングが怖くて、あらゆる値を設定ファイルや管理画面に切り出してしまうケース。実際にはその設定を変えることが年に一度もなく、設定が増えるほど「この値を変えたらどこに影響するの?」という疑問だけが増えていきます。

## YAGNIではないもの

誤解を防ぐために線引きをしておきます。YAGNIはこういう意味ではありません。

まず、設計をするなという意味ではありません。コードを直しやすく書く努力(良い命名、小さな関数、テスト)は今すぐ価値があることなので、YAGNIの対象にはなりません。むしろコードが直しやすくあってこそ、「必要になったその時」に素早く作れるのです。

そして、後戻りしにくい決定まで先送りせよという意味でもありません。データベースの選定やAPIの公開仕様のように、一度決めたら変えにくいものは前もって検討するのが正解です。YAGNIが狙っているのは「後から簡単に追加できるのに、あらかじめ作ってしまう機能」です。

---

## 私が使っている判断基準

私は「先に作っておこうか?」と迷ったとき、2つのことを自問します。

> この機能が必要だという根拠は、ロードマップにあるのか、それとも自分の想像の中にあるだけなのか?

> 後から追加すると、今より格段にコストがかかるのか?

根拠が想像で、後から追加してもコストがさほど変わらないなら、作りません。ほとんどの場合、これに当てはまります。

## まとめ

YAGNI原則から持ち帰るべきことは3つです。

- 機能は必要になったときに作りましょう。必要になりそうなときではなく。
- あらかじめ作った機能は、実装・遅延・維持・修理のコストを四重に請求してきます。
- ただし、良い設計習慣と後戻りしにくい決定はYAGNIの対象ではありません。

KISS、DRYに続いてYAGNIまで並べてみると、3つの原則は結局ひとつの文章に集約されます。今日必要なものを、シンプルに、一箇所にだけ作れ。

「いつか使うだろう」と思って眠らせたままの機能がひとつでもあるなら、私のように2年後に自分の手で削除することになるかもしれません。
