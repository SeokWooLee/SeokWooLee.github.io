オーバーロードとオーバーライド。ここまで名前が似ている必要があったのかと思う2つの概念です。

試験や面接の定番ですが、混同しやすいのは名前のせいが大きいです。動作の仕組みは実はまったく違います。片方は「同じ名前の関数を複数用意すること」、もう片方は「継承した関数を改めて定義し直すこと」です。

決まるタイミングも違います。片方はコンパイル時、もう片方は実行時です。この違いが静的ディスパッチ・動的ディスパッチというパフォーマンスの話にもつながっていきます。

この記事ではSwiftの例ではっきり切り分けておきます。

まずは要点のまとめです。

1. オーバーロード（Overloading）：同じ名前で、異なる引数を持つ関数を複数定義すること。継承とは無関係
2. オーバーライド（Overriding）：親のメソッドを子が再定義すること。継承が前提
3. オーバーロードはコンパイル時に、オーバーライドは実行時にどの関数を呼ぶかが決まる
4. オーバーライドこそがポリモーフィズムの核となる仕組み

---

## オーバーロード：同じ名前を複数用意する

オーバーロードとは、名前は同じでも引数の構成（数・型・ラベル）が異なる関数を複数定義することです。

```swift
func area(radius: Double) -> Double { .pi * radius * radius }
func area(width: Double, height: Double) -> Double { width * height }
func area(side: Double) -> Double { side * side }
```

呼び出し側で渡された引数を見て、コンパイラがどの関数を使うか決定します。area(radius: 3)と書いた瞬間に、1つ目の関数に確定します。実行する前にすでに決まっているわけです。

この仕組みがなかったC言語を思い出すと、そのありがたみが分かります。絶対値を求める関数がabs、labs、fabsと型ごとに名前が違っていました。オーバーロードは「概念が同じなら名前も揃える」ための仕組みなのです。

Swiftはprint(_:)、min(_:_:)のような標準ライブラリのあちこちでオーバーロードを使っています。演算子のオーバーロードにより、+ ひとつで整数の足し算、文字列の連結、配列の結合をすべて処理できたりもします。

1つ注意点があります。戻り値の型だけが異なるオーバーロードは避けたほうがいいです。Swiftでは文法上可能な場合もありますが、呼び出し側で型を明示していないと曖昧さのエラーになりますし、読む側も混乱します。

---

## オーバーライド：継承したものを再定義する

オーバーライドとは、親クラスから継承したメソッドを、子クラスが同じシグネチャで再定義することです。継承関係が前提になります。

```swift
class Animal {
    func speak() { print("...") }
}
class Dog: Animal {
    override func speak() { print("ワンワン") }
}
class Cat: Animal {
    override func speak() { print("ニャー") }
}
```

核心となるのはこの場面です。

```swift
let animals: [Animal] = [Dog(), Cat()]
for animal in animals {
    animal.speak() // ワンワン、ニャー
}
```

変数の型はAnimalなのに、実際に実行されるのはインスタンス本来のメソッドです。「どのspeakを呼ぶか」が、実行時に実際のオブジェクトを見て決まるわけです。これが動的ディスパッチであり、ポリモーフィズムが機能する仕組みです。

Swiftがoverrideキーワードを強制していることも押さえておく価値があります。うっかり親のメソッドと同じシグネチャを作ってしまったり、タイプミスで再定義に失敗したりする事故を、コンパイラが防いでくれます。逆に親クラス側でfinalを付ければ、再定義を禁止することもできます。

## 表でまとめて比較

| 区分 | オーバーロード | オーバーライド |
|---|---|---|
| 定義 | 同じ名前、異なる引数 | 同じシグネチャを再定義 |
| 継承 | 不要 | 必須 |
| 決定タイミング | コンパイル時（静的） | 実行時（動的） |
| 目的 | 名前の一貫性、APIの利便性 | ポリモーフィズム、振る舞いの置き換え |
| Swiftのキーワード | なし | override（必須）、finalで禁止可能 |

決定タイミングの違いはパフォーマンスにもつながります。コンパイル時に確定する呼び出し（静的ディスパッチ）はそのままジャンプすればいいのですが、実行時に決まる呼び出し（動的ディスパッチ）は実際の型のメソッドテーブルを経由する必要があります。Swiftではfinalやprivateを付けることで、コンパイラが動的ディスパッチを静的に変換し、最適化できる余地が生まれます。

## 混同しないための覚え方

英単語の意味そのままで覚えればいいです。

**Overload = 過積載**。同じ名前の上に関数を複数「積み込む」ことです。名前は1つ、機能は複数。

**Override = 上書きして踏み越える**。親の定義を「上書きする」ことです。定義は1つ、中身を差し替える。

面接の回答はこう始めればいいでしょう。

> 「オーバーロードは引数が異なる同じ名前の関数を複数用意することで、コンパイル時に決定されます。オーバーライドは継承したメソッドを再定義することで、実行時に実際の型を基準に決定されます。ポリモーフィズムを生み出すのはオーバーライドの方です」

---

## まとめ

- オーバーロード：同じ名前で、異なる引数構成の関数を複数定義する。継承は不要
- オーバーライド：親のメソッドを子が同じシグネチャで再定義する。継承が必須
- オーバーロードはコンパイル時（静的）、オーバーライドは実行時（動的）に呼び出し対象が決まる
- ポリモーフィズムの核となる仕組みはオーバーライドと動的ディスパッチである
- Swiftはoverrideキーワードを強制することでミスを防ぎ、finalで再定義を禁止できる
- 覚え方：Overloadは名前に過積載、Overrideは定義を上書き
