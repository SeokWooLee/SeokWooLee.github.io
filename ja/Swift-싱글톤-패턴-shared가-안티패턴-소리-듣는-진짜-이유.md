Swift開発をしていると、`static let shared` の一行で作るシングルトンに本当によく出会います。

私も最初にiOSアプリを作ったとき、ネットワークマネージャー、ユーザーセッション、キャッシュ管理を全部シングルトンで埋め尽くしていました。

楽なんですよね。どこからでも `Manager.shared` を呼べばいいので。

ただ、プロジェクトが大きくなるにつれて妙なことが起き始めました。テストができません。画面を一つ直しただけなのに、思わぬところでバグが出てくるんです。

そのとき初めて「ああ、シングルトンがアンチパターン扱いされるのには理由があるんだな」と感じました。

この記事では、Swiftのシングルトンパターンとは正確には何なのか、なぜ乱用するとアンチパターン扱いされるのか、それでもどんな場面なら使ってよいのかを、実際の経験ベースで整理してみます。

結論から言うと、シングルトン自体が悪いのではなく、「どこからでも自由に状態を変更できるように開放してしまうやり方」が問題なんです。だからsharedをむやみに乱用すると、テストが詰まり、依存関係が隠れ、並行処理の問題まで生まれてしまいます。

## Swiftのシングルトンパターン、そもそも何なのか

シングルトンとは、「アプリ全体でインスタンスがただ一つだけ存在することを保証する」デザインパターンです。

Swiftでは本当に短く書けます。これが魅力であり、落とし穴でもあります。

```swift
final class NetworkManager {
    static let shared = NetworkManager()  // アプリ内でただ一つ
    private init() {}                     // 外部からの生成をブロック
    func request(_ url: URL) { /* ... */ }
}
```

`static let` はSwiftがスレッドセーフに一度だけ初期化してくれます。なので、別途ロックをかけなくてもインスタンス生成自体は安全です。

`private init()` で外部から新たに作れないようにするのも重要なポイントです。これが抜けていると、それはただのグローバルオブジェクトであって、シングルトンとは呼べません。

ここまでは実にすっきりしています。問題は、これをどこからでも呼べてしまう点から始まります。

---

## sharedインスタンスがなぜアンチパターン扱いされるのか

最もよく指摘される理由を三つ、経験した順に説明します。

一つ目、テストが地獄になります。

`NetworkManager.shared` をコードの中で直接呼んでしまうと、テストの際にこれを偽物(モック)のオブジェクトに差し替える方法がありません。

実際のサーバーにリクエストが飛んでしまったり、テスト同士で状態を共有してしまい、順番が変わっただけで結果が変わってしまいます。

二つ目、依存関係が隠れてしまいます。

ある関数が内部でこっそり `UserSession.shared` を使っていると、関数のシグネチャだけを見てもその関数が何を必要としているのか分かりません。

共同開発の際、これは本当に恐ろしいです。コードを全部開いて見ないと依存関係が見えないからです。

三つ目、グローバルな可変状態の並行処理問題です。

シングルトンが内部にプロパティを持っていて、複数のスレッドが同時にそれを読み書きすると、データレースが発生します。

インスタンス生成が安全だからといって、その中の状態変更まで安全なわけではありません。この二つを混同すると、クラッシュに遭遇することになります。

まとめるとこうなります。

> シングルトンが批判される本当の理由は「一つしか存在しないから」ではなく、「どこからでも取り出して使え、どこからでも変更できてしまうから」です。

## では、シングルトンは絶対に使ってはいけないのか?

いいえ。私は今も特定の状況では使っています。

そもそも一つだけ存在するのが自然な対象があります。例えば `UserDefaults.standard`、`FileManager.default`、`URLSession.shared` のように、Appleも標準ライブラリでシングルトンを使っています。

基準はこう考えると楽です。

- 状態をほとんど変更せず、読み取り中心である → シングルトンでもOK
- アプリ全体で本当に一つでなければならない → シングルトンを検討
- テストでは異なる動作をさせる必要がある → 依存性注入を推奨

核心は「sharedをコードの中で直接呼ぶのではなく、外から注入してもらおう」ということです。

<figure>
  <img src="/assets/images/posts/b71a69ec-a38f-40af-8715-df457527f22f/3-1783847709763.png" alt="sharedはデフォルト値として、テストには偽物を注入する構造">
  <figcaption>sharedはデフォルト値として、テストには偽物を注入する構造</figcaption>
</figure>

同じシングルトンを使うにしても、こう変えるだけでテスト可能性がぐっと上がります。

```swift
protocol Networking { func request(_ url: URL) }
extension NetworkManager: Networking {}

final class FeedViewModel {
    private let network: Networking
    init(network: Networking = .shared) {  // デフォルト値はシングルトン、テストではモックを注入
        self.network = network
    }
}
```

普段はデフォルト値としてシングルトンを使うので楽ですし、テストのときは偽物のオブジェクトを入れられるので柔軟です。

シングルトンを捨てるのではなく、呼び方だけを変えたわけです。

---

## 実務で私が守っている3つのルール

最後に、私が現場で自分なりに定めた基準を共有します。

1. 可変状態を複数の場所で変更するシングルトンは作らない。状態が必要なら所有者を明確にする。
2. `shared` を関数の中で直接呼ばず、コンストラクタやパラメータで注入してもらう。
3. 同時アクセスがあるシングルトンはactorにするか、直列キューで保護する。

この三つさえ守れば、「シングルトンのせいでプロジェクトが腐る」状況はほぼ起きません。

シングルトンは間違った道具ではなく、簡単すぎて乱用されやすい道具です。

楽だからといって、どこにでも `shared` をばらまく前に、「これ、後でテストできるかな?」と一度だけ自問してみてください。その問い一つが、半年後の自分を救います。
