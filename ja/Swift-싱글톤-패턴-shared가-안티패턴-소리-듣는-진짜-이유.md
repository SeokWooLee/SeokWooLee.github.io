Swiftで開発していると、`static let shared`の一行で作るシングルトンに本当によく出会います。

私も初めてiOSアプリを作ったとき、ネットワークマネージャー、ユーザーセッション、キャッシュ管理を全部シングルトンで埋め尽くしていました。

楽なんですよね。どこからでも`Manager.shared`と呼べば済むので。

ところがプロジェクトが大きくなるにつれ、おかしなことが起き始めました。テストができないんです。画面を一つ直しただけなのに、思わぬところでバグが爆発する。

そのとき初めて「ああ、シングルトンがアンチパターン呼ばわりされるのには理由があるんだな」と実感しました。

この記事では、Swiftのシングルトンパターンとは正確に何なのか、なぜ乱用するとアンチパターン扱いされるのか、それでもどんなときなら使ってよいのかを、実体験ベースで整理してみます。

結論から言うと、シングルトン自体が悪いのではなく、「どこからでも状態を書き換えられるように開けっぱなしにする方式」が問題なんです。だからsharedを無分別に乱用すると、テストが詰まり、依存関係が隠れ、さらには並行処理の問題まで生まれるわけです。

## Swiftのシングルトンパターン、そもそも何なのか

シングルトンとは「アプリ全体でインスタンスがただ一つだけ存在することを保証する」デザインパターンです。

Swiftでは本当に短く書けます。これが魅力であり、同時に落とし穴でもあります。

```swift
final class NetworkManager {
    static let shared = NetworkManager()  // アプリにたった一つ
    private init() {}                     // 外部からの生成をブロック
    func request(_ url: URL) { /* ... */ }
}
```

`static let`はSwiftがスレッドセーフにたった一度だけ初期化してくれます。だから、別途ロックを設けなくてもインスタンス生成自体は安全です。

`private init()`で外部から新しく作れないようにするのも肝心な点です。これが抜けていると、それは単なるグローバルオブジェクトであって、シングルトンとは言えません。

ここまではとてもすっきりしています。問題は、これをどこからでも呼び出せるという点から始まります。

---

## sharedインスタンスがなぜアンチパターンと言われるのか

最もよく指摘される理由を、私が実際に経験した順に説明します。

まず一つ目、テストが地獄になります。

`NetworkManager.shared`をコードの中で直接呼び出すと、テスト時にこれを偽物(モック)オブジェクトに差し替える方法がありません。

実際のサーバーにリクエストが飛んでしまったり、テスト同士で状態を共有してしまい、実行順序が変わっただけで結果が変わってしまいます。

二つ目、依存関係が隠れてしまいます。

ある関数が内部でこっそり`UserSession.shared`を使っていると、関数のシグネチャだけを見てもその関数が何を必要としているのか分かりません。

共同作業をする際、これは本当に厄介です。コードを全部開いてみないと依存関係が見えないからです。

三つ目、グローバルな可変状態による並行処理の問題です。

シングルトンが内部にプロパティを持っていて、複数のスレッドが同時にそれを読み書きすると、データ競合が起きます。

インスタンス生成は安全でも、その中の状態変更まで安全というわけではありません。この二つを混同すると、クラッシュに出くわすことになります。

> シングルトンが批判される本当の理由は「一つしか存在しないから」ではなく、「どこからでも取り出して使え、どこからでも変更できるから」です。

## では、シングルトンは絶対に使ってはいけないのか?

いいえ。私も今なお特定の状況では使っています。

そもそも一つしか存在しないのが自然な対象があります。例えば`UserDefaults.standard`、`FileManager.default`、`URLSession.shared`のように、Appleも標準ライブラリでシングルトンを使っています。

判断基準はこう置くと分かりやすいです。

- 状態をほとんど変更せず、読み取り中心である → シングルトンで問題なし
- アプリ全体で本当に一つでなければならない → シングルトンを検討
- テストで異なる挙動をさせる必要がある → 依存性注入を推奨

ポイントは「sharedをコードの中で直接呼び出すのではなく、外から注入してもらおう」ということです。

同じシングルトンを使うにしても、こう変えるだけでテスト可能性がぐっと上がります。

```swift
protocol Networking { func request(_ url: URL) }
extension NetworkManager: Networking {}

final class FeedViewModel {
    private let network: Networking
    init(network: Networking = .shared) {  // デフォルト値はシングルトン、テストではmockを注入
        self.network = network
    }
}
```

普段はデフォルト値としてシングルトンを使うので楽で、テストの際にはモックオブジェクトを入れられるので柔軟です。

シングルトンを捨てるのではなく、呼び出し方だけを変えたわけです。

## 実践で私が守っているルール3つ

最後に、私が実務で自分なりに決めている基準を共有します。

1. 可変状態を複数箇所から変更するシングルトンは作らない。状態が必要なら所有者を明確にする。
2. `shared`を関数の中で直接呼び出さず、コンストラクタやパラメータで注入してもらう。
3. 同時アクセスが発生するシングルトンはactorにするか、直列キューで保護する。

この三つさえ守れば、「シングルトンのせいでプロジェクトが腐る」状況はほとんど起きません。

シングルトンは間違った道具ではなく、あまりに手軽なせいで乱用されやすい道具です。

楽だからという理由でどこにでも`shared`をばら撒く前に、「これ、後でテストできるかな?」と一度だけ自問してみてください。その一つの問いが、半年後の自分を救います。
