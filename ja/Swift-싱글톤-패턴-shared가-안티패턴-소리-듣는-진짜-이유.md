Swift開発をしていると、`static let shared`の一行で作るシングルトンに本当によく出会います。

最初にiOSアプリを作るときは、ネットワークマネージャー、ユーザーセッション、キャッシュ管理まで全部シングルトンで埋め尽くしてしまいがちです。

楽なんですよね。どこからでも`Manager.shared`を呼べばいいので。

ところがプロジェクトが大きくなると、おかしなことが起き始めます。テストができません。画面を一つ直しただけなのに、思わぬところでバグが爆発したりします。

シングルトンがアンチパターンと言われるのには、それなりの理由があります。

この記事では、Swiftのシングルトンパターンとは正確には何なのか、なぜ乱用するとアンチパターン扱いされるのか、それでもどんなときなら使ってもいいのかを、実務的な視点で整理してみます。

結論から言うと、シングルトン自体が悪いわけではなく、「どこからでも自由に状態を変更できる状態にしておく方式」が問題なのです。だからsharedを無分別に乱用すると、テストが詰まり、依存関係が隠れ、並行処理の問題まで発生するわけです。

## Swiftのシングルトンパターン、一体何なのか

シングルトンは「アプリ全体でインスタンスがただ一つだけ存在することを保証する」デザインパターンです。

Swiftでは本当に短く書けます。これが魅力であり、落とし穴でもあります。

```swift
final class NetworkManager {
    static let shared = NetworkManager()  // アプリにたった一つ
    private init() {}                     // 外部からの生成を禁止
    func request(_ url: URL) { /* ... */ }
}
```

`static let`はSwiftがスレッドセーフに一度だけ初期化してくれます。だから別途ロックを使わなくても、インスタンス生成自体は安全です。

`private init()`で外部から新規作成できないようにするのもポイントです。これがないと、ただのグローバルオブジェクトであって、シングルトンではありません。

ここまで見ると実にすっきりしています。問題は、これをどこからでも呼び出せるという点から始まります。

---

## sharedインスタンスがなぜアンチパターンと言われるのか

最もよく指摘される理由を3つ、体感した順に解説します。

1つ目、テストが地獄になります。

`NetworkManager.shared`をコードの中で直接呼び出すと、テストのときにこれを偽物(モック)オブジェクトに差し替える方法がありません。

実際のサーバーにリクエストが飛んでしまったり、テスト同士で状態を共有してしまい、実行順序が変わるだけで結果が変わったりします。

2つ目、依存関係が隠れてしまいます。

ある関数が内部でこっそり`UserSession.shared`を使っていると、関数のシグネチャだけを見ても、この関数が何を必要としているのか分かりません。

チームで作業するとき、これは本当に厄介です。コードを全部開いて初めて依存関係が見えてくるからです。

3つ目、グローバルな可変状態による並行処理の問題です。

シングルトンが内部にプロパティを持っていて、複数のスレッドが同時にそれを読み書きすると、データレースが発生します。

インスタンス生成が安全だからといって、その中の状態変更まで安全というわけではありません。この2つを混同すると、クラッシュに見舞われることになります。

まとめるとこうなります。

> シングルトンが叩かれる本当の理由は、「一つしか存在しないから」ではなく、「どこからでも取り出せて、どこからでも変更できるから」です。

## では、シングルトンは絶対に使ってはいけないのか

いいえ。今でも特定の状況では使います。

そもそも一つだけ存在するのが自然な対象があります。例えば`UserDefaults.standard`、`FileManager.default`、`URLSession.shared`のように、Appleも標準ライブラリでシングルトンを使っています。

判断基準はこう考えると分かりやすいです。

- 状態をほとんど変更せず、読み取り中心である → シングルトンで問題なし
- アプリ全体で本当に一つでなければならない → シングルトンを検討
- テストで異なる挙動をさせる必要がある → 依存性注入を推奨

ポイントは「sharedをコードの中で直接呼び出すのではなく、外から注入してもらう」ということです。

<figure>
  <img src="/assets/images/posts/b71a69ec-a38f-40af-8715-df457527f22f/3-1783847709763.png" alt="sharedはデフォルト値として、テストにはモックを注入する構造">
  <figcaption>sharedはデフォルト値として、テストにはモックを注入する構造</figcaption>
</figure>

同じシングルトンを使うにしても、こう変えるとテスト可能性がぐっと上がります。

```swift
protocol Networking { func request(_ url: URL) }
extension NetworkManager: Networking {}

final class FeedViewModel {
    private let network: Networking
    init(network: Networking = .shared) {  // デフォルト値はシングルトン、テストではmockを注入
        self.network = network
    }
}
```

普段はデフォルト値としてシングルトンを使うので楽で、テストのときは偽物のオブジェクトを渡せるので柔軟です。

シングルトンを捨てるのではなく、呼び出し方だけを変えたわけです。

## 実務で守っている3つのルール

最後に、現場で自分なりに定めている基準を共有します。

1. 可変状態を複数の場所から変更するシングルトンは作らない。状態が必要な場合は、所有者を明確にする。
2. `shared`を関数の中で直接呼び出さず、コンストラクタやパラメータで注入してもらう。
3. 同時アクセスがあるシングルトンはactorにするか、直列キューで保護する。

この3つさえ守れば、「シングルトンのせいでプロジェクトが腐る」という状況はほとんど起こりません。

シングルトンは間違った道具ではなく、あまりにも簡単に使えるがゆえに乱用されやすい道具です。

楽だからという理由でどこにでも`shared`をばらまく前に、「これ、後でテストできるだろうか」と一度だけ自問してみてください。その一つの問いが、半年後の自分を救います。
