コードの中で必要なオブジェクトがあると、その場でそのまま生成して使うことが多いですよね。ところがテストコードを書こうとした瞬間に壁にぶつかります。クラスの中にがっちり埋め込まれた生成コードのせいで、どうしても差し替えができないんです。

依存性注入(DI)とは何かを調べても「制御の反転がどうの、IoCコンテナがどうの」といった説明ばかりで、かえって混乱しやすいところがあります。

そこで今日は、できるだけシンプルな形で解きほぐしてみようと思います。まず結論からお伝えしますね。

依存性注入(DI)とは、結局のところクラスの中で直接作っていたオブジェクトを、外で作って渡してあげることです。たったこの一つの移動が本質です。オブジェクトを生成する場所をクラスの外に出すこと。それがDIのほぼすべてと言っていいでしょう。

この記事を最後まで読めば、DIがなぜ必要なのか、コードはどう変わるのか、そしてSwinjectのようなDIライブラリがなぜこれを代わりにやってくれるのかまで、感覚がつかめるはずです。

## オブジェクト生成がクラスの中にあると何が問題なのか

例えば、注文を処理するクラスがあるとしましょう。このクラスは決済のために決済モジュールを必要とします。

最もよく見かける初心者的なコードはこんな形です。

```swift
class OrderService {
    // クラスの中で直接決済オブジェクトを生成
    private let pay = KakaoPay()

    func order() { pay.pay() }
}
```

問題は、このOrderServiceがKakaoPayにぴったりくっついてしまっていることです。

KakaoPayの代わりに別の決済サービスに切り替えたくなったらどうでしょうか。クラスの中に入って生成部分を書き直さなければなりません。

テストの際に偽の決済オブジェクトを差し込みたくても、方法がありません。すでに内部で本物のKakaoPayを作ってしまっているからです。

つまり、オブジェクト生成をクラスの中に置いておくと、そのクラスは特定の実装に手足を縛られてしまうのです。

---

## そこでオブジェクト生成を外に出してみました

解決策は思ったよりあっさりしています。直接作らずに、外で作ってもらって受け取るだけでいいんです。

```swift
class OrderService {
    private let pay: Pay
    // 外で作られたものを受け取るだけ
    init(pay: Pay) { self.pay = pay }

    func order() { pay.pay() }
}
```

変わったのはたった一つです。直接生成が消えて、イニシャライザ(コンストラクタ)で受け取る形になりました。

これでOrderServiceは、決済がKakaoPayなのか別のサービスなのかを知らなくても構いません。ただ「Payというものを受け取って使う」ということだけを知っていればいいのです。

> DIの本質は大げさなものではありません。オブジェクト生成をクラスの外に移し、何を渡すかを決める権限を外側に渡すこと、それだけです。

こうして外からオブジェクトを渡してあげることを「注入する(injection)」と呼びます。依存するオブジェクトを注入する、だから依存性注入なんですね。

名前が難しかっただけで、やっていることはたった今見た内容がすべてです。

---

## DIをするとどんなメリットがあるのか

言葉だけで「良い」と言われてもピンとこないので、実際に何が変わるのか整理してみました。

- 差し替えが簡単になります: KakaoPayを別の決済サービスに変えても、OrderServiceは一行も修正しません。渡すオブジェクトだけ変えればいいのです。
- テストが楽になります: 本物の決済の代わりに偽の(モック)決済オブジェクトを渡すことで、お金を使わずにロジックだけ検証できます。
- 役割が分離されます: OrderServiceは「注文処理」だけに集中し、「どの決済を使うか」は外側が決めます。

個人的には三つ目が一番実感できました。クラスが自分の仕事だけに集中することで、コードがぐっと読みやすくなるんです。

まとめるとこうなります。

| 区分 | 生成が内部にある場合 | 生成を外に出した場合(DI) |
|------|------|------|
| 決済の切り替え | クラスの修正が必要 | 渡すオブジェクトだけ差し替え |
| テスト | 偽オブジェクトを差し込めない | モック注入が可能 |
| 関心事 | 生成とロジックが混在 | ロジックだけに集中 |

## ではSwinjectのようなDIライブラリはなぜ必要なのか

ここまで来ると自然に湧いてくる疑問があります。「外で作って渡すのはわかったけど、その『外』は誰が管理するの?」というものです。

オブジェクトの数が少なければ手作業で作って渡せば済みます。ですが実務のプロジェクトではオブジェクトが何百個もあり、互いに絡み合っています。

これを人間が一つひとつ順序を合わせて作って渡すのは地獄です。

そこでこの「外で作って渡す作業」を代わりにやってくれるツールが登場します。Swift界隈のSwinjectやFactoryのようなDIコンテナがまさにその役割です。

必要なオブジェクトを登録しておけば、コンテナが勝手に作って必要な場所に渡してくれます。自分で直接生成する場面がほとんどなくなるわけです。

よく言われる「制御の反転(IoC)」もここから出てきます。オブジェクトを作って結びつける制御権が、自分のコードからコンテナへと移った、という意味です。

言ってみればDIは概念であり、DIコンテナはその概念を代わりに実行してくれる働き者なのです。

## よくある質問

Q. DIとIoCは同じ意味ですか?
いいえ。IoC(制御の反転)は「制御権を渡す」という広い原則で、DIはそれを実現する具体的な方法の一つです。DIはIoCの一種と考えればいいでしょう。

Q. コンストラクタ以外の注入方法もありますか?
はい。コンストラクタ注入、プロパティ注入、メソッド注入があります。ただしSwiftではコンストラクタ注入が推奨されます。依存関係がletで固定され値が変わらず、テストもしやすいからです。

Q. ライブラリがないとDIとは言えませんか?
そんなことはありません。先ほど手作業でコンストラクタに渡したものも立派なDIです。ライブラリはそれを自動化してくれるだけです。

DIが難しく感じられたなら、一つだけ覚えておいてください。「クラスの中で作らず、外で作って渡そう。」この一文さえあれば、あとは自然についてきます。

テストコードを前にして手が止まった経験があるなら、今日見た例を実際に自分の手で書き換えてみることをおすすめします。目で見るだけより、はるかにはっきりと感覚がつかめるはずです。
