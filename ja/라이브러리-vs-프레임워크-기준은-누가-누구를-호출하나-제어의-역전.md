Alamofireはライブラリと呼ばれ、SwiftUIはフレームワークと呼ばれます。でもどちらも「他人が作ったコードを使わせてもらう」という点では同じですよね。何が違うのでしょうか。

「フレームワークのほうが大きいものだ」という答えをよく耳にしますが、大きさは本質ではありません。小さなフレームワークもあれば、巨大なライブラリもあるからです。

本当の判断基準はただひとつです。**誰が誰を呼ぶか。**

この基準ひとつを押さえておくと、制御の反転(IoC)という重要な概念にも自然につながります。依存性注入を学ぶときにも再び出会う概念なので、ここでしっかり理解しておくと得です。

要点をまとめます。

1. ライブラリ:自分のコードが必要なときに呼び出す道具の集まり。制御権は自分にある
2. フレームワーク:流れをフレームワークが握り、自分のコードが「呼ばれる」骨組み。制御権はフレームワークにある
3. この逆転した呼び出し方向を制御の反転(Inversion of Control)と呼ぶ
4. ハリウッドの原則:「こちらから連絡はしません、あちらから連絡します」

---

## ライブラリ:自分が呼ぶ道具

ライブラリは特定の機能をあらかじめ実装しておいたコードのまとまりです。いつ、どんな順序で使うかは、すべて自分のコードが決めます。

```swift
// 流れの主導権は自分のコードにある
let json = try JSONDecoder().decode(User.self, from: data)
let hash = SHA256.hash(data: input)
```

プログラムの開始から終了までの流れ全体を自分で組み立て、その途中で必要な道具を取り出して使います。工具箱からドライバーを取り出して使うようなものです。ドライバーが作業の順序を決めるわけではありませんよね。

Alamofire、Kingfisherといったものがこのカテゴリに入ります。いつネットワークリクエストを送るか、いつ画像を読み込むかは自分が決めます。

---

## フレームワーク:自分を呼ぶ骨組み

フレームワークはアプリケーション全体の構造と実行の流れをあらかじめ用意しておいた骨組みです。自分はその骨組みが空けておいた場所にコードを埋め込んでいきます。

iOSアプリを思い浮かべるとわかりやすいです。アプリの起点、イベントループ、画面のライフサイクルはすべてUIKit・SwiftUIが握っています。自分が書くのはviewDidLoad、body、onAppearのような、**フレームワークが決まったタイミングで呼び出してくれる**コードです。

```swift
struct ProfileView: View {
    var body: some View {   // 自分では呼び出さない
        Text("Hello")       // SwiftUIが必要なときに呼び出す
    }
}
```

bodyを自分の手で呼び出すことはありません。いつ何回呼ばれるかはSwiftUIが決めます。流れの主導権が入れ替わっているわけです。

## 制御の反転、そしてハリウッドの原則

この逆転した関係には名前がついています。制御の反転(IoC、Inversion of Control)です。

通常の手続き型プログラムでは、自分のコードが制御の流れを握って外部のコードを呼び出します。フレームワークベースのプログラムでは、フレームワークがその流れを握り、自分のコードは登録しておいたポイントで呼び出される側になります。制御権が逆になったので「反転」なのです。

これを機知に富んだ表現で言い表したのがハリウッドの原則(Hollywood Principle)です。

> "Don't call us, we'll call you." — こちらから連絡はしません、あちらから連絡します。

オーディションを受けた俳優が制作会社に電話をかけまくるのではなく、キャスティングが決まったら制作会社側から連絡が来る構造。フレームワークと自分のコードの関係もまさにこれです。

デリゲートパターンも同じ原理です。UITableViewDataSourceを実装するとき、cellForRowAtを自分で呼び出すでしょうか。いいえ、違います。テーブルビューが必要なときに自分を呼び出すのです。フレームワークの世界の文法があちこちのコードに染み込んでいるわけです。

---

## では実務では何が変わるのか

**学習方法が違います**。ライブラリは「どんな機能があるか」を調べればよいですが、フレームワークは「いつ自分が呼ばれるのか」(ライフサイクル・呼び出し規約)を先に理解する必要があります。UIKitの学習がライフサイクルの学習から始まるのはこのためです。

**乗り換えコストが違います**。ライブラリは呼び出し部分だけ直せば別のものに乗り換えられますが、フレームワークはコード全体がその骨組みの上に乗っているため、乗り換えは事実上の書き直しになります。UIKitからSwiftUIへの移行が関数を数個変える程度の作業では済まない理由です。

**テスト戦略が違います**。フレームワークが呼び出してくれるコードは、フレームワークなしで単独実行するのが難しいものです。だからこそ、ロジックをフレームワーク非依存の層(純粋なSwift)に切り出せというアドバイスが繰り返し出てくるわけです。

## 面接では一文で

> 「ライブラリは自分のコードが呼び出す道具で、フレームワークは制御の流れを握ったまま自分のコードを呼び出す骨組みです。この制御権の方向の違いを制御の反転と呼びます。」

次に来る質問は「IoCの例を挙げてみて」(ライフサイクルメソッド、デリゲート)、「それがなぜ良いのか」(流れを標準化し、開発者はビジネスロジックに集中できる)という順で続きます。

---

## まとめ

- 区別の基準は大きさではなく、呼び出しの方向である
- ライブラリ:自分のコードが必要なときに呼び出す道具。制御権は自分にある
- フレームワーク:流れを握り、決まったタイミングで自分のコードを呼び出す骨組み。制御権はフレームワークにある
- この逆転した制御権が制御の反転(IoC)、別名ハリウッドの原則
- viewDidLoad・body・デリゲートメソッドは、すべて「呼び出される」側のコードである
- 実務上の違い:フレームワークはライフサイクルの学習が先に必要で、乗り換えが難しく、ロジックの分離がテストの要になる
