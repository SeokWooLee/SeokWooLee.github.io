SwiftUIの紹介ドキュメントの冒頭には必ず「宣言型(declarative)フレームワーク」という言葉が出てきます。ReactもJetpack Composeも、自分たちを宣言型と呼んでいますよね。

ところが、いざ「宣言型って何ですか?」と聞かれると答えに詰まってしまいます。「コードがスッキリしていること」は正解ではありません。

宣言型と命令型の違いは、一文で言い表せます。**「どうやって(How)」を書くか、「何を(What)」を書くか。**

この記事では、この基準を日常の例で押さえたうえで、UIKitとSwiftUIのコードで確認し、最後に宣言型がタダではないという話まで扱います。

要点をまとめておきます。

1. 命令型: 望む結果にたどり着くまでの手順を段階的に指示する — How
2. 宣言型: 望む結果が何かを記述し、手順はシステムに任せる — What
3. SQL、HTML、map/filter、SwiftUIが宣言型の代表例
4. 宣言型UIの本質: 「状態がこうならば画面はこうなる」を宣言すれば、状態変化に応じた更新はフレームワークが行う

---

## タクシーに例えると

命令型は、運転手さんに道を直接案内するようなものです。「右に曲がってください、300メートル直進、信号で左折…」と手順を一つひとつ指示し、その手順の積み重ねが目的地到着という結果を生みます。

宣言型は「江南駅までお願いします」です。目的地(What)だけを伝えて、経路選択(How)は運転手さんとナビに任せます。

どちらも目的地には着きます。違うのは、**自分が手順を持つのか、それとも結果の記述だけを持つのか**という点です。

---

## コードでは同じことを2通りの方法で書く

偶数だけを選んで二乗するコードを、2つの方法で書いてみます。

```swift
// 命令型: どう巡回してどこに格納するかを自分で指示
var result: [Int] = []
for n in numbers {
    if n % 2 == 0 {
        result.append(n * n)
    }
}

// 宣言型: 何を求めているかだけを記述
let result = numbers.filter { $0 % 2 == 0 }.map { $0 * $0 }
```

命令型のバージョンには、ループ変数や中間配列、順序といった「手順の部品」がそのまま露出しています。宣言型のバージョンには「偶数を絞り込んで二乗したもの」という意図だけが残ります。巡回方法はfilterとmapの内部に隠れています。

もっと身近な宣言型も、実はすでに使っているはずです。SQLは「この条件に合う行をください」とだけ書き、インデックス探索の方法までは書きません。HTMLも「ここにタイトル、ここに段落」と構造だけを宣言し、レンダリングの手順は書きませんよね。

---

## UIになると違いが劇的に大きくなる

命令型UI(UIKit)は、画面を**変更する手順**を書きます。

```swift
// UIKit: 状態が変わるたびに画面操作を直接指示
func updateBadge(count: Int) {
    if count > 0 {
        badgeLabel.isHidden = false
        badgeLabel.text = "\(count)"
    } else {
        badgeLabel.isHidden = true
    }
}
```

この方式の難しさは、「今の画面がどんな状態なのか」を開発者がずっと追い続けなければならない点にあります。更新の経路が複数あると、どれか一つを漏らしやすく、それがまさに「データは変わったのに画面はそのまま」というバグになります。

宣言型UI(SwiftUI)は、**状態に対する画面の姿**を宣言します。

```swift
// SwiftUI: countがこうならば画面はこうなる、を宣言
struct BadgeView: View {
    let count: Int
    var body: some View {
        if count > 0 {
            Text("\(count)").badgeStyle()
        }
    }
}
```

countが変わったとき画面をどう直すかは、自分では書きません。SwiftUIが以前の宣言と新しい宣言を比較し、必要な部分だけを更新します。UIを「状態の関数」にしたわけです。更新手順の所有権をフレームワークに渡した以上、漏れる更新コードそのものが存在しません。

## 宣言型はタダではない

バランスを取るために、反対側も見ておく必要があります。

**手順を隠した代償として、手順が必要な場面ではもどかしさが残ります**。「スクロールをこのオフセットに正確に動かして」といった命令型の要求は、宣言型フレームワークではかえって迂回路が必要になります。

**デバッグの勝手が違います**。命令型は手順をたどればいいのですが、宣言型では「なぜ再描画されたのか」といった、フレームワークの判断そのものを追跡する必要があります。隠れたHowをまったく知らなくていいわけではないのです。

**パフォーマンスチューニングも結局は内部理解に戻ってきます**。SwiftUIでビューが過剰に再計算されているとき、それを解決するにはdiffingと依存関係トラッキングがどう動いているかを知る必要があります。

ですから正確な視点は「宣言型が優れている」ではなく、**抽象化レベルの選択**です。手順の所有権を手放せばコードは意図中心にシンプルになりますが、その代わりに細かな制御と内部理解が課題として残ります。

## まとめ

- 命令型は結果にたどり着く手順(How)を指示し、宣言型は望む結果(What)を記述する
- SQL・HTML・map/filterは、実はすでに馴染みのある宣言型である
- 命令型UIは画面を変更する手順を書くため、状態と画面の不整合バグが起きやすい
- 宣言型UIは「状態がこうならば画面はこうなる」を宣言し、更新はフレームワークが担う(UI = 状態の関数)
- 宣言型の代償: 細かな手順の制御が難しく、フレームワーク内部の動作理解が結局必要になる
- 本質は優劣ではなく、手順の所有権を誰が持つかという選択である
