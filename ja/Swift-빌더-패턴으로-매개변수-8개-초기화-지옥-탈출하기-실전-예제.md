Swiftでアプリを作っていると、ある瞬間に初期化コードがこうなっていることがあります。

`User(name:age:email:phone:address:isVerified:profileURL:createdAt:)`

括弧の中に引数が8個。

呼び出し側のコードを見ると、どの値がemailでどれがphoneなのか、目で一つずつ数えることになりますよね。

個人開発でもよく出くわす状況ですが、Swiftのビルダーパターンで整理するとすっきり解決します。

結論から言うと、引数の多い初期化はビルダーパターンで「1行に1つずつ」設定する形に変えると、ぐっと読みやすくなります。順序を覚える必要も、使わない値に無理やりnilを詰め込む必要もなくなるからです。

## 引数8個のinit、なぜ地獄なのか

引数が3〜4個のときは、実はそれほど問題になりません。

問題は数が増えてから始まります。

まず、順序ミスが起きます。email の位置に phone を入れても、両方 String なら コンパイラは検知してくれません。

次に、任意値の扱いが煩雑になります。今すぐ使わない値まで `nil, nil, nil` と埋めなければなりません。

さらに、呼び出し部分が長くなり、何のオブジェクトを作っているのか一目で分かりにくくなります。

> 引数が多いほどinitは「埋めるべき空欄の答案用紙」になり、空欄が増えるほどミスも一緒に増えていきます。

phone と profileURL の位置を入れ替えてしまうバグは、丸一日以上かかって発見されることもあります。型が同じだと、この手のミスは本当に目立ちません。

## ビルダーパターンとは何か

ビルダーパターンは、複雑なオブジェクトを一度に作るのではなく、値を一つずつ埋めていって最後に完成させる方式です。

初期化の時点ですべての値を渡す代わりに、必要な値だけを選んで設定し、`build()` のようなメソッドで最終的なオブジェクトを受け取る構造だと考えてください。

レストランで注文するときを思い浮かべると分かりやすいです。材料を一度にまとめて言うのではなく、「パンは全粒粉、チーズは追加、ソースは抜きで」というように項目ごとに伝えるイメージですね。

値を設定するメソッドが自分自身を返すようにすると、点をつなぐようにチェーンで書けます。以下は最もシンプルな形のビルダーです。

```swift
final class UserBuilder {
    private var name = ""
    private var email: String?
    // 値を設定してselfを返すことでチェーンを可能にする
    func setName(_ v: String) -> Self { name = v; return self }
    func setEmail(_ v: String) -> Self { email = v; return self }
    func build() -> User { User(name: name, email: email) }
}
```

設定メソッドが `Self` を返すので、`.setName(...).setEmail(...)` のようにつなげて書けます。

## Swiftでビルダーパターンをどう使うか

実際の呼び出し部分がどう変わるのか、一番気になるところだと思います。

先ほど作ったビルダーを使うと、オブジェクト生成コードはこう変わります。

```swift
// 必要な値だけ、ラベルを付けて順序を気にせず設定
let user = UserBuilder()
    .setName("イ・ソグ")
    .setEmail("hello@example.com")
    .build()
// 使わない値にnilを詰める必要もない
```

見ての通り、順序を気にする必要がありません。メソッド名自体がラベルになるので、email の位置に phone を入れてしまうミスが構造的に防げます。

使わない値は単に呼び出さなければいいだけです。

Swiftではクラスの代わりに構造体(struct)でビルダーを作ることもありますし、クロージャを渡して1つのブロック内で設定する方式もよく使われます。プロジェクトのスタイルに合わせて選んでください。

## ビルダーパターン、いつ使い、いつ避けるか

良さそうだからといって何にでも導入すると、かえってコードが増えるだけになります。

筆者の基準を表にまとめてみました(2026年時点での個人プロジェクトの経験に基づきます)。

| 状況 | 推奨 |
|---|---|
| 引数5個以上、任意値が多い | ビルダーパターン |
| 同じ型の引数が複数ある | ビルダーパターン |
| 引数2〜3個、すべて必須 | 通常のinit |
| 値がシンプルで固定的 | 通常のinit |

まとめると、引数が少なくすべて必須であれば、あえてビルダーを作る理由はありません。

逆に、任意値が入り混じった大きなオブジェクトなら、ビルダーの方が確実に扱いやすくなります。

**Q. ビルダーを作るとコードがかえって増えませんか?**

その通りです。ビルダークラス自体はコードが増えます。ただ、そのオブジェクトを作る呼び出し箇所が複数あるなら、呼び出し側がすっきりする恩恵の方が大きくなります。

**Q. Swiftにはデフォルト引数がありますが、それでは足りませんか?**

引数が3〜4個程度なら、デフォルト引数(default parameter)だけで十分です。数が多く組み合わせが多様なときにビルダーの利点が生きてきます。

---

引数8個のinitにストレスを感じていたなら、まさにその箇所だけビルダーパターンを一度適用してみてください。すべてを作り直す必要はなく、一番複雑なオブジェクト一つから始めれば負担も少なく済みます。今日も楽しいコーディングを!
