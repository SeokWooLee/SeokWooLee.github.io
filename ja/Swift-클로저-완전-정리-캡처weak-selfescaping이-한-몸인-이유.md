Swiftのコードにおいてクロージャは空気のような存在です。ソート条件、ネットワーク完了ハンドラ、ボタンのアクション、SwiftUIのbodyまで、波括弧のブロックを渡すコードは一日に何十回も登場します。ところが「クロージャが値をキャプチャするとは正確にどういうことか」「[weak self]はなぜ使うのか」「@escapingはなぜ付けるのか」を説明してみてと言われると、言葉に詰まることが少なくありません。

この3つの問いは実は一体です。クロージャが周囲の値を捕まえる仕組み(キャプチャ)を理解すれば、参照型である理由も、循環参照が生まれる理由も、escaping表示が必要な理由も芋づる式に解けます。Swift基礎シリーズ第2回となる今回は、その繋がりを順に見ていきます。

## クロージャとは — 名前より大事なのは「捕まえる」という性質

まずは文法を最小限に整理しておきましょう。クロージャは実行コードのブロックを値のように扱う文法です。変数に入れたり、引数として渡したり、返り値にしたりできます。

```swift
let add: (Int, Int) -> Int = { a, b in a + b }
add(2, 3) // 5
```

実は`func`で宣言する関数も、名前付きのクロージャです。Swiftでは関数とクロージャは同じ仲間で、`{ }`という文法は名前を付けずにその場で作るバージョンにすぎません。

ですがクロージャ(closure)という名前は「コードブロック」ではなく別の性質から来ています。周囲の変数を包み込んで閉じ込める(close over)という性質です。

```swift
func makeCounter() -> () -> Int {
    var count = 0
    return {
        count += 1
        return count
    }
}

let counter = makeCounter()
counter() // 1
counter() // 2
counter() // 3
```

不思議なことが起きています。`count`はmakeCounterのローカル変数なので関数が返った時点で消えるはずですが、返されたクロージャを呼ぶたびに増え続けます。クロージャが自分の体の外にある変数`count`をキャプチャして、関数が終わった後も生かし続けているのです。これがクロージャの本質であり、この記事の残り全部がこのキャプチャの派生話になります。

## キャプチャの正確な意味 — コピーではなく参照

Swiftのクロージャのデフォルトのキャプチャ方式は参照です。値をコピーして持っていくのではなく、その変数自体との繋がりを保ち続けます。だからこういう結果になります。

```swift
var multiplier = 2
let times = { (n: Int) in n * multiplier }

times(10)      // 20
multiplier = 3
times(10)      // 30 — クロージャは変わった値を見ている
```

クロージャを作った時点のmultiplier(2)ではなく、実行時点のmultiplier(3)が使われています。クロージャは変数のスナップショットではなく、変数そのものを持ち歩いているのです。

作った時点の値を固定したいならキャプチャリストを使います。

```swift
let times = { [multiplier] (n: Int) in n * multiplier }

times(10)      // 20
multiplier = 3
times(10)      // 20 — 作った時点の2に固定される
```

角括弧の中に書いた変数は、クロージャ生成時点でコピーされ、クロージャ内に定数として封じ込められます。まとめるとこうです。デフォルトは参照キャプチャ(生きた繋がり)、キャプチャリストは値キャプチャ(生成時点のスナップショット)。[weak self]を説明する前に、この区別をまず押さえておく必要があります。

そしてこのキャプチャの保管場所があるからこそ、クロージャは参照型なのです。キャプチャされた変数はクロージャと寿命を共にする必要があるためヒープに保存され、クロージャの値をコピーするとその保管場所を共有する参照が一つ増える、という構造です。struct中心のSwiftにおいてクロージャがクラスのように振る舞うのはこのためです。値型と参照型の区別に馴染みがない場合は、値型優先主義の回を先に読んでおくとよいでしょう。

## 循環参照と[weak self] — キャプチャが引き起こす代表的な事故

参照キャプチャの代償こそが循環参照です。構図は常に同じです。

```swift
class ProfileViewModel {
    var onUpdate: (() -> Void)?
    var name = ""

    func bind() {
        onUpdate = {
            print("名前: \(self.name)")
        }
    }
}
```

ビューモデルはプロパティonUpdateでクロージャを強く所有しています。ところがそのクロージャはself、つまりビューモデルを参照キャプチャしています。ビューモデル → クロージャ → ビューモデルという所有の輪が閉じてしまい、ARCの世界ではこの二つは互いを掴んだまま永遠に解放されません。画面を閉じてもビューモデルがメモリに残るリークの完成です。

その解決策がキャプチャリストの2つ目の使い方です。`[weak self]`はselfを弱参照でキャプチャせよという指示です。クロージャがselfを掴みはしても所有はしないので、輪が断ち切られます。代わりにselfが先に解放される可能性があるため、クロージャの中でselfはオプショナルになり、通常は`guard let self else { return }`から始めることになります。オプショナルの回で見た早期リターンのパターンがここで再び登場するわけです。

```swift
onUpdate = { [weak self] in
    guard let self else { return }
    print("名前: \(self.name)")
}
```

重要なのは、[weak self]が万能な接頭辞ではないという点です。循環の条件は「selfがクロージャを所有し、そのクロージャがselfをキャプチャする」ことです。この条件がなければweakも不要です。例えば`DispatchQueue.main.asyncAfter`に渡すクロージャは実行後にシステムが破棄するので、selfが少し長く生きるだけでリークにはなりません。反射的にweakを付けるのではなく、「このクロージャを誰がどれだけ長く所有するのか」を問うのが正確な習慣です。NSTimerのように所有構造が特殊で有名になった事例は別の記事で扱いました。

## @escaping — クロージャが関数より長生きするとき

最後のパズルのピースです。関数のパラメータとして受け取るクロージャには、時々@escapingが付きます。

```swift
func fetchUser(completion: @escaping (User) -> Void) {
    URLSession.shared.dataTask(with: url) { data, _, _ in
        let user = parse(data)
        completion(user)   // 関数が返ってからずっと後に実行される
    }.resume()
}
```

区別の基準は実行のタイミングです。関数が返る前に内部で全て実行して捨てるクロージャはnon-escaping(デフォルト)で、プロパティに保存したり非同期処理に渡したりして関数が返った後に実行される可能性があるものはescaping、つまり関数の外へ脱出するクロージャです。

なぜこの区別を強制するのでしょうか。脱出するかどうかでコンパイラと開発者双方の計算が変わるからです。non-escapingは関数の実行中しか生きていないことが保証されているため、コンパイラがキャプチャの保存を最適化でき、循環参照の心配も根本的にありません。mapやfilterに渡すクロージャでselfを気にしなくていいのはそのためです。一方escapingはクロージャがどこかに保存されて長く生きるという意味なので、上で見た循環参照の検討が必要な候補になります。@escapingという表示は「このクロージャは長生きするのでキャプチャに気をつけろ」というAPIレベルの警告ラベルというわけです。

ちなみにcompletionハンドラ形式のescapingクロージャは、async/awaitの導入以降新しいコードでは減少傾向にありますが、既存APIを読み解いてブリッジするには依然として正確に理解しておくべき概念です。

## 身につけるべき3つの基準

理論を実務基準に圧縮すると3つになります。

**第一に、クロージャを見たらまず寿命を問います。** このクロージャは関数の中で消費されて終わるのか(non-escaping)、どこかに保存されて長く生きるのか(escaping)。この問い一つでキャプチャをどれだけ気にすべきかが決まります。

**第二に、weakは反射ではなく判断で使います。** 所有の輪が実際に閉じる箇所(プロパティとして保存するハンドラ、デリゲート的なコールバック)には[weak self]、輪がない一回限りの実行には不要。判断に迷ったらInstrumentsのLeaksやdeinitのログで確認する習慣が答えです。

**第三に、キャプチャリストで意図を文書化します。** 値を固定したいなら[value]、所有しないなら[weak self]。キャプチャリストはパフォーマンスの仕組みである以前に、「このクロージャが外の世界とどう繋がっているか」をコードに明記するドキュメントです。

## まとめ

- クロージャの本質はコードブロックではなくキャプチャ、つまり周囲の変数を包み込んで閉じ込め、寿命を延ばす性質です。
- デフォルトのキャプチャはコピーではなく参照のため、クロージャは実行時点の値を見ます。キャプチャリスト[x]は生成時点の値で固定します。
- キャプチャの保管場所を共有する必要があるため、クロージャは参照型です。
- selfが所有するクロージャがselfをキャプチャすると循環参照になり、[weak self] + guard let selfが標準的な解決策です。ただし所有の輪がなければweakも不要です。
- @escapingは「関数より長生きするクロージャ」という警告ラベルであり、キャプチャの検討が必要な箇所を示してくれます。
