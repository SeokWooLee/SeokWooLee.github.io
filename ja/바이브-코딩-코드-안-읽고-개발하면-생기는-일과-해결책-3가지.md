AIが書いてくれたコードを読まずにAccept Allだけを押しながら開発する方式、いわゆる「バイブコーディング」です。立ち上がりが驚くほど速く、一度味わうと元には戻れないという声も多いですが、コードを読まない代償がどこでどう請求されるのかは、事前に知っておいた方がいいでしょう。

この記事では、バイブコーディングの正確な定義とメリット、コードを読まないと実際に起きる問題3つ、全部読まなくても事故を減らせる解決策3つを順に整理します。

まず結論から見ていきます。

> バイブコーディングは立ち上がりを10倍速くしてくれますが、コードを読まなければ、その速さを維持コストとして後で払い戻すことになります。

プロトタイプや自分だけで使うツールなら十分試す価値があります。ただ、他人にサービスを提供したり長く保守したりするコードとなると話は変わってきます。なぜそうなるのか、一つずつ紐解いていきましょう。

## バイブコーディングとは正確には何なのか

まず用語から押さえておきます。「バイブコーディング(vibe coding)」は2025年2月、アンドレイ・カルパシがXに投稿した言葉から始まりました。

核心はこうです。AIに自然言語でやりたいことを伝えるとAIがコードを書いてくれて、人はそのコードを細かく読まない。

カルパシ本人も「私はいつもAccept Allを押して、もうdiffは読まない」と語っています。エラーが出たら、そのメッセージをそのままコピーしてAIに投げ返すだけだそうです。

ちなみにこの単語は2025年、コリンズ英語辞典の「今年の言葉」にも選ばれました。

バイブコーディングの本質は「AIコーディング」というより「コードを読まないこと」に近いです。AIを使いながらもdiffを全部読んでいるなら、それは単なるAI補助開発であってバイブコーディングではありません。

---

## メリットは明確です

バイブコーディングが人気を集めているのには理由があります。

**第一に、立ち上がりの速さ。** 一日単位でかかっていたプロジェクト初期のセットアップが数十分単位に短縮されます。画面に何かが表示されるまでの時間が劇的に短くなります。

**第二に、苦手分野の補完。** バックエンドエンジニアが言葉でCSSレイアウトを説明して作らせるように、慣れていない領域の参入障壁が大きく下がります。

**第三に、心理的ハードルの低下。** 「とりあえず作ってみよう」という気持ちが大きくなり、先延ばしにしていたアイデアを実際に形にできます。

問題は、この満足感が通常プロジェクト初期、コード量が少ないうちしか続かないことです。

---

## コードを読まないと本当に起きること

コードベースがある程度大きくなると、共通して三つの問題が現れます。

**1. バグを直せなくなります。** エラーが出たらそのままAIに丸投げする方式は、AIが直せなくなった瞬間に一緒に行き詰まります。コードを読んだことがないので、どこに問題があるのか見当がつかないのです。

**2. 同じ機能があちこちに生まれます。** AIは以前書いたコードを覚えていないことが多く、似たような関数を何度も新しく作ってしまいます。後で一つを直しても、残りはそのまま放置されがちです。

**3. 静かにセキュリティの穴が空きます。** 代表的なのが、APIキーがクライアントコードにそのまま埋め込まれるケースです。AIは指示された通りにしか動かず、人間もチェックしないためです。

三つ目が一番怖いです。例えばこんなコードです。

```swift
// AIが書いたコード — キーがクライアントに露出している
let apiKey = "sk-live-abc123"  // アプリのバイナリにそのまま入る
let url = URL(string: "https://api.example.com?key=\(apiKey)")!
URLSession.shared.dataTask(with: url).resume()
```

こういうコードは、公開した瞬間に誰でもキーを見られるようになります。diffを読まなければ、こういうミスは見逃してしまいます。

---

## ではどうすればいいのか？解決策3つ

幸い、「全部読め」に戻らなくても事故を大きく減らす方法があります。

**解決策1. 致命的な3か所だけは自分で読む。** 全部は読めなくても、事故が起きると取り返しがつかない部分だけは決めておいて読むのです。

```swift
// 最低限この3つは自分で確認する
// 1) 認証・キー関連のコード
// 2) 決済・お金に関わるロジック
// 3) ユーザーデータを削除・変更する部分
```

この三つさえ押さえておけば、致命的な事故の確率がぐっと下がります。

**解決策2. コードの代わりにAIの要約を読む。** コードを読みたくないなら、せめて「今書いたコードのセキュリティ・お金・データ削除に関するリスクだけ要約して」ともう一度指示して、コードの代わりにその要約だけでも読みましょう。このとき、コードを書いたセッションではなく新しいセッション、できれば別のAIにレビューさせると、自分が書いたコードをかばう偏りも減らせます。

**解決策3. 人が読まないなら機械にでも読ませる。** gitleaksのようなシークレットスキャナーをpre-commitフックに一つ仕込んでおけば、上で見たAPIキーの露出はコミットの段階で自動的に検出されます。リンターとテストをCIに組み込むのも同じ原理です。

三つの解決策に共通するのはこれです。読むコストは減らしつつ、検証をゼロにはしない。

## 結局、バイブコーディングはやるべきなのか、やめるべきなのか

結論は「状況による」です。

| 状況 | おすすめ度 | 理由 |
|---|---|---|
| プロトタイプ・デモ | 積極的におすすめ | すぐ作って捨ててもいい |
| 個人利用のツール | おすすめ | 壊れても困るのは自分だけ |
| サイドプロジェクト | 条件付き | 核心ロジックは読む |
| 本番サービス・チーム開発 | 非推奨 | 維持コストが速さを食いつぶす |

ポイントは「コードをまったく読まないか、重要な部分だけは読むか」の違いです。

Q. 初心者がバイブコーディングで開発を学んでもいいですか？
A. 作る楽しさを感じるにはいい方法です。ただ、コードをまったく読まないと実力はなかなか伸びません。作った後にAIへ「このコード、なぜこう書いたの？」と聞きながら読む習慣を添えることをおすすめします。

Q. AIが書いたコード、どこまで信じていいですか？
A. 動くことと安全であることは別の話です。動くことはAIを信じていいとしても、安全で保守可能かどうかは人間か別の検証手段で確認する必要があります。

---

まとめると「速さはAIに、判断は自分に」です。全部読む必要はありませんが、まったく読まなければいつか代償を払うことになります。上の三つの解決策のうち一つでも今日のプロジェクトに仕込んでおけば、その代償を大きく減らせます。

もう一歩進めると、そもそもコードを指示する前に要件をドキュメントとして先に確定させる仕様駆動開発(SDD)という方法論もあります。バイブコーディングの対極にあるアプローチですが、これは次の記事で詳しく取り上げます。
