Swiftでアプリを作っていると、必ず一度は突き当たる悩みがあります。

「依存性注入って、結局どの方式でやるのが正しいんだろう？」

結論から言いますね。

> 特別な理由がない限り、**コンストラクタ（init）インジェクション**を使うのが正解にいちばん近いです。

なぜそうなのか、そして残り二つの方式はどんな時に使うのかを、実際の経験をもとに解説していきます。

## 三つのインジェクション方式、まずは骨格だけ整理

混乱する前に、まず骨組みを押さえておきましょう。Swiftで依存性を渡す方法は大きく三つあります。

1. **コンストラクタ（init）インジェクション** — `init` のパラメータで依存性を受け取る
2. **プロパティインジェクション** — `var` プロパティに後から代入する
3. **メソッドインジェクション** — 専用メソッドで注入する

まずは最も推奨されるコンストラクタインジェクションのコードです。

```swift
final class OrderService {
    private let repo: MemberRepository // letで不変性を保証

    init(repo: MemberRepository) { // コンストラクタ（init）インジェクション
        self.repo = repo
    }
}
```

外部ライブラリなしで、Swiftの標準構文だけで完結するのでコードがすっきりしています。

一方、プロパティインジェクションはこんなに短く書けます。

```swift
final class OrderService {
    var repo: MemberRepository! // 一行で楽そうに見えますが…
}
```

見た目だけならプロパティインジェクションの方が手軽そうです。ただ、この手軽さが後で足を引っ張ることになります。

---

## プロパティインジェクションはなぜ避けた方がいいのか？

実際に起きたことから話しましょう。

プロパティインジェクションはテストを書くときに本当に厄介です。オブジェクトを生成した後に注入を忘れるとオプショナルの強制アンラップでクラッシュしますし、型を見ただけでは何を渡せば完成形になるのか分かりません。

コンストラクタインジェクションなら `OrderService(repo: mockRepo)` のように、純粋なSwiftコードでモックオブジェクトをそのまま渡せます。

二つ目の問題は `let` が使えないことです。

コンストラクタインジェクションならプロパティを `let` で宣言できるので、一度注入されたら変わらない不変オブジェクトになります。

逆にプロパティインジェクションは `var` なので値をいつでも書き換えられ、ミスの余地が生まれます。

三つ目は循環参照の問題です。

AがBを、BがAを互いに参照していると、プロパティインジェクションではアプリが実行されるまでそれに気づけません。該当の画面に入って初めて落ちます。

コンストラクタインジェクションならオブジェクトを生成する時点で問題がすぐ表面化する（設計が破綻していればそもそもコンパイルが通らない）ので、問題をはるかに早く発見できます。

## ひと目でわかる比較表

言葉だけだと混乱しやすいので、表にまとめました。（2026年時点のSwiftコミュニティの推奨傾向）

| 項目 | コンストラクタ（init）インジェクション | プロパティインジェクション | メソッドインジェクション |
|------|------------|------------------|------------------|
| 不変性（let） | 可能 ⭕ | 不可 ❌ | 不可 ❌ |
| テストのしやすさ | 高い | 低い | 普通 |
| 循環参照の検知 | 生成時点で即座に | 発見が遅れる | 発見が遅れる |
| 必須／任意の依存性 | 必須向き | 区別が曖昧 | 任意向き |
| コードの簡潔さ | 普通 | 非常に簡潔 | 普通 |

表を見るだけで傾向がわかりますよね。

コンストラクタインジェクションはほとんどの項目で優位に立っています。だからSwinjectやFactoryといったDIライブラリのドキュメントでも、コンストラクタインジェクションが基本として推奨されているのです。

## では残り二つの方式はいつ使うのか？

コンストラクタインジェクションだけを使えという話ではありません。それぞれに合った場面があります。

**メソッドインジェクション**は任意の依存性によく合います。

注入する対象がなくても構わない場合、つまりあれば使うしなくても問題ない依存性なら、`configure(with:)` のようなメソッドで柔軟に渡すことができます。

**プロパティインジェクション**にも使いどころがあるとすれば、Storyboardで作られたビューコントローラのように、初期化のタイミングを自分たちで制御できない場面くらいです。

通常のアプリケーションコードでは、できるだけ避けた方がいいでしょう。

まとめるとこうなります。

- 必須の依存性 → コンストラクタインジェクション
- 任意の依存性 → メソッドインジェクション
- 通常のコードでのプロパティインジェクション → 避ける

## よくある質問

**Q. FactoryのようなDIライブラリを使うと、コンストラクタインジェクションがもっと楽になりますか？**

はい。SwinjectやFactoryに依存性の作り方を登録しておけば、コンストラクタに渡すオブジェクトをコンテナが代わりに生成してくれます。コードが短くなる一方で、`let` による不変性のようなメリットはそのまま維持できます。

**Q. コンストラクタのパラメータが多くなりすぎたら？**

それはインジェクション方式の問題ではなく、そのクラスが仕事を抱えすぎているというサインです。責務を分けてクラスを分割することをまず検討してみてください。

**Q. 小規模なプロジェクトでもDIライブラリは必須ですか？**

いいえ。画面数個程度の規模なら、`init` で直接渡す純粋なコンストラクタインジェクションだけで十分です。

最初はプロパティインジェクションの手軽さに惹かれがちですが、テストを書いたりチームで開発したりするうちに、コンストラクタインジェクションのメリットがなぜあれほど強調されるのか、身をもって実感することになります。

迷っているなら、まずはコンストラクタインジェクションから始めてみてください。コードが大きくなるほど、後になってありがたみを感じる選択です。
