让AI帮忙写代码、自己却不读代码、只顾着点Accept All推进开发的方式，如今被称为“Vibe Coding”（氛围编程）。这种做法上手速度快得惊人,一旦尝到甜头就很难回头，但不读代码所要付出的代价究竟会在何时、以何种方式反噬回来，最好提前心里有数再用。

本文将依次梳理Vibe Coding的准确定义与优点、不读代码时实际会出现的三个问题，以及即便不逐行阅读代码也能减少事故的三个解决办法。

先看结论。

> Vibe Coding能让项目启动速度提升十倍，但如果不读代码，这份速度迟早会以维护成本的形式被原样吐回去。

如果是做原型或者只给自己用的工具，完全可以放心一试。但如果是要面向用户上线、需要长期维护的代码，情况就不一样了。下面就来逐条拆解原因。

## Vibe Coding到底是什么？

先把术语讲清楚。“Vibe Coding”这个说法始于2025年2月Andrej Karpathy在Twitter上的一条发言。

核心是这样的：用自然语言告诉AI自己想要什么，由AI来写代码，而人不去仔细阅读那些代码。

Karpathy本人也说过：“我总是直接点Accept All，不再去看diff。”一旦报错，他就把错误信息原样复制粘贴丢回给AI。

顺带一提，这个词还被2025年的柯林斯词典（Collins Dictionary）评为“年度词汇”。

Vibe Coding的本质与其说是“AI编程”，不如说更接近于“不读代码”。如果一边用AI一边把diff全都读一遍，那其实就只是AI辅助开发，算不上真正的Vibe Coding。

---

## 优点是显而易见的

Vibe Coding之所以能流行起来,是有原因的。

**第一，启动速度。** 原本要花上一整天的项目初期搭建工作，如今能缩短到几十分钟。从零到屏幕上出现内容的时间被大幅压缩。

**第二，弥补短板领域。** 比如后端开发者只需用语言描述需求，就能让AI生成CSS布局代码，不熟悉领域的入门门槛因此大大降低。

**第三，心理门槛的下降。** “先做出来再说”的心态被放大，原本一直搁置的想法也因此真正被付诸实践。

问题在于，这种满足感通常只能维持到项目初期、代码量还不大的阶段。

---

## 不读代码，真的会出问题

当代码库膨胀到一定规模后，通常会出现以下三个共性问题。

**1. 无法修复bug。** 一旦报错就直接丢给AI处理的方式，只要AI也修不好，事情就会卡在那里。因为从未读过代码，根本无从判断问题出在哪里。

**2. 同一功能在多处重复出现。** AI往往记不住之前写过的代码，于是不断重新造出类似的函数。日后修改其中一处时，其余的地方就被原封不动地遗留下来。

**3. 悄无声息地留下安全漏洞。** 最典型的例子就是API密钥被直接硬编码进客户端代码。AI只会照做，而人又不去核对确认。

第三点最令人担忧。举个例子，比如下面这样的代码。

```swift
// AI写的代码 —— 密钥暴露在客户端
let apiKey = "sk-live-abc123"  // 原样被打包进App二进制文件
let url = URL(string: "https://api.example.com?key=\(apiKey)")!
URLSession.shared.dataTask(with: url).resume()
```

这类代码一旦发布上线，任何人都能看到密钥。如果不读diff，这种问题很容易被漏掉。

---

## 那该怎么办？3个解决办法

好在，不必回到“必须全部读一遍”的老路，也有能大幅减少事故的方法。

**解决办法1：只亲自阅读三处最致命的地方。** 即使做不到全读，也可以事先划定一旦出事就难以挽回的部分，只读这几处。

```swift
// 至少要亲自确认以下三类代码
// 1) 认证、密钥相关代码
// 2) 支付、涉及资金的逻辑
// 3) 删除或修改用户数据的部分
```

哪怕只顾好这三处，发生重大事故的概率也会大幅下降。

**解决办法2：读AI的摘要，而不是代码本身。** 如果实在不想读代码，至少可以再让AI补一句：“把刚才写的代码里涉及安全、资金、数据删除的风险总结一下”，然后去读那份摘要，而不是原始代码。这时最好换一个新的会话，甚至换一个不同的AI来做审查，而不是让写代码的那个会话自己审查——这样能减少AI偏袒自己所写代码的倾向。

**解决办法3：人不愿意读，那就让机器来读。** 在pre-commit钩子里挂一个像gitleaks这样的密钥扫描工具，前面提到的API密钥泄露问题就能在提交阶段被自动拦下。把linter和测试挂在CI上，也是同样的道理。

这三个解决办法的共同点是：把“阅读”的成本降下来，但不能把“校验”降为零。

---

## 所以，Vibe Coding到底该不该用？

结论是“看情况”。

| 场景 | 推荐程度 | 理由 |
|---|---|---|
| 原型、演示 | 强烈推荐 | 做得快、扔了也无妨 |
| 个人自用工具 | 推荐 | 出问题也只影响自己 |
| 副业项目 | 有条件推荐 | 核心逻辑要读 |
| 正式上线服务、团队协作 | 不推荐 | 维护成本会把速度优势吃掉 |

关键区别在于“完全不读代码”和“只读重要部分”之间的差异。

问：新手也可以用Vibe Coding来学开发吗？
答：作为体验“做出东西”的乐趣是不错的。但如果完全不读代码，实力很难有提升。建议在做完之后再养成追问AI“这段代码为什么这么写”并阅读的习惯。

问：AI写的代码，能信到什么程度？
答：“能跑起来”和“安全”是两码事。代码能不能跑，可以交给AI；但是否安全、是否可长期维护，需要靠人或者另外的校验机制来确认。

---

总结一下，就是“速度交给AI，判断留给自己”。不需要全部读完，但如果完全不读，总有一天要付出代价。上面三个解决办法哪怕今天就在项目里落地一个，都能大幅降低这份代价。

再往前走一步，还有一种方法论叫作规格驱动开发（SDD），主张在让AI写代码之前，先把需求以文档的形式确定下来。这是与Vibe Coding相对的另一种思路，下一篇文章会详细展开。
