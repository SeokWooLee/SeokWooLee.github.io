代码明明能跑起来，但为什么每次想改一个小功能，打开文件后总要东改西改？

只是换了个按钮颜色，结果支付逻辑就崩了——这种经历，大家应该都遇到过吧。

能用两个词精准解释这个原因的概念，就是耦合度和内聚度。

> 耦合度越低越好，内聚度越高越好。好设计的八成，都是从这一句话出发的。

这篇文章会讲清楚这两个概念到底是什么、为什么被称为软件设计原则的根基，以及如何在实际代码中运用它们。

先把核心内容整理一下。

1. 耦合度（Coupling）：模块之间的纠缠程度 → **越低**越好
2. 内聚度（Cohesion）：一个模块内部的代码有多专注于一件事 → **越高**越好
3. 这两个概念是 SOLID、设计模式等知名原则的底层基础
4. 最终目标只有一个：写出"容易修改、容易替换"的代码

---

## 耦合度和内聚度，到底是什么？

先用一句话概括这两个概念。

耦合度指的是模块与模块之间的关系，内聚度指的是一个模块内部的凝聚力。

方向正好相反，很容易混淆，我是这么记的：

耦合度是"与外部的距离"，内聚度是"内部的凝聚"。

耦合度高，修改 A 时会连带牵扯出 B、C，就像多米诺骨牌一样接连倒下。

内聚度低，则会出现一个类里混杂着支付逻辑、邮件发送、日志记录的情况。光看类名，根本猜不出它到底在做什么。

好的设计会把这两者往相反的方向推：对外要松散，对内要紧密。

---

## 为什么这是设计原则的"根"？

耦合度和内聚度是在 1970 年代的结构化设计理论中首次被系统整理出来的概念，据说是由拉里·康斯坦丁（Larry Constantine）提出的。

虽然已经过去将近半个世纪，但它依然适用，原因就在于这是不受特定语言或潮流影响的本质规律。

把我们熟知的那些著名原则拆开来看，最终都会归结到这里。

| 原则/模式 | 归根结底想说的是 |
|---|---|
| 单一职责原则（SRP） | 提高内聚度 |
| 依赖倒置原则（DIP） | 降低耦合度 |
| 接口隔离原则（ISP） | 切断不必要的耦合 |
| 大多数设计模式 | 低耦合 + 高内聚 |

看到了吧，名字各不相同，但根源只有一个。

所以每次学习新的原则或模式时，我都会先问一句："这是在降低耦合度，还是在提高内聚度？"这样一问，问题基本就理解了一半。

## 降低耦合度的方法，用代码来看看？

光说概念比较抽象，来看一个简短的例子。

下面是耦合度较高的代码，订单类直接知道了具体是哪家支付公司。

```swift
class Order {
    let pay = KakaoPay()  // 死死绑定在 KakaoPay 上
    func checkout() {
        pay.send()        // 想换成别的支付方式就得动这里
    }
}
```

一旦要把支付公司换成 Toss，就必须打开 Order 进行修改，这正是高耦合度的典型表现。

这次用协议（protocol）把两者之间的距离拉开。

```swift
class Order {
    let pay: Payment                // 只依赖"支付"这个约定
    init(pay: Payment) { self.pay = pay }
    func checkout() { pay.send() }  // 不管传进来的是什么都无所谓
}
```

现在 Order 不再关心到底是哪家支付公司，只要替换一下就行了。

在实际工作中，这样改造之后，即便再有新增支付方式的需求，也几乎不用动原有代码了。这就是低耦合度的力量。

## 提高内聚度要怎么做？

判断内聚度，就看"这个模块是不是只做一件事"。

我的判断标准很简单：说出一个类的名字，如果里面的方法都能自然而然地联想到，那内聚度就是高的。

举个例子，如果 UserService 里混着这些内容，就该拉响警报了：

- 会员注册处理（本职工作）
- 营销邮件发送（不相干的事）
- CSV 报表生成（又一件不相干的事）

这就好比毫不相关的几件事挤在同一个屋檐下。

这种情况下，把邮件功能拆到 EmailSender，报表功能拆到 ReportGenerator，各自安个家就好了。

这样一来，以后修改邮件逻辑时就不用再打开 UserService 了，修改带来的连锁影响也会随之变小。

低耦合度和高内聚度，最终指向的是同一个方向：阻止修改的影响范围扩散。

## 常见问题（Q&A）

**Q. 耦合度和内聚度，应该先关注哪个？**

我建议先抓内聚度。让一个模块专注于做一件事，慢慢拆分下来，耦合度往往也会自然而然地跟着理顺。

**Q. 耦合度是不是越接近 0 越好？**

并不是。如果模块之间完全没有任何关联，程序根本就跑不起来。目标不是归零，而是"只保留必要的连接，并且尽量松散"。

**Q. 一开始就要做到完美设计吗？**

没那个必要。可以先让代码能跑起来，之后每当修改变得吃力时，再一点点切断耦合、聚拢内聚。重构本来就是一个反复迭代的过程。

---

耦合度和内聚度算不上什么炫酷的新技术，但它们是那些写出经得起时间考验的代码的人共有的习惯。

下次写完代码，不妨问自己一句："如果改这里，影响会波及到哪里？"这个问题问得多了，设计的直觉自然就会一点点养成。加油！
