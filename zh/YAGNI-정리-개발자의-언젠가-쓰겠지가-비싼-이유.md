"以后应该会用得上吧,要不要先做出来?"

开发的时候,这种诱惑真的经常出现。配置项提前抽出来,多语言支持的结构提前搭好,后台能改的表也提前建好。

我以前也做过一个"总有一天会用到"的功能。结果两年后亲手把那段代码删掉的人,就是我自己。因为它一次都没被用过。

今天要聊的YAGNI,正是对这种诱惑正面说"不"的原则。它和KISS、DRY一起,被称为开发原则三巨头。

YAGNI是"You Aren't Gonna Need It"的缩写,意思是"反正你用不上的"。

这个说法出自极限编程(XP)这一开发方法论,是Kent Beck、Ron Jeffries等XP阵营的人挂在嘴边的口头禅,后来固化成了一条原则。

核心主张只有一个。

> 功能要在真正需要的时候再实现,而不是在预感可能需要的时候。

---

## 提前做好到底有什么问题?

"提前做好,以后不是更方便吗?"这种反驳马上就会出现。问题在于,这种预测大多数时候都是错的。

Martin Fowler就这个主题写过一篇文章,把提前做好的功能的成本分成了四类。

| 成本 | 内容 |
|------|------|
| 实现成本 | 花时间做了一个现在根本用不上的功能 |
| 延迟成本 | 本该用这段时间做的真正需要的功能被拖后了 |
| 维护成本 | 就算不用,测试、重构、修bug照样得跟着做 |
| 修复成本 | 真到了需要用的那一刻,需求早已变样,还得推倒重做 |

最痛的是最后一项。等"以后"真的到来时,那时候的需求几乎总是和我预想的不一样。结果就是,提前做好的代码还得再花一遍成本去改。

---

## 实战中YAGNI是怎么被打破的

道理上大家都点头认同,可实战中它总会以这些方式悄悄溜进来。

**1. 提前预留参数**

```javascript
// 现在只有邮件通知这一种
function sendNotification(user, message, channel = 'email',
  retryCount = 3, priority = 'normal', template = null) {
  // 只有channel为'email'时才会生效的代码
}
```

调用这个函数的地方全都只有`sendNotification(user, msg)`这一种写法。其余那些参数是"为了以后接入Slack通知"而准备的,可Slack通知的需求始终没有出现。

**2. 提前搭好接口**

明明只有一个实现类,却先把接口做出来。就是那种做一个`UserService`的同时,顺手也做一个`IUserService`的模式——在第二个实现出现之前,那个接口只是白白多出来的一个文件,纯装饰品。

**3. "抽成配置不是挺好的吗"**

因为害怕硬编码,把所有的值都搬到配置文件和后台界面里。结果那个配置一年都改不了一次,配置越多,"改这个值会影响到哪里?"这种疑问反而越来越多。

## 哪些不属于YAGNI

为了避免误解,先划清界限。YAGNI不是这些意思。

首先,它不是说不要做设计。让代码更容易修改的努力(好的命名、小函数、测试)是当下就有价值的事情,不属于YAGNI要限制的对象。恰恰相反,代码越容易改,"真正需要的那一刻"才能更快做出来。

其次,它也不是说要把难以撤回的决定也一并推迟。像数据库选型、API公开规范这类一旦定下就很难改的东西,提前考虑清楚才是对的。YAGNI针对的是"以后可以轻松追加,却偏偏提前做好的功能"。

---

## 我用的判断标准

每当我犹豫"要不要先做出来"的时候,我会问自己两个问题。

> 这个功能的需求是写在路线图里,还是只存在于我的想象里?

> 以后再加,成本会比现在高出很多吗?

如果依据只是想象,而且以后再加成本也差不多,那我就不做。大多数情况其实都属于这一类。

## 结语

YAGNI原则要带走的东西有三点。

- 功能要在真正需要的时候再做,而不是在"感觉会需要"的时候。
- 提前做好的功能,会把实现、延迟、维护、修复这四项成本一并算给你。
- 但好的设计习惯和难以撤回的决定,不属于YAGNI的限制对象。

把KISS、DRY和YAGNI放在一起看,这三条原则最终会汇成一句话:只做今天需要的东西,做得简单,只放在一个地方。

如果你也有一个"总有一天会用到"却一直沉睡着的功能,说不定两年后,亲手删掉它的人也会是你自己。
