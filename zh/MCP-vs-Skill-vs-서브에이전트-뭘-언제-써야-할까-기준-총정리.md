最近扩展 AI 编程工具的方法明显变多了。用 MCP 接入工具，用 Skill 注册流程，用子代理来分派任务。

问题在于这三者的说明都是「扩展模型的能力」，所以真到要选的时候反而容易犯迷糊。

先用一句话把标准定下来：MCP 是手，Skill 是说明书，子代理是同事。

这篇文章会梳理这三者各自扩展的是什么、该按什么标准来选、组合起来又是什么样子。

先看核心要点。

1. MCP 扩展的是模型能做的事（工具、数据访问）
2. Skill 扩展的是模型知道的方法（作业流程、领域知识）
3. 子代理扩展的是干活的人（拥有独立上下文的执行主体）
4. 三者不是竞争关系，而是组合关系。子代理读取 Skill、再调用 MCP 工具的结构是很自然的

---

## 三者各自扩展的东西

先用表格对比一下，再逐个来看。

| 区分 | MCP | Skill | 子代理 |
| --- | --- | --- | --- |
| 扩展对象 | 能力（工具、数据） | 知识（流程、经验） | 执行主体 |
| 本质 | 协议、服务器进程 | Markdown 文件夹 | 独立上下文会话 |
| 比喻 | 手 | 说明书 | 同事 |

MCP 是伸向模型外部系统的通道。查数据库、调用公司内部 API、操作浏览器，这些模型单靠自己在物理上做不到的事，MCP 都能让它变成可能。它的特点是存在一个执行代码的独立服务器。

Skill 给的不是新能力，而是让本来就能做的事做得更好。发布说明的格式、评审清单、公司内部文档规范，这类关于「怎么做」的知识被装进文件里保存下来。它的本质就是普通的 Markdown。

子代理是拥有独立上下文窗口的另一个模型实例。它可以在不污染主线对话记录的前提下，独立承担探索、调研之类的工作。哪怕让它翻遍大量文件，返回主线上下文的也只有结论。

---

## 决定该用哪个的三个问题

拿不准该选哪个时，按顺序问自己三个问题就行。

第一，这是模型现在在物理上做不到的事吗？如果像查询公司内部数据库那样，连访问权限本身都没有，答案就是 MCP。再怎么塞知识进去，也变不出原本没有的手。

第二，虽然能做，但每次都得重新说明方法吗？那就是 Skill。凡是会重复出现的提示词，都是 Skill 的候选。

第三，工作量太大导致上下文被污染了吗？如果调研、探索的结果把对话弄得一团乱，那就该用子代理来隔离了。

> 反过来说也成立：没有访问权限找 MCP，没有诀窍找 Skill，没有（不够用的）上下文找子代理。

## 实战中要把三者组合起来用

这三者不是二选一的关系。实际上运转良好的自动化流程，大多长这个样子。

以代码评审自动化为例：先定义一个评审员子代理（主体），这个代理遵循团队的评审清单 Skill（方法），再通过 GitHub MCP 服务器在 PR 上留下评论（能力）。

可以看到这不是角色重叠，而是分属不同的层。谁来做 / 怎么做 / 用什么做，正好对应这三个问题各一个。

## 两个常见的错误选择

边界看清楚了，反模式也就浮现出来了。

其一，是给用 Skill 就足够解决的事搭建 MCP 服务器。比如「应用提交信息规范」本来是知识层面的问题，几行 Markdown 就能搞定，为此专门写一个服务器就是本末倒置。服务器伴随而来的是维护成本和安全审查成本。

其二，是把所有事都塞进主线这一个会话里处理。如果在主线里直接探索大规模代码库，文件转储会把上下文塞满，等真正到实现阶段反而没有空间可用了。正确做法是把探索工作交给子代理，主线只接收结论。

---

## 结语

总结一下：没有能力就用 MCP，没有诀窍就用 Skill，没有人手就用子代理。

各自的详细内容已经分别在 MCP 篇和 Skill 篇讨论过，建议大家用这篇文章的标准，先判断一下自己现在想搭建的自动化到底属于哪一层的问题。
