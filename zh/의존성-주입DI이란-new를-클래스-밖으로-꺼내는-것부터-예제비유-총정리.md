代码里如果需要某个对象,很多人会直接在用到的地方 new 出来用。可是一旦要写测试代码,就会撞上一堵墙。因为对象是死死焊在类里面创建的,根本没法替换。

搜索"依赖注入(DI)是什么",出来的大多是"控制反转"、"IoC 容器"之类的说法,反而更让人摸不着头脑。

所以这次想用最简单的方式把它讲清楚。先说结论。

依赖注入(DI)说到底,就是把原本在类内部直接创建的对象,改成从外部创建好再传进去。核心就是这一个动作的转移——把对象创建的位置从类内部搬到类外部。DI 的绝大部分内容,其实就是这一句话。

读完这篇文章,你会明白 DI 为什么必要、代码层面会发生怎样的变化,以及 Swinject 这类 DI 库为什么能帮我们代劳这件事。

## 对象创建放在类内部,问题出在哪?

举个例子,假设有一个处理订单的类。这个类要完成支付,就需要用到支付模块。

最常见的写法是这样的。

```swift
class OrderService {
    // 在类内部直接创建支付对象
    private let pay = KakaoPay()

    func order() { pay.pay() }
}
```

问题在于,这个 OrderService 已经和 KakaoPay 死死绑在一起了。

如果想把 KakaoPay 换成别的支付方式呢?那就得钻进类内部,改掉创建那一行代码。

写测试时想塞进一个假的支付对象也没办法,因为类内部已经自己创建好一个真正的 KakaoPay 了。

也就是说,把对象创建放在类内部,这个类就会被特定的实现死死捆住手脚。

---

## 于是把对象创建挪到了外面

解决办法其实出乎意料地简单。不要自己在类里创建,而是从外部创建好,接收进来就行。

```swift
class OrderService {
    private let pay: Pay
    // 只接收外部创建好并传进来的对象
    init(pay: Pay) { self.pay = pay }

    func order() { pay.pay() }
}
```

变化只有一处。直接创建的代码消失了,换成了通过构造函数(初始化器)接收的形式。

现在 OrderService 不需要知道支付方式究竟是 KakaoPay 还是 NaverPay,它只需要知道"接收一个 Pay 类型的东西来用"就够了。

> DI 的本质并不是什么了不起的东西。它只是把对象创建挪到类外面,把"该塞进去什么"的决定权交给外部而已。

这种从外部把对象塞进来的做法,叫做"注入(injection)"。注入所依赖的对象,所以叫依赖注入。

名字听起来很唬人,但实际做的事,就是刚才看到的这些而已。

---

## 用了 DI 到底会带来什么好处?

光说好处不够直观,这里具体列一下实际会发生什么变化。

- 替换变得容易:把 KakaoPay 换成 NaverPay,OrderService 一行代码都不用改,只需要换传进去的对象。
- 测试变得轻松:可以传入假的(Mock)支付对象代替真实支付,不用真的花钱就能验证逻辑。
- 职责得到分离:OrderService 只专注于"处理订单",至于"用哪种支付方式"由外部决定。

第三点对我触动最大。类只专注于自己的事情,代码读起来也顺畅了不少。

整理一下如下。

| 区分 | 创建放在内部时 | 把创建挪到外部时(DI) |
|------|------|------|
| 更换支付方式 | 需要修改类 | 只需替换传入的对象 |
| 测试 | 无法传入假对象 | 可以注入 Mock |
| 关注点 | 创建与逻辑混在一起 | 只专注于逻辑 |

## 那 Swinject 这类 DI 库又是为什么需要呢?

看到这里自然会冒出一个疑问:"知道要从外部传进去了,可这个'外部'到底该由谁来管理?"

如果对象数量不多,手动创建再传进去也没问题。但实际项目里对象往往有成百上千个,彼此还相互纠缠。

要靠人工一个个按顺序创建再传递,简直是地狱。

于是就出现了专门代劳这件"从外部创建并传入"工作的工具。Swift 阵营里的 Swinject 或 Factory 这类 DI 容器,扮演的正是这个角色。

只要把需要的对象事先注册好,容器就会自动创建并塞到需要的地方。这样一来,几乎不需要自己手动创建对象了。

常说的"控制反转(IoC)"也是由此而来。意思是创建和连接对象的控制权,已经从我的代码转移到了容器手里。

换句话说,DI 是一个概念,而 DI 容器则是替这个概念执行落地的工人。

## 常见问题

Q. DI 和 IoC 是同一回事吗?
不是。IoC(控制反转)是"把控制权交出去"这样一个更宽泛的原则,而 DI 是实现它的具体方法之一。可以把 DI 理解为 IoC 的一种。

Q. 除了构造函数,还有别的注入方式吗?
有。还有属性注入、方法注入等方式。不过在 Swift 中推荐使用构造函数注入,因为依赖会被 let 固定住,值不会改变,测试起来也更方便。

Q. 一定要用库才算 DI 吗?
不是。刚才那种手动传进构造函数的做法,本身就是不折不扣的 DI。库只是把这个过程自动化了而已。

如果觉得 DI 很难懂,只需要记住一句话:"不要在类内部创建,而是从外部创建好再传进去。"记住这一句,剩下的自然就会跟上来。
