声明一个变量之后，它的值到底存放在哪里呢？

如果回答"内存里"，只能算说对了一半。因为在内存内部，放在栈上的值和放在堆上的值，命运是完全不同的。

栈溢出这个报错的名字、类和结构体之间的性能差异、为什么需要 ARC——所有这些，都要理解这两个区域才能说清楚。

这篇文章会梳理栈和堆各自是如何运作的，速度差异从何而来，以及在 Swift 里到底什么东西会被分配到哪里。

先看核心要点。

1. 栈：随函数调用而堆叠、随函数返回而消失的自动管理区域。速度快
2. 堆：在运行时按需借用空间的动态区域。灵活，但需要付出管理成本
3. 速度差异的本质在于"分配与释放的方式"以及"由谁来负责清理"
4. 在 Swift 中，结构体大多存在栈上，类的实例则存在堆上

---

## 栈：叠放的盘子

栈是用来堆叠函数调用信息的区域。正如它的名字，就像叠盘子一样。这是一种"后进先出"（LIFO）的结构，最后放上去的盘子最先被取下来。

每次调用函数时，该函数的局部变量、参数、返回地址会作为一个整体（栈帧）被压入栈中。函数返回时，这个栈帧会整体消失。

这正是栈的优势所在。

**分配速度非常快**。只需要把栈指针向上移动即可，不需要四处寻找空闲位置。

**释放也是免费的**。函数结束时，只要把指针移回去就行，不需要考虑谁来清理。

不过它也有限制：大小必须在编译期确定，函数一结束就必然消失，而且总容量有限（通常只有几 MB）。如果递归函数没有终止条件、不断调用自身，栈帧就会超出容量上限，这个报错正是所谓的栈溢出（stack overflow）。

---

## 堆：宽敞但需要管理的仓库

堆是在运行时按需借用空间的区域。不需要提前知道大小，函数结束后依然存活,空间也宽敞得多。

但随之而来的是管理成本。

**分配相对较慢**。需要寻找符合请求大小的空闲空间，如果多个线程同时申请分配,还需要进行协调。

**释放必须有人负责**。因为函数结束后它不会自动消失，所以必须由某个角色判断"现在不再使用了"。C 语言需要程序员手动调用 free，Java 由垃圾回收器定期清理，而 Swift 由 ARC 通过计数引用数，一旦归零就立即释放。

一旦这个判断出错，就会引发两类事故：释放得太晚会造成内存泄漏，释放得太早则会导致访问已释放内存（悬垂指针）。

## 在 Swift 中，什么会被分配到哪里

Swift 官方文档和 WWDC 演讲中反复强调过这样一个结构：值类型（结构体、枚举）默认分配在栈上，引用类型（类）默认分配在堆上。

```swift
struct PointStruct { var x, y: Double }
class PointClass { var x = 0.0, y = 0.0 }

func run() {
    let a = PointStruct(x: 1, y: 2) // 整体存放在栈帧内
    let b = PointClass()            // 分配在堆上，栈中只存引用
}
```

结构体 a 的值会整体存入栈帧。函数结束后随着栈帧一起消失，所以也不需要引用计数。

类的实例 b 则驻留在堆上，栈中只存放它的地址。由于可能被多处同时引用，ARC 必须对其进行计数。

这正是苹果建议"优先考虑使用结构体"这一性能建议的依据所在——因为它可以整体省去堆分配、引用计数和加锁的开销。

不过，"结构体 = 一定在栈上"并不成立。作为类的属性存在的结构体，会随着该类一起存在堆上；而 String、Array 这类类型虽然表面上是结构体，实际的数据缓冲区却存放在堆上。更准确的理解是："结构体具备被放在栈上的条件"，而非绝对保证。

## 面试常见追问

**"为什么栈比堆快？"**——因为栈只需移动指针就能完成分配和释放，而堆需要搜索空闲空间，并且要处理释放管理（引用计数或 GC）。

**"栈溢出是怎么产生的？"**——栈的容量上限本来就小，一旦因深层递归或过大的局部变量导致栈帧超出这个上限,就会发生栈溢出。

**"局部变量一定存在栈上吗？"**——不一定。即便是局部变量，如果它是引用类型，本体依然存在堆上，栈中存放的只是引用。

---

## 总结

- 栈是随函数调用自动堆叠、自动消失的区域，仅靠指针移动即可完成管理，速度快
- 堆是在运行时按需借用空间的区域，灵活但需要付出空间搜索和释放管理的成本
- 释放责任：栈是自动的，堆则因语言而异（C 靠手动释放，Java 靠 GC，Swift 靠 ARC）
- 栈溢出是深层递归等原因导致栈容量超限时产生的错误
- 在 Swift 中，结构体大多存在栈上，类则存在堆上——这正是优先推荐值类型的性能依据
- 但类内部的结构体依然存在堆上，String、Array 的内部缓冲区也存放在堆上
