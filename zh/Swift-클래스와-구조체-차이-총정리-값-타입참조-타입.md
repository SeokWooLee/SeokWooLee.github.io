用 Swift 开发时，总有一个瞬间会让人犹豫一下。

“这个到底该用 class 写，还是用 struct 写？”

我自己刚学 Swift 的时候，也是全凭感觉在这两者之间随便选。后来踩了一个莫名其妙的 bug，才真正静下心来把两者的区别搞清楚。

先说结论。

> 如果没有特殊理由，请优先选择结构体（struct）。
>
> 只有在真正需要“一份被共享的状态”时，才使用类。

苹果官方指南也是同样的立场。今天就把原因一条一条讲清楚。

## 先看核心要点（3 条）

为赶时间的朋友先做个总结。

1. **结构体是值类型**。一旦复制，就变成完全独立的另一个值。
2. **类是引用类型**。即便复制了，也依然指向同一个实例。
3. 拿不准的时候**先用 struct**，只有需要继承或共享状态时才用 class。

记住这三句话，基本上就理解了一半。

---

## 值类型和引用类型，到底哪里不一样？

光靠文字讲很容易听得云里雾里。最快的办法还是直接看代码。

先看结构体。一旦复制，彼此就变得毫无关系了。

```swift
struct Point { var x = 0 }

var a = Point()
var b = a      // 值被复制
b.x = 10

print(a.x, b.x)
// 输出：0 10
```

即使改了 `b`，`a` 依然保持原样。因为它们各自持有自己的值。

再看类。代码看起来差不多，结果却完全不同。

```swift
class Box { var x = 0 }

let a = Box()
let b = a      // 引用被复制（同一个对象）
b.x = 10

print(a.x, b.x)
// 输出：10 10
```

明明只改了 `b`，`a` 也变成了 10。

因为两者指向的是同一个实例。困扰过我的那个 bug，正是这个原因。

## 那到底什么时候该用哪个？（选择标准）

在实际项目中做判断时，我脑子里会浮现这张表。

| 情况 | 判断 |
|------|------|
| 单纯用来承载数据的模型 | 结构体 |
| 希望复制后彼此保持独立 | 结构体 |
| 必须要用到继承 | 类 |
| 需要在多处共享同一个实例 | 类 |
| 需要与 Objective-C 互通 | 类 |

再展开说一说，大致是这样的：

- 像坐标、颜色、尺寸这类表示**值本身**的东西，用结构体会更自然。
- 像屏幕控制器或网络管理器这种**需要多处共同持有同一份状态**的对象，用类才合适。
- 苹果的 SwiftUI 把 View 设计成 struct，也是出于同样的理由——够轻量，行为也更可预测。

顺带一提，Swift 标准库里的 `Array`、`String`、`Dictionary` 全都是结构体。这也说明值类型才是 Swift 的默认思路。

## 为什么优先推荐结构体

值类型更容易预测。

不用担心自己传出去的值会在别处被偷偷改掉。就算传给函数，原始值也是安全的。

在多线程环境下也会更让人安心。因为没有被共享的状态，数据竞争自然就少了。

相反，类虽然强大，但需要格外小心。如果多个地方都持有同一个实例，就很难追踪这个值到底是在哪里被改动的。

所以“默认用 struct，只有需要时才用 class”这个习惯，能大幅减少 bug 的发生。

---

## 该用的时候用，该避免的时候避开

最后把判断标准简单总结一下。

- **该用结构体**：以数据为中心的模型、需要独立复制、线程安全很重要的场景
- **该用类**：需要继承、需要共享同一个实例、标识（identity）本身有意义的场景
- **应当避免**：明明是简单模型却习惯性地用 class；明明需要共享，却硬用 struct 去绕过去

---

### 面试通常会这样问

**Q. 结构体和类最大的区别是什么？**

结构体是值类型，类是引用类型。结构体一旦复制就会变成完全独立的值，而类即便复制了，也依然指向同一个实例。此外，只有类才能被继承，也只有类才能拥有析构器（deinit）。

**Q. Swift 为什么推荐使用结构体？**

因为值类型没有共享状态，副作用更少，行为也更可预测。原始值不会被意外改动，在多线程环境下也更安全，从而有利于减少 bug。

一开始我也以为这两者只是语法上的区别，越深入了解才发现，这其实是设计层面的问题。

把今天这些判断标准记在手里，下次做项目时就不会再那么犹豫了。值类型还是共享，先问自己这一个问题就够了！
