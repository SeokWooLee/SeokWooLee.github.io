有没有遇到过这种情况：明明逻辑写得好好的，一想给它加测试，代码却总是直接连到真实数据库、真实 API，结果为此熬到半夜？

先说结论。

> 单例是在代码内部直接拿来用的对象，所以测试时没办法把它换成假对象。

相反，依赖注入是把需要的对象从外部传进来的方式，因此在测试中可以随意替换成想要的版本。

今天就通过实际代码，慢慢梳理一下这两者的区别。

## 单例为什么会拖垮测试？

单例是整个程序里只存在一个实例的对象。

用起来确实方便，随便在哪里写一行 `Database.shared` 就能直接拿来用。

但问题也从这里开始。

如果在函数内部直接取用这个对象，这个函数就会和真实数据库死死绑在一起。

测试时根本没办法告诉它"别用真的，用假数据库"。

```swift
// 在函数内部直接取用单例
func saveOrder(_ order: Order) {
    // 这里直接绑死了真实数据库
    let db = Database.shared
    db.insert(order) // 测试中无法阻止这一步
}
```

要测试这段代码，就必须让真实数据库处于运行状态。

数据库一旦没开，测试也会跟着失败。

明明逻辑没问题，却因为基础设施的原因亮起红灯。

---

## 依赖注入怎么做？

依赖注入其实没有想象中那么复杂。

不用在函数内部去取对象，只需要在外部创建好之后传进来就行。

顾名思义，就是把需要的东西"注入（injection）"进去。

```swift
// 将 DB 作为参数从外部传入
func saveOrder(_ order: Order, db: Database) {
    db.insert(order) // 具体用哪个 DB 由调用方决定
}
```

差别就只有一行，对吧？

只是把原本直接调用 `Database.shared` 的方式，改成了通过参数接收。

但这个小小的改动，会彻底改变测试的方式。

因为在实际代码中传入真实数据库，在测试中传入假数据库就可以了。

<figure>
  <img src="/assets/images/posts/bb76dac7-edf6-4e9c-b700-37fe51837a6f/4-1783847645964.png" alt="只是换了传入的对象，结果却天差地别">
  <figcaption>只是换了传入的对象，结果却天差地别</figcaption>
</figure>

通过构造函数接收依赖的方式也很常用。

```swift
class OrderService {
    private let db: Database
    // 创建对象时从外部接收
    init(db: Database) { self.db = db }
}
```

只要接收一次，类里任何地方都能用 `self.db` 调用，写起来会清爽很多。

## 在测试中替换成假对象

现在到真正有意思的部分了。

既然已经把依赖改成从外部接收，那测试时用假数据库（通常叫 Mock）代替真数据库传进去就行了。

```swift
func test_保存订单时会传给DB() {
    let fake = FakeDatabase() // 不是真的，是假的
    saveOrder(Order(), db: fake)
    XCTAssertEqual(fake.savedCount, 1) // 只确认是否调用了保存
}
```

不需要启动真实数据库。

也不需要网络，不需要外部服务器，在内存里瞬间就能跑完。

因此测试速度更快，最重要的是不会再飘忽不定。

因为可以精准验证逻辑本身，不受外部环境影响。

## 单例 vs 依赖注入，一图看懂

把两者的差异整理成了表格。

| 区分 | 直接使用单例 | 依赖注入 |
|---|---|---|
| 对象获取位置 | 在函数内部直接取用 | 从外部传入 |
| 替换为假对象 | 几乎不可能 | 可以自由替换 |
| 测试速度 | 慢（需要真实资源） | 快（在内存中处理） |
| 测试稳定性 | 受外部环境影响而波动 | 只验证逻辑本身 |
| 耦合度 | 高 | 低 |

单例本身并不是绝对的坏东西。

像配置值这种真的只需要一份实例的情况，用单例反而方便。

只是"在任何地方都直接拿来用"这个习惯才是问题所在，因为它会束缚住测试的手脚。

## 常见问题（Q&A）

**Q. 要做依赖注入，是不是必须用 Swinject 之类的 DI 库？**

不是的。像今天代码里那样，通过参数或构造函数传递，本身就已经是依赖注入了。库只是把这个过程自动化管理起来而已。

**Q. 参数太多的话不会显得很乱吗？**

没错。所以通常会用构造函数注入一次性接收，或者把相关的依赖打包在一起。如果参数一直在增加，也可能是在提示这个类承担的职责太多了。

如果测试总是被真实资源拖累，不妨按今天说的方向，先改一个地方试试看。

把直接取用对象的代码，改成从外部接收。

这一个小小的习惯，就能让写测试这件事变得有滋有味。祝大家今天也能舒心地写代码！
