在第 3 篇中，我们看了如何用 /clear 清空对话记录。但 /clear 之后如果检查一下上下文剩余量，会发现并不是 100%。在 Claude Code 里输入 /context 命令就能看到原因。系统提示词、工具定义、CLAUDE.md、MCP 服务器注册的工具，早已占用了数万个 token。甚至在对话还没开始之前。

这一篇要讨论的正是这种"常驻加载上下文"。如果说对话记录是变动成本，那这部分就是固定成本。它每一轮都会被携带，不会因为 /clear 而消失，并且在整个会话期间持续占用模型的一部分注意力。如果固定成本的设计有缺陷，那么无论哪次会话都会从一开始就背着劣势起步，所以这是比对话管理更需要优先处理的地方。

## 会话开始之前就已经装载好的东西

智能体的上下文窗口并不是从一张白纸开始的。在第一条用户输入到达之前，已经铺好了好几层。

最底层是系统提示词。这是智能体执行环境（比如 Claude Code 这样的运行环境）植入的基本指令，其中包含工具使用规则、响应格式等内容，规模可达数千个 token。这是用户无法触碰的区域。

往上一层是工具定义。模型要使用工具，就必须知道每个工具的名称、说明和参数模式（schema），而这些内容全部以文本形式进入上下文。如果只有基础工具，负担并不大，但一旦开始连接 MCP 服务器，情况就不同了。一个服务器注册几十个工具的情况很常见，只要连接几个服务器，光是工具定义就能消耗掉数万个 token。那些一次都不会用到的工具，却要在每一轮对话中往返传输。

最上层是 CLAUDE.md 这类项目指南文件。全局设置、项目设置，乃至子目录设置都会被自动加载，而这一层，是用户唯一可以亲自设计的一层。

## CLAUDE.md 的原则：只写"永远成立"的内容，并且要短

CLAUDE.md 里该放什么，判断标准只有一个：这项内容"在这个项目的所有任务中是否始终成立"。因为这是每一轮、每一项任务都会被携带的文件，如果内容只对某些任务有用、对另一些任务无关，那就没有占用这个位置的价值。

构建、测试命令，代码库的整体结构，以及少数不可违背的规则——这些能达到及格线。反之，某个具体功能的详细规格、某个库的完整使用说明、过往的工作记录，这些只在做相关任务时才需要，应该被淘汰。

关于长度，有一个悖论：指令写得越多，似乎越容易被遵守，但实际情况恰恰相反。正如第 2 篇所说，上下文一旦变长，分配到每个条目上的注意力就会变得稀薄，在数百行的指南文件中，真正重要的规则反而会被淹没在中间（lost in the middle）。写 50 条规则，往往是 50 条都被模糊地遵守；写 10 条规则，则更接近 10 条都被清晰地遵守。如果智能体总是违反 CLAUDE.md 里的规则，那么在增加规则之前，先精简文件可能才是正确的顺序。

## 不要全部塞进去，只留一个指针

那么，那些没能达到及格线的信息——也就是偶尔才需要的详细文档——该放在哪里呢？答案是："放在上下文之外，只告诉它位置"。

与其把数据库迁移流程的全文塞进 CLAUDE.md，不如只写一行"迁移流程请参考 docs/migration.md"。智能体只有在做迁移工作时才会去读那个文件。详细内容只会在需要的那次会话中进入上下文，而无关的会话只需要付出一行指针的成本。

将同样的原理体系化后，就是 Claude Code 的技能（skill）。技能是针对特定任务的操作说明，平时上下文里只挂着名称和一行说明，等到相关任务开始时才会加载正文。与其让"部署流程"常驻在 CLAUDE.md 里，不如把它做成技能——这样在 99% 不涉及部署的对话轮次里，就能省下那部分 token。

工具这边也需要整理。如果连接了却不常用的 MCP 服务器，只要关掉它，就能省下数万个 token 的固定成本。越来越多执行环境开始支持"延迟加载"——工具定义平时只保留名称，需要时才加载完整模式（schema）——方向是一致的：不是把所有东西都常驻加载，而是需要什么，就在需要的时候加载什么。

## 固定成本检查例行流程

常驻加载上下文一旦膨胀，很难被察觉，因为这是每次会话都同样支出的成本，没有对比对象。所以有必要偶尔有意识地检查一下。

在 Claude Code 中，可以用 /context 拆解查看当前上下文的具体去向和占用量。系统提示词、工具、MCP、内存文件各自的 token 数都会显示出来，如果发现工具定义比对话本身还大这种畸形情况，那就该先从整理 MCP 服务器入手。CLAUDE.md 建议每季度打开一次，以"上个月这一行是否真的派上过用场"为标准来删减内容。指南文件如果放任不管只会不断变长，只有把"修剪"变成例行流程，才能保持它的简短。

## 小结

- 系统提示词、工具定义、CLAUDE.md 是每一轮都会被携带的固定成本。它们不会因 /clear 而消失，所以必须在对话管理之前就设计好。
- CLAUDE.md 的判断标准是"是否在所有任务中都始终成立"。文件越长，单条规则就越模糊，如果规则得不到遵守，优先该做的是精简文件而不是增加规则。
- 详细文档不放正文、只留指针，重复性流程拆分为技能，不用的 MCP 服务器就关掉。原则是：需要什么，只在需要的时候加载什么。

做到这一步，单次会话的上下文就会相当干净。但无论怎么节省，总会遇到一个大任务塞不进一次会话的时刻。下一篇要讲的正是这种情况下的结构性解法——子智能体（subagent）。让探索工作交给另一个上下文去做、只把结论带回来的隔离模式，如何守护上下文，我们下篇再看。
