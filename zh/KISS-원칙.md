写代码久了,有时会莫名产生一种错觉——好像代码写得越复杂,才越显得自己厉害。

塞一堆设计模式,叠好几层抽象,还想着"以后说不定要扩展",于是提前挖好一堆接口。

我刚入行那会儿也是这样。可几年后再打开那些代码,真正吃苦头的,是未来的我自己。

今天想聊聊站在它对立面的一个原则,KISS。

先说结论,KISS 是"Keep It Simple, Stupid"的缩写。直译过来大概是"简单点,别整那么复杂"。

它是一种设计哲学:系统不是越复杂越好用,而是越简单、越能保持简单,运行起来才越顺畅。

---

## KISS 原则到底是从哪儿来的?

这句话最初并不是软件领域的术语。

据说它出自20世纪60年代美国洛克希德公司的航空工程师凯利·约翰逊(Kelly Johnson)之口。他就是设计出著名的U-2、SR-71黑鸟侦察机的那个人。

据说凯利·约翰逊当时对团队提出的要求是这样的:

> 造一架在战斗环境下,普通维修人员只靠基本工具就能修好的飞机。

意思是,不管设计多精妙,如果现场修不了,那就没用。

这种精神后来延续到了软件领域。不管代码写得多巧妙,如果同事看不懂、改不了,那就算不上好代码。

顺带一提,这里的"Stupid"并不是说开发者是笨蛋,更接近"简单到连笨蛋都能看懂"的意思。

---

## 代码里 KISS 被破坏的那些瞬间

说起来容易,实际代码长什么样呢?我来分享几个自己经常见到(也亲手犯过)的模式。

**1. 看起来很厉害的一行代码**

```javascript
// 这样写显得很聪明,但是
const result = arr.reduce((a, c) => ({ ...a, [c.id]: c }), {});

// 这样写反而更容易读懂
const result = {};
for (const item of arr) {
  result[item.id] = item;
}
```

上面那段代码虽然短,但第一次看的人得盯着看半天。下面这段,不管谁看,3秒就能明白。

**2. 用不上的扩展性**

就是明明只做一个按钮,却把工厂模式、策略模式全用上了。担心的是"万一以后按钮种类变多了怎么办",可那个"以后",大多数时候根本不会来。

**3. 条件语句迷宫**

if 里套 if,里面再套一层 if。嵌套超过三层,脑子里的堆栈就要溢出了。这种时候只要用 early return 把逻辑摊平,代码马上就清爽多了。

## 坚持 KISS 的三条实战标准

所以我写代码时,基本以这三条为标准。

| 标准 | 问题 |
|------|------|
| 讲解测试 | 这段代码,能不能在1分钟内讲给旁边的同事听懂? |
| 未来测试 | 半年后的自己再看,还能立刻看懂吗? |
| 必要性测试 | 这段代码是现在就需要,还是只是觉得"以后可能用得上"才加的? |

只要有一条答案是"不能",我就会重新想想有没有更简单的办法。

尤其是第三条特别重要,这其实和 YAGNI(You Aren't Gonna Need It)原则是相通的。KISS 和 YAGNI 基本上是一套组合拳。

---

## 简单和敷衍,是两码事

这里有个容易被误解的地方。

KISS 不是说"别动脑子随便写"。恰恰相反。

把东西搞复杂很容易,想到什么就往上加就行。把东西做简单才难,得拼命想清楚该去掉什么。

帕斯卡在一封信里写过一句很有名的话。

> 我没有时间把信写得更短,所以只好写长了。

代码也是一样的道理。简单的代码不是敷衍了事的产物,而是费尽心思去除复杂性之后的结果。

## 写在最后

把 KISS 原则总结一下,大概是这样:

- 代码被阅读的时间,远比被编写的时间长得多。为了读它的人,请写得简单一点。
- 现在用不上的扩展性,就别加进去。那种担心真正变成现实的情况,其实比想象中少得多。
- 简单不是能力不足的表现,而恰恰是能力的证明。

最近我在代码评审里,写得最多的评论大概就是"这里能不能再简单一点"。神奇的是,光靠这一句话,bug 就少了,评审时间也短了。

你的代码,旁边的同事能在1分钟内看懂吗?今天不妨问问自己这个问题。
