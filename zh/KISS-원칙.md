写代码的时候,有时会有一种奇怪的错觉,觉得代码写得越复杂,才越显得厉害。

设计模式一股脑往里塞,抽象层一层一层叠,还想着"以后说不定要扩展",于是提前把接口都定义好。

可是几年之后再打开那段代码,真正吃苦头的,往往是未来的自己。

今天就来聊聊站在它对立面的一个原则——KISS。

先说结论,KISS 是 "Keep It Simple, Stupid" 的缩写。直译过来大概是"保持简单,笨蛋"的意思。

这是一种设计哲学:系统不是在被做得复杂时表现最好,而是在被保持简单时才最稳定可靠。

---

## KISS原则,最初从哪里来?

这个说法最初其实并不属于软件领域。

据说它出自20世纪60年代美国洛克希德公司的航空工程师凯利·约翰逊(Kelly Johnson)之口,他正是设计出著名的U-2、SR-71"黑鸟"侦察机的那位工程师。

据说凯利·约翰逊当年对团队的要求是这样的:

> 要造出这样一架飞机——在战斗环境下,一名普通的机械师仅凭基本工具就能修好它。

意思是,再精妙的设计,如果在现场无法修理,那也没有意义。

这种理念后来延伸到了软件领域。哪怕代码写得再巧妙,如果同事读不懂、改不动,那也算不上好代码。

顺带一提,这里的"Stupid"并不是在骂开发者笨,而更接近"简单到连外行都能看懂"的语气。

---

## 代码中KISS原则被打破的那些瞬间

道理说起来简单,但落到实际代码里会是什么样子呢?下面分享几个我经常见到的模式。

**1. 看起来很"高级"的一行代码**

```swift
// 这样写显得很聪明
let result = arr.reduce([Int: Item]()) { $0.merging([$1.id: $1]) { _, new in new } }

// 但这样写更容易读懂
var result = [Int: Item]()
for item in arr {
    result[item.id] = item
}
```

上面那段代码虽然短,但第一次看到的人得盯着看好一会儿才能理解。下面那段,任谁看了都能在3秒内看懂。

**2. 根本用不上的扩展性**

明明只是做一个按钮,却动用了工厂模式加策略模式。原因往往是"万一以后按钮种类多了怎么办",可这个"以后",大多数时候根本不会到来。

**3. 条件语句迷宫**

if 里套 if,里面又套一层 if。哪怕只嵌套三层,脑子里的"堆栈"就快撑不住了。这种情况下,只要用提前返回(early return)把逻辑展开,代码立刻就能清爽不少。

## 坚守KISS原则的3个实用标准

所以我写代码时,通常会用下面这三条标准来自我检验。

| 标准 | 问题 |
|------|------|
| 讲解测试 | 这段代码,能在1分钟内讲清楚给旁边的同事听吗? |
| 未来测试 | 6个月后的自己再看,还能马上看懂吗? |
| 必要性测试 | 这段代码是现在就用得上,还是只是"感觉以后可能用得上"才加进去的? |

只要这三条里有一条答案是"不能",我就会重新想想有没有更简单的办法。

尤其是第三条特别关键,这其实和YAGNI(You Aren't Gonna Need It,你不会需要它)原则是相通的。KISS和YAGNI基本上是配套出现的。

---

## 简单和敷衍,是两回事

这里有一点容易被误解,需要特别说明一下。

KISS并不是说"不用动脑子随便写",恰恰相反。

把代码写复杂其实很容易,想到什么就往上堆就行了。把代码写简单才是真正难的地方,因为要绞尽脑汁去想该去掉什么。

帕斯卡在信中留下过一句名言:

> 我没有时间把它写得更短,所以写得更长了。

代码也是同样的道理。简单的代码不是敷衍了事的产物,而是努力剔除复杂性之后的结果。

## 总结一下

KISS原则,概括起来就是这些:

- 代码被阅读的时间,远比被编写的时间要长得多。为了读代码的人,请尽量写得简单。
- 不要加入现在用不上的扩展性。那种担心变成现实的情况,其实比想象中少见得多。
- 简单不是实力不足的表现,而是实力的证明。

最近在做代码审查时,我留下最多的评论好像就是"这里能不能再简单一点?"。神奇的是,光靠这一句问话,bug就少了,审查花的时间也短了不少。

不妨今天问问自己:你写的代码,旁边的同事能在1分钟内看懂吗?
