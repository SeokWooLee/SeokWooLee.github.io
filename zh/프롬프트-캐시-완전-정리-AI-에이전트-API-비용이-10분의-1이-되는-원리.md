系列的最后一篇聊聊钱的事。用过 API 跑智能体的人，大概都在账单上发现过一个奇怪的现象：输入 token 的费用远远高于输出。回想第一篇讲过的结构，这其实是理所当然的。每一轮都要把系统提示词到整个对话历史重新发送一遍，如果是一个 50 轮的智能体会话，同一份系统提示词就相当于被计费了 50 次。

把这种重复成本大幅削减下来的机制，就是提示词缓存（prompt caching）。用得好的话，输入成本能降到十分之一，响应也会更快，但有一个前提条件：需要按照缓存喜欢的方式来堆叠上下文。这一篇就来看看它的原理，以及那些不经意间就会让缓存全军覆没的失误。

## 相同的前缀不会重新计算

原理在第一篇里其实已经讲了一半。模型处理输入时，每个 token 都会产生注意力的中间计算结果，这些结果会作为 KV 缓存堆积在 GPU 显存里。提示词缓存就是让这份 KV 缓存在一次请求结束后不被丢弃，而是保留下来；等到下一次请求的输入以相同内容开头时，就把那一段的运算整个复用。

看看这跟智能体对话有多契合。第 10 轮的输入是「系统提示词 + 第 1~9 轮 + 新消息」，第 11 轮的输入则是「系统提示词 + 第 1~10 轮 + 新消息」，前半部分完全相同。有了缓存，服务器就可以直接取出已经算好的、到第 10 轮为止的结果，只需要计算新加上去的尾部即可。

计费方式也是照着这个结构走的。以 Anthropic API 为例，从缓存中读取的输入 token 只按原价的 10% 收费。写入缓存的新 token 会加收 25% 的费用，但在「写一次、读多次」的智能体模式下，这部分很快就能被抵消。如果是长达数百轮的会话，有没有缓存能拉开好几倍的成本差距。处理时间也会因为跳过了缓存部分而缩短，首个 token 出现前的延迟会明显降低。

## 条件只有一个，前缀必须一字不差

天下没有免费的午餐。缓存能被复用的条件是前缀（prefix）一致。只有从输入最开头开始、内容完全相同的那一段才能命中缓存，从第一个出现差异的地方开始，后面的全部都要重新计算。

这条规则为什么成立，第一篇也已经解释过了。在注意力机制中，每个 token 的 KV 值都依赖于它前面的所有 token。只要前面有一个 token 发生变化，后面所有 token 的计算结果都会跟着改变，于是缓存就用不上了。这也是为什么上下文越靠前的位置，改动的代价越昂贵。

所以，对缓存友好的上下文原则可以用一句话概括：稳定的内容放前面，可变的内容放后面，历史记录只追加、不修改（append-only）。

## 那些不经意间就会让缓存全军覆没的失误

原则很简单，但违反它的方式却出奇地多、也很隐蔽。

第一个是系统提示词里的可变值。如果在系统提示词里精确到秒地写入当前时间，会发生什么？上下文的最开头每次请求都会变化，缓存命中率直接归零。就算要加入时间信息，也应该粗略到「日期」这个级别，或者放在上下文的靠后位置。会话 ID、随机值也是同样的地雷。

第二个是工具定义的变更。工具的 schema 通常放在上下文的最顶端，如果在会话中途新增或删除工具，其后的整个缓存都会失效。第四篇里建议过整理 MCP 服务器，从缓存的角度看，这里还得再补一句：要整理就在会话开始之前整理，中途不要动。

第三个是对历史记录的中途编辑。有人会试图通过修改上一轮的内容、或删除较早的轮次来节省上下文，但这样一来前缀就被破坏了，从那个位置开始缓存全部失效。为了省下几千个 token，却搭上整个会话重新计算的成本，很容易变成得不偿失的买卖。如果非要从历史记录里删掉点什么，不如索性做个了断，把整段历史清空重来。

最后，第三篇提到的 /compact，从缓存的角度再看一次也会有新的理解。压缩会把整个历史替换成一份新的摘要，缓存因此被完全重置。当然，之后的每一轮都是建立在变短的历史之上重新堆积缓存，长期看是划算的，但也需要有这样一个意识：「compact 不是免费的清理，而是一个伴随缓存重建成本的事件」。这也是为什么在很多情况下，按阶段收尾时用 /clear，反而比频繁 compact 更划算。

## 系列总结，驾驭上下文的七种直觉

把贯穿七篇的原则各自折叠成一句话。

1. 上下文窗口是有限的，这个极限来自注意力运算和 KV 缓存这层物理现实。
2. 放进去不代表能被用好，长度本身就会拉低质量。
3. 每个任务边界默认用 /clear，只有需要延续脉络时才带上指令使用 /compact。
4. 系统提示词、CLAUDE.md、工具定义都是固定成本，只放永远为真的内容，而且要简短。
5. 过程繁重、结论轻巧的工作，交给子智能体的上下文去隔离。
6. 需要跨会话保留的知识要外部化成文件：计划文件、记忆，规模大了就上 RAG。
7. 上下文要以 append-only 的方式堆叠，缓存才能活下来。稳定的放前面，可变的放后面。

如果要浓缩成一句话，大概是这样：上下文不是模型的能力,而是使用者要设计的一种预算。即便到了百万 token 的时代，这种直觉大概率依然成立。窗口越大，挑选往里面放什么的眼光，就越能决定表现的差距。
