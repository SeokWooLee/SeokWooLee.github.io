“这得写到什么时候啊……”光是做一个功能时间就已经很紧张了,如果还要写测试代码,说实话让人忍不住叹气。

先说结论,写测试代码的真正原因是“为了保护未来的自己”。现在多花10分钟,是为了避免3个月后凌晨突然爆发的故障。

## 为什么要写测试代码?

最大的原因是,修改代码不再让人害怕。

代码不是写一次就结束的。它会不断被修改,不断新增功能。

没有测试的话,每改一行代码都会让人不安。“改了这里,别的地方会不会出问题?”

有了测试,改完之后运行一次就能确认。绿灯就放心,红灯的话马上就能看出哪里出了问题。

<figure>
  <img src="/assets/images/posts/9cb89987-2d6c-4e11-ae8e-b6d95c4e0868/4-1783847709666.png" alt="改完运行一次,不是绿灯就马上查原因">
  <figcaption>改完运行一次,不是绿灯就马上查原因</figcaption>
</figure>

这叫做“回归测试”。它能帮你发现原本运行正常的功能又被弄坏的情况。

## 明明嫌麻烦却还要写的真正原因

说实话,测试代码在当下会让人觉得是一种损耗。因为它并不会增加看得见的功能。

但随着项目规模变大,情况就不一样了。

功能增加到30个、50个之后,靠人工一一确认就不可能了。总不能每改一处就把所有功能都点一遍吧。

这时候如果有测试,一条命令就能在几秒内验证几十个功能。

第二个原因是,测试代码本身就是一份“文档”。

看一段写得好的测试代码,就能一眼看出这个函数在什么输入下会产生什么结果。

> 注释放久了会说谎,但测试因为要被执行,所以没法说谎。

来看一个简单的例子。这是一个计算折扣金额的函数的测试。

```swift
// 1万元打9折 → 应该得到9千元
func test_10퍼센트_할인을_적용한다() {
    let result = applyDiscount(10000, rate: 0.1)
    XCTAssertEqual(result, 9000)
}
```

光看这段测试,就能马上明白applyDiscount是个做什么的函数。

## 测试代码要怎么开始写?

如果一开始就想把所有东西都测试到,很容易累到放弃。

我建议这样开始:

1. 从金额计算、登录这类“一旦出问题就是大事”的核心逻辑开始
2. 条件分支多、容易搞混的函数
3. 曾经出过bug的地方(用于防止复发)

相反,简单的界面代码或者经常变动的部分,可以放到后面再做。

目标不是100%覆盖所有内容,而是要权衡投入产出比。

## 常见问题

**Q. 写测试会不会拖慢开发速度?**

A. 一开始确实会变慢。但从项目中期开始,反而会变快,因为查找和修复bug的时间会大幅减少。

**Q. 测试覆盖率做到多少百分比比较好?**

A. 没必要过分执着于数字。70%~80%就足够了,更重要的是核心逻辑是否被覆盖到。

---

测试代码不是用来炫耀实力的,而是为了保护未来的自己和同事的一道安全装置。

不妨现在就给今天写的一个函数配上一个测试试试看。当你感受到那盏小小的绿灯带来的安心感之后,自然就会明白大家为什么都劝你写测试了。
