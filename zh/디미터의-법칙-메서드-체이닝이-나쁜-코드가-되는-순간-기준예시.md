做代码审查的时候，多少都会听到这样一句吐槽。

“这一行，点（.）也太多了吧？”

连成一串写下来看起来反而更简洁，可到底问题出在哪儿呢？

这个问题的核心，正是**得墨忒耳定律（Law of Demeter）**。

这篇文章就来聊聊得墨忒耳定律到底是什么，以及为什么有些链式调用没问题，有些却会变成烂代码，配上具体例子一起看。

先把结论放在前面。

> 得墨忒耳定律的核心是“不要跟陌生对象说话”。
>
> 链式调用本身并不是问题，问题在于用链式调用一路深入到别人的内部结构时，才会变成烂代码。

只要记住这一句话，这篇文章的一半内容就算掌握了。

---

## 什么是得墨忒耳定律

得墨忒耳定律（Law of Demeter）也被称为“最少知识原则”。

说得直白一点就是这样。

一个对象，应该只和自己直接认识的“亲近朋友”打交道。

一旦开始牵扯到朋友的朋友、朋友的朋友的朋友，问题就来了。

在方法内部可以调用的对象，通常归纳为以下几类。

- 自身（self）的方法
- 作为方法参数传进来的对象
- 方法内部直接创建的对象
- 自己持有的属性（成员）对象

也就是说，只在这个范围内对话就够了。

由此也诞生了那句著名的经验法则——“一行代码只用一个点”。

当然，点的数量终究只是一个提示，并不是绝对规则。

## 链式调用为什么会变成烂代码？

下面用我实际遇到过的代码来说明。

场景是：从订单里取出客户所在城市的名称。

```swift
// 订单 → 客户 → 地址 → 城市，一路深入下去
let city = order.customer
                .address
                .city
                .name
```

乍一看挺简洁的。

但这一行代码意味着，Order 不仅知道 Customer 的内部结构，还知道其中的 Address，甚至再往里的 City。

问题就在这里爆发。

一旦 Address 的结构发生变化，或者 City 被移除，这段代码也会跟着崩溃。

因为手已经伸进了别人家的抽屉深处，那家的结构一变，自己的代码就跟着出问题。

这种沿着对象图一路往下挖的链式调用，通常被称为“火车事故（train wreck）”。

## 那要怎么区分好的链式调用和坏的链式调用？

这是今天这篇文章里最重要的部分。

因为并不是所有的链式调用都是坏的。

判断标准只有一个。

**“在链式调用的过程中，是不是一直在掏别人的内部对象？”**

像前面的例子那样，customer → address → city，不断深入各种不同的、属于别人的对象，这就违反了原则。

反过来，如果链式调用处理的是同一种类型的对象，并且每次都返回自身，那就没问题。

```swift
// filter·map 链式调用每次返回的都是“同一语境”，并不违反原则
let names = users
    .filter { $0.isActive }
    .map(\.name)
```

这种高阶函数链式调用，或者构建者（Builder）模式，不管点有多少个，都不会去翻别人的内部结构。

它只是每次都返回自身（同一条流程），继续往下延伸而已。

把两种代码放在表格里对比一下，就是这样。

| 区分 | 坏的链式调用 | 好的链式调用 |
| --- | --- | --- |
| 对象 | 不断掏出别人的内部对象 | 返回同一流程/类型 |
| 例子 | order.customer.address | filter { }.map { } |
| 耦合度 | 高（对结构变化脆弱） | 低 |
| 判定 | 违反得墨忒耳定律 | 不违反 |

---

## 违反原则的代码，这样修复

解决方法其实比想象中简单。

只要记住“别问，直接吩咐（Tell, Don't Ask）”这句话就行。

不是把内部结构掏出来自己处理，而是拜托对方把这件事做好。

以前面城市的例子来说，可以这样改。

```swift
// 只向 Order 询问需要的结果
let city = order.shippingCity
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="别问，直接吩咐——原来一行代码就够了">
  <figcaption>别问，直接吩咐——原来一行代码就够了</figcaption>
</figure>

让 Order 内部自己去经过 Customer 和 Address，把城市结果返回出来。

这样一来，即使 Address 的结构发生变化，外部代码也不会受影响。

因为需要修改的范围，被牢牢限制在 Order 内部。

亲自这样重构过之后就会发现，日后调整结构时需要改动的地方明显少了很多。

---

最后总结一下。

得墨忒耳定律说的不是要数点的个数，而是提醒我们不要随意去动别人的内部结构。

下次再碰到很长的链式调用时，只需要问自己一个问题。

“这是不是在打开别人家的抽屉？”

光是问自己这一句话，代码就会清爽很多。
