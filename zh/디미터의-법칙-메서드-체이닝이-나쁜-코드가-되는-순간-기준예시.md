做代码评审时，多少都会遇到这样的指摘。

"这一行,点(.)是不是太多了?"

我一开始也不太明白这有什么问题。一行写到底反而看起来更简洁嘛。

但这背后的核心，正是**得墨忒耳定律(Law of Demeter)**。

今天就借着几个例子,聊聊得墨忒耳定律到底是什么,以及为什么有些方法链没问题,有些却会变成烂代码。

先说结论。

> 得墨忒耳定律说的是"不要跟陌生对象搭话"。
>
> 方法链本身并不是问题,问题在于借着链式调用一路挖到别人内部的时候。

只要记住这一句话,今天这篇文章就已经理解了一半。

---

## 什么是得墨忒耳定律

得墨忒耳定律也被称为"最少知识原则"。

说得直白一点就是:

一个对象只应该和自己直接认识的"熟人"打交道。

一旦开始动用朋友的朋友、朋友的朋友的朋友,问题就来了。

通常可以在方法内部调用的对象,大致可以归纳为以下几种。

- 自身(this)的方法
- 作为方法参数传进来的对象
- 方法内部自己创建的对象
- 自己持有的字段(成员)对象

只在这个范围内对话就行了。

由此也衍生出一条有名的经验法则——"一行代码只用一个点"。

当然,点的数量终究只是一个提示,并不是绝对规则。

---

## 方法链为什么会变成烂代码?

我用自己实际遇到过的代码来说明一下。

场景是从订单里取出客户所在城市的名字。

```java
// 订单 → 客户 → 地址 → 城市,一路挖到底
String city = order.getCustomer()
                   .getAddress()
                   .getCity()
                   .getName();
```

乍一看挺简洁的。

但这一行代码意味着,Order 不仅知道 Customer 的内部结构,还知道里面的 Address,甚至 Address 里的 City。

问题就出在这里。

一旦 Address 的结构发生变化,或者 City 被移除了,这段代码也会跟着崩掉。

就好比把手伸进了别人家的抽屉,那家的结构一变,自己的代码就跟着出问题。

这种顺着对象图一路往下挖的链式调用,常被称为"火车事故(train wreck)"。

## 那好的链式调用和不好的链式调用该怎么区分?

这是今天这篇文章最重要的部分。

方法链并不是全都不好。

判断标准只有一个。

**"这个链式调用,是不是在不断掏出别人的内部对象来用?"**

像前面的例子那样,靠 getCustomer → getAddress → getCity 不断深入到各种不同的他人对象里,就是违反了这个原则。

反过来,下面这种处理同一类对象、每次都返回自身的链式调用,则完全没问题。

```java
// 流(Stream)、构建器每次返回的都是"同一个语境",所以不算违反
List<String> names = users.stream()
                          .filter(User::isActive)
                          .map(User::getName)
                          .toList();
```

流(Stream)或构建器(Builder)模式,不管点有多少个,都不会去翻别人的内部结构。

它们每次返回的都是自己(同一个流程),只是不断延续下去而已。

把两种代码放在一张表里对比,大致是这样。

| 区分 | 不好的链式调用 | 好的链式调用 |
| --- | --- | --- |
| 对象 | 不断掏出别人的内部对象 | 返回同一流程/类型 |
| 示例 | order.getCustomer().getAddress() | stream().filter().map() |
| 耦合度 | 高(结构一变就脆弱) | 低 |
| 判定 | 违反得墨忒耳定律 | 不算违反 |

---

## 违反规则的代码,这样改

解决办法其实比想象中简单。

只要记住"别问,直接叫它做(Tell, Don't Ask)"就行。

不要把内部信息掏出来自己用,而是拜托对方帮你把事情做完。

拿前面城市的例子来说,可以这样改。

```java
// 只向 Order 索要需要的结果
String city = order.getShippingCity();
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="别问,直接叫它做——原来一行代码就够了">
  <figcaption>别问,直接叫它做——原来一行代码就够了</figcaption>
</figure>

让 Order 内部自己去处理 Customer 和 Address,再把城市名返回出来。

这样一来,即使 Address 的结构发生变化,外部代码也不会受影响。

因为修改范围被牢牢限制在 Order 内部。

我自己这样重构过之后发现,后面调整结构时需要动的地方明显少了很多。

---

最后总结一下。

得墨忒耳定律说的不是让你去数点的数量,而是提醒你别随便伸手动别人的内部结构。

下次再遇到很长的链式调用时,只要问自己一个问题就够了。

"这是不是在打开别人家的抽屉?"

光是问出这一句,代码就会清爽不少。
