一个800行的视图文件里塞满了所有东西——这样的代码,想必大家多少都见过。

负责画界面的视图代码里混着API调用,旁边又是折扣率计算逻辑,再往下是日期格式化函数,中间还穿插着分析事件的发送。哪怕只想改一个小地方,也得把800行全部读一遍。

这类文件违反的正是关注点分离原则,英文叫Separation of Concerns。开发原则系列的最后一篇,讲的就是这个原则。

关注点分离(SoC)的意思是:把程序按不同的关注点拆分开,让每个部分只处理一件事。

所谓“关注点”,指的是程序需要操心的那类事情。界面怎么展示、数据从哪里取、业务规则怎么应用——这些都是彼此独立的关注点。

提出这个术语的人是计算机科学家迪杰斯特拉(Dijkstra),他在1974年的一篇文章里这样写道:

> 一次只专注于一个方面去思考——这是我所知道的唯一有效的思维技巧。

人的大脑没办法同时处理多个关注点,所以代码也应该做到:一段代码只装一个关注点。

---

## 从熟悉的例子说起:HTML、CSS、JS

如果做过Web开发,其实你早就每天都在使用关注点分离了。

| 技术 | 关注点 |
|------|--------|
| HTML | 结构——有什么内容 |
| CSS | 表现——长什么样 |
| JavaScript | 行为——怎么响应 |

像以前那样把`style`属性和`onclick`一股脑塞进HTML标签里,哪怕只想改个按钮颜色,也得翻遍HTML。而拆成三个文件之后,设计师只看CSS就行,负责标记的人只看HTML就够了。

后端的控制器-服务-仓储结构,iOS里把视图和视图模型、数据层分开——都是同一个原理。什么MVC、分层架构,说到底都是把关注点分离固定成模式而已。

## 给前面那个800行视图动手术

把前面提到的那个800行视图,按关注点拆开看看。

```swift
// Before: 所有关注点都挤在一个文件里
struct ProductPage: View {
    // 关注点1:获取数据(网络请求、加载状态、错误处理,150行)
    // 关注点2:业务规则(折扣率、库存计算,200行)
    // 关注点3:分析事件(点击追踪,100行)
    // 关注点4:绘制界面(body,350行)
}
```

```swift
// After: 每个关注点各归其位
ProductStore(id:)               // 获取数据
calcDiscount(product, user)     // 业务规则(纯函数)
Tracker.track("product")        // 分析事件

struct ProductPage: View {      // 只负责绘制界面
    @State private var store: ProductStore

    var body: some View {
        let price = calcDiscount(store.product, user)
        ...
    }
}
```

拆开之后好处很明显。折扣政策一变,只需要看`calcDiscount`就行,而这个函数和界面无关、是个纯函数,测试起来也很方便。API规格一变,改`ProductStore`就够了。相比过去要读完800行,现在需要改动的范围缩小到了原来的十分之一。

## 到底该拆到什么程度?

关注点分离里最难的问题是“边界到底该划在哪里”。拆得太细,就会变成一个要在几十个文件之间来回翻找的迷宫。

我用的标准是“变化的理由”,这和SOLID那篇里讲的SRP其实是同一个问题:

> 这两段代码,是不是出于同一个理由、在同一个时间点发生变化?

折扣政策和界面布局的变化理由不同,所以要分开。反过来,一个按钮和它的点击处理函数几乎总是一起变化,所以放在一起就行。SwiftUI以视图为单位,反而把结构、样式、行为都塞进同一个文件——从这个角度看也并不矛盾,只是换了一种划分方式:不是按技术种类,而是按“一起变化的单位”重新划定了关注点。

## 系列收尾

关于关注点分离,想留下的核心是这些:

- 一段代码只处理一个关注点。人脑做不到真正的多任务。
- 边界不该按技术种类划,按“一起变化的单位”划更实用。
- 拆得太过头会变成迷宫。拆分本身不该成为目的。

开发原则系列到这里就告一段落了。KISS(保持简单)、DRY(知识只放一处)、YAGNI(只做现在需要的)、SOLID(把变化的影响范围控制到最小)、最小惊讶原则(让代码按预期运行),再到关注点分离(一次只做一件事)。

回头看看,这六条原则其实都指向同一个地方:代码是给人读的,不是给计算机读的;好代码,就是容易修改的代码。原则的名字忘了也没关系,记住这一句就够了。
