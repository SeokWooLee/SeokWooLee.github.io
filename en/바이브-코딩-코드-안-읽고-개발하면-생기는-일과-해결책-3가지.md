Building by hitting Accept All on whatever AI writes, without reading a line of it — that's what people now call "vibe coding." Getting started is astonishingly fast, and plenty of people say once you've tasted it, there's no going back. But it's worth knowing upfront where and how you end up paying for not reading the code.

This post walks through the precise definition of vibe coding and its upsides, three problems that actually show up when you don't read your code, and three fixes that cut down the risk without requiring you to read everything.

Let's start with the conclusion.

> Vibe coding gets you started 10x faster, but if you never read the code, you end up paying that speed back as maintenance cost.

For a prototype or a tool only you'll use, it's well worth trying. But if you're shipping something to other people or maintaining code long-term, that's a different story. Let's break down why.

## What exactly is vibe coding?

Let's start with the term. "Vibe coding" traces back to a tweet Andrej Karpathy posted in February 2025.

The core idea: you tell an AI what you want in natural language, the AI writes the code, and you don't carefully read that code.

Karpathy himself put it this way: "I just see stuff, say stuff, run stuff, and copy paste stuff, and it mostly works." ... he's said he always hits Accept All and stops reading diffs. When an error pops up, he just copies the message and throws it back at the AI.

For what it's worth, Collins Dictionary named it 2025's Word of the Year.

The essence of vibe coding isn't really "AI coding" — it's closer to "not reading the code." If you're using AI but still reading every diff, that's just AI-assisted development, not vibe coding.

---

## The upsides are real

There's a reason vibe coding has caught on.

**First, speed to start.** Initial project setup that used to take a full day now takes tens of minutes. The time from zero to something showing on screen shrinks dramatically.

**Second, it fills in your weak spots.** A backend developer can describe a CSS layout in words and have it generated, which drastically lowers the barrier to entry in areas you're not comfortable in.

**Third, the psychological hurdle drops.** The "let's just build it" mindset grows stronger, and ideas that had been sitting on the shelf actually get touched.

The catch: this satisfaction usually holds only in the early stage of a project, while the codebase is still small.

---

## What actually happens when you don't read the code

Once a codebase grows past a certain size, three problems consistently show up.

**1. You become unable to fix bugs.** If your approach to an error is just to hand it straight to the AI, you get stuck the moment the AI can't fix it either. Since you've never read the code, you have no sense of where the problem even is.

**2. The same functionality ends up duplicated in multiple places.** AI often doesn't remember code it wrote earlier, so it keeps creating similar functions from scratch. Fix one later, and the rest are left untouched.

**3. Silent security holes creep in.** A classic example is an API key hardcoded straight into client-side code. The AI just does what it's told, and no one checks it.

The third one is the scariest. Here's an example.

```swift
// AI-generated code — key exposed on the client
let apiKey = "sk-live-abc123"  // baked directly into the app binary
let url = URL(string: "https://api.example.com?key=\(apiKey)")!
URLSession.shared.dataTask(with: url).resume()
```

The moment code like this ships, anyone can see the key. If you never read the diff, this is exactly the kind of thing you miss.

---

## So what do you do? 3 fixes

The good news: you don't have to go all the way back to "read everything" to cut the risk dramatically.

**Fix 1: Read only the three places where a mistake is catastrophic.** You can't read everything, but you can decide in advance on the handful of spots where a mistake is hard to undo — and read only those.

```swift
// At minimum, check these three areas yourself
// 1) Authentication / key-related code
// 2) Payment / money-handling logic
// 3) Anything that deletes or changes user data
```

Covering just these three areas dramatically cuts the odds of a catastrophic mistake.

**Fix 2: Read the AI's summary instead of the code.** If you really don't want to read the code, at least make one more ask: "Summarize just the security, financial, and data-deletion risks in the code you just wrote," and read that summary instead. Ideally, have a fresh session — or even a different AI — do this review rather than the session that wrote the code, which also reduces the bias of an AI defending its own work.

**Fix 3: If a human won't read it, at least make a machine read it.** Wiring a secrets scanner like gitleaks into a pre-commit hook automatically catches the kind of API key exposure shown above at commit time. Running linters and tests in CI works on the same principle.

What all three fixes have in common: they cut down reading effort, but they never bring verification to zero.

## So, should you actually vibe code or not?

The answer is: it depends on the situation.

| Situation | Recommendation | Reason |
|---|---|---|
| Prototype / demo | Strongly recommended | Fine to build fast and throw away |
| Personal-use tool | Recommended | If it breaks, only you're affected |
| Side project | Conditional | Read the core logic |
| Production service / team collaboration | Not recommended | Maintenance cost eats the speed gain |

The key distinction is between "not reading any of the code" and "reading only what matters."

Q. Can beginners learn to code through vibe coding?
A. It's a great way to feel the fun of building things. But if you never read the code at all, your skills won't really improve. It helps to pair it with a habit of asking the AI, "Why did you write it this way?" after the fact, and actually reading the answer.

Q. How much can you trust AI-written code?
A. Working and safe are two different things. You can trust the AI on whether the code runs — but whether it's safe and maintainable needs to be confirmed by a person or a separate verification step.

---

To sum it up: leave the speed to the AI, but keep the judgment for yourself. You don't need to read everything, but never reading anything eventually costs you. Wiring in even one of the three fixes above today can cut that cost significantly.

Taking it a step further, there's also a methodology called spec-driven development (SDD), where you nail down requirements in a document before ever telling the AI to write code. It's the opposite approach to vibe coding — I'll cover it in detail in the next post.
