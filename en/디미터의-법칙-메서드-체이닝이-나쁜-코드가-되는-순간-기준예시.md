If you've ever gone through a code review, you've probably heard this comment at least once.

"There are a lot of dots (.) in this line, don't you think?"

Honestly, I didn't get why that was a problem at first. Chaining everything into one line actually looked cleaner to me.

But the core idea behind that comment is exactly the **Law of Demeter**.

Today I want to walk through what the Law of Demeter actually is, and why some method chains are fine while others turn into bad code — with real examples.

Let me start with the conclusion.

> The Law of Demeter is basically the rule "don't talk to strangers."
>
> Method chaining itself isn't the problem — it becomes bad code when the chain reaches deep into someone else's internals.

If you remember just that one sentence, you've already got half of today's post covered.

---

## What Is the Law of Demeter?

The Law of Demeter is also known as the "principle of least knowledge."

In plain terms, it goes like this:

An object should only talk to its immediate "friends" — the things it directly knows about.

Once you start reaching for a friend's friend, and that friend's friend again, things start to break down.

The objects a method is generally allowed to call are:

- Its own (this) methods
- Objects passed in as parameters
- Objects it creates directly inside the method
- Objects it holds as its own fields (members)

The idea is to keep conversations strictly within that scope.

This is also where the well-known rule of thumb "one dot per line" comes from.

Of course, the dot count is just a rough hint, not an absolute rule.

---

## Why Does Method Chaining Turn Into Bad Code?

Let me show you with code I actually ran into.

The situation was pulling a customer's city name out of an order.

```java
// Digs all the way through Order → Customer → Address → City
String city = order.getCustomer()
                   .getAddress()
                   .getCity()
                   .getName();
```

At a glance, it looks clean.

But this single line means Order now knows about the internals of Customer, and inside that, Address, and inside that, City — all the way down.

And that's exactly where things break.

If the Address structure changes, or City goes away, this code breaks right along with it.

It's like reaching into someone else's drawers — if they rearrange their house, your code falls apart.

This kind of chaining that climbs down through an object graph is commonly called a "train wreck."

## So How Do You Tell Good Chaining from Bad Chaining?

This is the most important part of today's post.

Not all method chaining is bad.

There's really just one criterion:

**"Is the chain repeatedly reaching into someone else's internal objects?"**

Like the earlier example — getCustomer → getAddress → getCity — if you're drilling into a series of different objects that belong to someone else, that's a violation.

On the other hand, chaining that keeps operating on the same kind of thing and keeps returning itself, like below, is perfectly fine.

```java
// Streams and builders return the 'same context' each time, so this is not a violation
List<String> names = users.stream()
                          .filter(User::isActive)
                          .map(User::getName)
                          .toList();
```

No matter how many dots a stream or builder pattern has, it never digs into someone else's internals.

It just keeps returning itself — the same flow — over and over.

Here's a side-by-side comparison of the two:

| | Bad chaining | Good chaining |
| --- | --- | --- |
| Target | Keeps pulling out someone else's internal objects | Returns the same flow/type |
| Example | order.getCustomer().getAddress() | stream().filter().map() |
| Coupling | High (fragile to structural change) | Low |
| Verdict | Violates Demeter | Not a violation |

---

## How to Fix a Violation

The fix is simpler than you might expect.

Just remember: "Tell, Don't Ask."

Instead of reaching in and grabbing the internals yourself, ask the object to do the work for you.

For the city example above, you could rewrite it like this:

```java
// Just ask Order for the result you actually need
String city = order.getShippingCity();
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="Tell, don't ask — turns out one line like this is really all it takes">
  <figcaption>Tell, don't ask — turns out one line like this is really all it takes</figcaption>
</figure>

Now Order handles walking through Customer and Address internally and just hands back the city.

That way, even if the Address structure changes, the outside code stays completely untouched.

The blast radius of any change is contained entirely inside Order.

When I actually refactored code this way myself, I noticed how much smaller the area I had to touch got whenever the structure changed later.

---

To wrap up:

The Law of Demeter isn't about counting dots — it's about not reaching into things that aren't yours.

Next time you run into a long chain, just ask yourself one thing:

"Am I opening someone else's drawer here?"

Just asking that one question alone makes your code noticeably cleaner.
