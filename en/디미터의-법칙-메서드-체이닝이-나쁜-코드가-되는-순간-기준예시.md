Anyone who's been through a code review has probably heard this at some point.

"There are an awful lot of dots (.) in this line, aren't there?"

Writing it all on one line seems to make it cleaner, so what's the problem?

The heart of this question is the **Law of Demeter**.

Today, let's walk through what the Law of Demeter is, and why some method chains are fine while others turn into bad code — with examples.

Let's start with the conclusion.

> The Law of Demeter is the rule of "don't talk to strangers."
>
> Method chaining itself isn't bad — it becomes bad code when the chain reaches deep into someone else's internals.

If you remember just this one sentence, you've already got half of today's post covered.

---

## What Is the Law of Demeter?

The Law of Demeter is also known as the "principle of least knowledge."

In simple terms, it means this:

An object should only talk to its immediate "close friends" — the ones it directly knows about.

Problems start once you begin reaching for a friend's friend, and then that friend's other friend.

The objects a method is allowed to call typically fall into these categories:

- Methods on the object itself (self)
- Objects passed in as parameters to the method
- Objects created directly inside the method
- Property (member) objects the object itself holds

The idea is to stick to conversations within this scope only.

This is also where the well-known rule of thumb "only one dot per line" comes from.

Of course, the dot count is just a hint, not an absolute rule.

---

## Why Does Method Chaining Turn Into Bad Code?

Let me show you with code from a real situation.

This was a case of pulling a customer's city name out of an order.

```swift
// Reaches all the way down: order → customer → address → city
let city = order.customer
                .address
                .city
                .name
```

At a glance, it looks clean.

But this single line means that Order knows all about Customer's internals, and Address inside that, and City inside that.

And that's exactly where the problem erupts.

If the Address structure changes, or City disappears, this code collapses right along with it.

Since you're reaching into someone else's dresser drawers, when that person's house layout changes, your code breaks.

This kind of chaining that crawls down through an object graph is commonly called a "train wreck."

## So How Do You Tell Good Chaining From Bad Chaining?

This is the most important part of today's post.

Not all method chaining is bad.

There's a single criterion to judge it by.

**"Does the chain keep pulling out someone else's internal objects?"**

Like the earlier example, if you keep drilling into different objects that belong to someone else — customer → address → city — that's a violation.

On the other hand, chaining that keeps returning the same kind of thing to itself, as below, is fine.

```swift
// filter/map chaining always returns the 'same context,' so it's not a violation
let names = users
    .filter { $0.isActive }
    .map(\.name)
```

Higher-order function chains like this, or builder patterns, don't dig into anyone else's internals no matter how many dots pile up.

Each step just keeps returning itself (the same flow) and continuing on.

Here's a comparison of the two in table form.

| Category | Bad Chaining | Good Chaining |
| --- | --- | --- |
| Target | Keeps pulling out someone else's internal objects | Returns the same flow/type |
| Example | order.customer.address | filter { }.map { } |
| Coupling | High (fragile to structural change) | Low |
| Verdict | Violates Demeter | Not a violation |

---

## How to Fix Violating Code

The fix is simpler than you'd think.

Just remember "Tell, Don't Ask."

Instead of pulling out the internals yourself, ask the object to do the work for you.

For the city example above, you could rewrite it like this.

```swift
// Just ask Order for the result you need
let city = order.shippingCity
```

<figure>
  <img src="/assets/images/posts/13852154-b102-4b52-be80-085e3027ac40/3.png" alt="Tell, don't ask — turns out one line is really all it takes">
  <figcaption>Tell, don't ask — turns out one line is really all it takes</figcaption>
</figure>

Let Order handle going through Customer and Address internally and return the city itself.

This way, even if the Address structure changes, the outside code stays untouched.

The scope of any change stays neatly contained inside Order.

Refactoring things this way in practice tends to shrink the number of places you need to touch the next time the structure changes.

---

To wrap up:

The Law of Demeter isn't a rule about counting dots — it's about not reaching into someone else's internals without permission.

Next time you run into a long chain, just ask yourself one question.

"Am I opening someone else's drawer here?"

Asking that one question alone makes your code noticeably cleaner.
