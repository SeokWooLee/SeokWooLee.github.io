"When am I supposed to fit this in..." You're already tight on time just building the feature, and then someone tells you to write tests on top of it — honestly, the first reaction is a sigh.

To cut to the point, the real reason to write tests is to protect your future self. You spend 10 extra minutes now to prevent an outage that would otherwise blow up at 3 a.m. three months down the road.

## Why write tests in the first place?

The biggest reason is that changing code stops being scary.

Code is never really "done" after you write it once. It keeps getting modified, and features keep getting bolted on.

Without tests, every single line you touch comes with a nagging worry: "If I change this, will something else break?"

With tests, one run after the change tells you the answer. Green means you're safe, red means you immediately know what broke.

<figure>
  <img src="/assets/images/posts/9cb89987-2d6c-4e11-ae8e-b6d95c4e0868/4-1783847709666.png" alt="Change it, run it once — green means good, red means you know exactly where to look.">

This is called a "regression test." The whole point is catching cases where something that used to work gets broken again.

## The real reason people write tests despite the hassle

Honestly, writing tests can feel like a loss in the moment — it's not like it adds any feature you can actually see.

But that changes as the project grows.

Once you're up to 30 or 50 features, checking everything by hand becomes impossible. You can't click through every single screen every time you change one thing.

That's where tests come in — a single command verifies dozens of cases in seconds.

The second reason is that test code doubles as documentation.

A well-written test shows you, at a glance, what output a function produces for a given input.

> Comments can go stale and start lying to you, but tests actually run, so they can't.

Here's a simple example — a test for a function that calculates a discount amount.

```swift
// 10% off 10,000 won should give 9,000 won
func test_10퍼센트_할인을_적용한다() {
    let result = applyDiscount(10000, rate: 0.1)
    XCTAssertEqual(result, 9000)
}
```

Just from this test, it's immediately clear what applyDiscount is supposed to do.

## How do you actually get started with tests?

If you try to test everything from day one, you'll burn out and give up.

Here's the order I'd recommend starting with:

1. Core logic where a failure would be a disaster — money calculations, login, etc.
2. Functions with a lot of confusing branching logic
3. Places that have had a bug before (to prevent it from coming back)

On the flip side, simple screen code or parts that change frequently can wait.

The goal isn't 100% coverage — it's weighing cost against benefit.

## FAQ

**Q. Doesn't writing tests slow development down?**

A. At first, yes. But by the middle of the project, it actually speeds things up, because the time spent finding and fixing bugs drops sharply.

**Q. What's a good test coverage percentage?**

A. There's no need to obsess over the number. 70–80% is plenty — what matters more is whether the important logic is actually covered.

Test code isn't about showing off your skills — it's a safety net for your future self and your teammates.

Just try attaching one test to one function you wrote today. Once you feel the relief that little green light gives you, you'll understand firsthand why everyone keeps telling you to write tests.
