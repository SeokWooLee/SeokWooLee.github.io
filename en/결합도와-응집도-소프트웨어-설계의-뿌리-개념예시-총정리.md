The code clearly works, but why do you have to touch files all over the place just to fix one feature?

You change a button color and the checkout logic breaks — most of us have been there at least once.

The concept that explains this in just two words is coupling and cohesion.

> Coupling should be low, and cohesion should be high. About 80% of good design starts from this one sentence.

In this post, I'll walk through exactly what these two concepts mean, why they're called the root of software design principles, and how to apply them to real code — drawing on my own experience along the way.

Let's start with the essentials.

1. Coupling: how entangled modules are with each other → the **lower**, the better
2. Cohesion: how focused the code within a single module is on one job → the **higher**, the better
3. These two ideas underpin well-known principles like SOLID and various design patterns
4. The end goal is really just one thing: "code that's easy to fix and easy to swap out"

---

## So What Exactly Are Coupling and Cohesion?

Let's summarize both concepts in a single sentence first.

Coupling refers to the relationship between modules, while cohesion refers to the internal cohesiveness within a single module.

Since they point in opposite directions, they're easy to mix up. Here's how I remember them.

Coupling is "the distance to the outside," and cohesion is "the tightness inside."

When coupling is high, fixing A drags B and C along with it — like a chain of dominoes.

When cohesion is low, a single class ends up with payment logic, email sending, and logging all tangled together. Just looking at the name, you can't tell what the class actually does.

Good design pushes these two in opposite directions: loose on the outside, tight on the inside.

---

## Why Is This the "Root" of Design Principles?

Coupling and cohesion were first formalized in structured design theory back in the 1970s, credited to Larry Constantine.

Even though nearly half a century has passed, these ideas still hold up — because they capture something essential that isn't tied to any particular language or trend.

If you break down the well-known principles we all know, they ultimately converge here.

| Principle/Pattern | What It's Really Saying |
|---|---|
| Single Responsibility Principle (SRP) | Increase cohesion |
| Dependency Inversion (DIP) | Reduce coupling |
| Interface Segregation (ISP) | Cut unnecessary coupling |
| Most design patterns | Low coupling + high cohesion |

See? The names differ, but the root is the same.

That's why, whenever I learn a new principle or pattern, I first ask myself: "Is this trying to lower coupling, or raise cohesion?" That question alone gets me halfway to understanding it.

## How Do You Actually Lower Coupling in Code?

Talking about this abstractly doesn't help much, so let's look at a short example.

Here's code with high coupling. The Order class directly knows about a specific payment provider.

```swift
class Order {
    let pay = KakaoPay()  // tightly bound to KakaoPay
    func checkout() {
        pay.send()        // switching payment providers means tearing this apart
    }
}
```

The moment you switch payment providers to Toss, you have to open up Order and modify it. This is a textbook case of high coupling.

This time, let's put some distance between them using a protocol.

```swift
class Order {
    let pay: Payment                // depends only on the 'Payment' contract
    init(pay: Payment) { self.pay = pay }
    func checkout() { pay.send() }  // doesn't matter what comes in
}
```

Now Order doesn't care which payment provider is plugged in. You just swap it out.

After making this kind of change in real projects, I found that even when new payment provider requests came in, I barely had to touch the existing code. That's the power of low coupling.

## How Do You Raise Cohesion?

Cohesion is judged by asking: "does this module do exactly one job?"

My rule of thumb is simple: if you say the class name out loud and all the methods inside it naturally come to mind, that's high cohesion.

For example, if a UserService has all of this mixed in, that's a red flag.

- Handling sign-up (its actual job)
- Sending marketing emails (someone else's job)
- Generating CSV reports (also someone else's job)

It's like three unrelated tenants sharing one house.

In this case, give email its own room in EmailSender, and reports their own room in ReportGenerator.

Then, when you need to fix the email logic later, you won't need to open UserService at all. The blast radius of any change gets smaller.

Low coupling and high cohesion ultimately point in the same direction: stopping changes from spreading.

## Frequently Asked Questions (Q&A)

**Q. Between coupling and cohesion, which should I focus on first?**

I'd recommend tackling cohesion first. As you break a module down so it focuses on one job, coupling often naturally sorts itself out along the way.

**Q. Is zero coupling always ideal?**

No. If modules never connect to each other at all, the program simply won't run. The goal isn't zero — it's "only as much as necessary, and loose."

**Q. Do I need to design everything perfectly from the start?**

Not at all. I usually get something working first, then gradually cut coupling and gather cohesion whenever fixing something starts to hurt. Refactoring is inherently an iterative process.

Coupling and cohesion aren't flashy new technologies, but they are the shared habit of people who write code that survives the long haul.

Next time you write some code, try asking yourself just once: "If I change this, how far will the impact spread?" Keep asking that question, and your sense for design will grow tremendously. Rooting for you!
