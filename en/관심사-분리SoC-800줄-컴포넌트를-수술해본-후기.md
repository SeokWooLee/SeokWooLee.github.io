You've probably seen it before: a single 800-line view file with everything crammed inside.

API calls sitting right in the middle of the view code that draws the screen, discount-rate calculation logic parked next to it, a date-formatting function underneath that, and analytics event calls scattered throughout. Fix one thing, and you end up having to read all 800 lines.

The principle a file like this violates is Separation of Concerns (SoC). This is the final installment in the development principles series, and it's the one we're covering.

Separation of Concerns says: split a program into distinct units of concern, so each part deals with only one concern.

A "concern" is a category of thing the program has to worry about. How to display the screen, where to fetch data from, how to apply business rules — these are all separate concerns.

The term was coined by computer scientist Edsger Dijkstra, who put it this way in a 1974 paper:

> Let me try to explain to you what, to my taste, is characteristic for all intelligent thinking. It is, that one is willing to study in depth an aspect of one's subject matter in isolation for the sake of its own consistency, all the time knowing that one is occupying oneself only with one of the aspects.

The human mind can't juggle multiple concerns at once. So the idea is to keep each piece of code focused on a single concern too.

---

## Starting with a familiar case: HTML, CSS, JS

If you've done any web development, you've been practicing separation of concerns every day without necessarily naming it.

| Technology | Concern |
|------|--------|
| HTML | Structure — what exists |
| CSS | Presentation — how it looks |
| JavaScript | Behavior — how it responds |

Back when people plastered `style` attributes and `onclick` handlers directly onto HTML tags, changing a single button's color meant digging through the HTML. Split things into three files, and the designer only needs to touch CSS while whoever owns markup only needs to touch HTML.

The controller-service-repository structure on the backend, and the separation of view from view model and data layer on iOS — they all follow the same principle. Patterns like MVC or layered architecture are, at bottom, formalized versions of separation of concerns.

## Operating on that 800-line view

Let's take the 800-line view from earlier and split it by concern.

```swift
// Before: every concern crammed into one file
struct ProductPage: View {
    // Concern 1: fetching data (network calls, loading, error handling — 150 lines)
    // Concern 2: business rules (discount rate, stock calculation — 200 lines)
    // Concern 3: analytics events (tap tracking — 100 lines)
    // Concern 4: drawing the screen (body — 350 lines)
}
```

```swift
// After: each concern finds its own home
ProductStore(id:)               // fetching data
calcDiscount(product, user)     // business rules (pure function)
Tracker.track("product")        // analytics events

struct ProductPage: View {      // only draws the screen
    @State private var store: ProductStore

    var body: some View {
        let price = calcDiscount(store.product, user)
        ...
    }
}
```

Once it's split apart, the benefits are obvious. If the discount policy changes, you only need to look at `calcDiscount` — and since it's a pure function unrelated to the screen, it's easy to test too. If the API spec changes, you only need to touch `ProductStore`. Compared to reading all 800 lines, the scope of any given change shrinks to a tenth of what it was.

## Where do you draw the line?

The hardest part of separation of concerns is deciding where exactly to draw the boundaries. Split things too aggressively, and you end up with a maze you have to navigate across dozens of files just to understand one flow.

The criterion I use is the reason for change. It's the same question the Single Responsibility Principle asked in the SOLID installment of this series:

> Do these two pieces of code change for the same reason, at the same time?

Discount policy and screen layout change for different reasons, so they get separated. A button and its tap handler, on the other hand, almost always change together, so they stay bundled. Seen from this angle, it's not a contradiction that SwiftUI actually bundles structure, style, and behavior together within a single view file. It's redrawing the concern boundary around units that change together, rather than around technology type.

## Wrapping up the series

The core takeaway on separation of concerns is this:

- Keep each piece of code focused on a single concern. The human mind isn't built for multitasking.
- In practice, it's more useful to draw boundaries around "units that change together" than around technology type.
- Over-splitting creates a maze. Splitting things apart can't become a goal in itself.

And with that, the development principles series comes to a close: KISS (keep it simple), DRY (keep knowledge in one place), YAGNI (build only what you need now), SOLID (keep the blast radius of change small), Principle of Least Astonishment (make things behave as expected), and now Separation of Concerns (do one thing at a time).

Looking back, all six principles point at the same thing: code is read by people, not computers, and good code is code that's easy to change. You can forget the names of the principles — just hold onto that one sentence.
