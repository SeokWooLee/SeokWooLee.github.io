Ever spent a whole night stuck because a singleton wouldn't let you write a proper test?

The logic itself is fine, but the moment you try to attach a test, it keeps hitting a real database or calling a real API.

Let's cut to the point.

> Because a singleton pulls its object straight from inside the code, you can't swap it out for a fake when testing.

Dependency injection, on the other hand, hands the object in from the outside, so you can swap in whatever you want during a test.

Today we'll walk through the difference between the two, slowly, with real code.

## Why do singletons block testing?

A singleton is an object that exists exactly once across the entire program.

It's convenient — you can grab it anywhere with a single line like `Database.shared`.

But that's exactly where the trouble starts.

When a function reaches for the object directly from inside itself, that function ends up welded to the real database.

There's no way to tell it, "use a fake DB instead of the real one" during a test.

```swift
// Grabbing the singleton directly inside the function
func saveOrder(_ order: Order) {
    // This is where it gets welded to the real DB
    let db = Database.shared
    db.insert(order) // No way to block this in a test
}
```

To test this code, the real DB has to be up and running.

If the DB is down, the test fails right along with it.

The logic isn't wrong — the test just turns red because of infrastructure.

---

## So how do you do dependency injection?

Dependency injection is a lot less grand than it sounds.

Instead of reaching for the object from inside the function, you just build it outside and hand it in.

That's literally what "injection" means here — you're injecting what's needed.

```swift
// Receive the DB as a parameter from outside
func saveOrder(_ order: Order, db: Database) {
    db.insert(order) // The caller decides which DB this is
}
```

The difference is just one line, right?

All we did was change from pulling `Database.shared` directly to receiving it as a parameter.

But that small change completely transforms how testing works.

In production code you pass in the real DB, and in tests you pass in a fake one.

  <img src="/assets/images/posts/bb76dac7-edf6-4e9c-b700-37fe51837a6f/4-1783847645964.png" alt="Just changing what gets passed in makes all the difference.">

</figure>

Receiving it through an initializer is another common approach.

```swift
class OrderService {
    private let db: Database
    // Received from outside when the object is created
    init(db: Database) { self.db = db }
}
```

Once you receive it there, you can use `self.db` anywhere inside the class, which makes things much cleaner.

## Swapping in a fake object for tests

Now for the genuinely fun part.

Since the dependency is now received from outside, in tests you can pass in a fake DB (usually called a mock) instead of the real one.

```swift
func test_savingAnOrder_passesItToTheDB() {
    let fake = FakeDatabase() // Not the real thing
    saveOrder(Order(), db: fake)
    XCTAssertEqual(fake.savedCount, 1) // Just check that save was called
}
```

There's no need to spin up a real DB.

No network, no external server required. It finishes instantly, entirely in memory.

That makes tests faster and, more importantly, much more stable.

You can verify the logic alone, completely independent of external conditions.

## Singleton vs. dependency injection, side by side

Here's the difference laid out in a table.

| Aspect | Using a singleton directly | Dependency injection |
|---|---|---|
| Where the object comes from | Pulled directly inside the function | Passed in from outside |
| Swapping in a fake | Practically impossible | Freely possible |
| Test speed | Slow (needs a real resource) | Fast (handled in memory) |
| Test stability | Shaky, depends on external conditions | Only the logic gets verified |
| Coupling | High | Low |

A singleton isn't inherently bad.

For something like a config value, where there really should only be one instance, it's genuinely convenient.

The problem is the habit of reaching for it directly from anywhere — that's what ties testing's hands.

---

## FAQ

**Q. Do I need a DI library like Swinject to do dependency injection?**

No. Passing something in through a parameter or an initializer, like in today's code, already counts as dependency injection. A library just automates the management of that.

**Q. Doesn't it get messy once there are too many parameters?**

True. That's usually handled by grouping things together through constructor injection, or bundling related dependencies together. If the parameter list keeps growing, it's also a sign that class is doing too much.

If your tests keep getting held hostage by real resources, try changing just one thing in the direction we covered today.

Take the code that pulls an object directly, and change it to receive that object from outside instead.

That one small habit is enough to make writing tests actually enjoyable. Happy coding!
