Swift를 배울 때 처음 만나는 관문이 옵셔널입니다. `String?`의 물음표, `if let`, `guard let`, `!`, `??`. 문법 종류가 많아서 각각을 따로 외우다 보면 규칙이 뒤죽박죽되기 쉽습니다.

그런데 이 문법들을 한 번에 정리해주는 사실이 하나 있습니다. 옵셔널은 특별한 문법이 아니라 그냥 enum이라는 것. 표준 라이브러리에 정의된 평범한 타입이고, 물음표는 그 타입의 별명일 뿐입니다. 이 정체를 알고 나면 옵셔널 관련 문법 전부가 하나의 원리에서 파생된 것으로 보이기 시작해요.

이 글은 Swift 기초 시리즈 1편입니다. 옵셔널이 왜 존재하는지(철학), 실체가 뭔지(구현), 언제 어떤 언래핑을 쓰는지(실무 기준)까지 한 번에 정리합니다.

## 왜 존재하나 — "값이 없음"을 타입으로 승격시키기

옵셔널의 존재 이유는 철학 시리즈 1편에서 다룬 안전 우선 원칙의 대표 사례라, 여기서는 요점만 짚습니다.

대부분의 언어에서 "값이 없음"(null, nil)은 어떤 참조에나 숨어들 수 있습니다. String을 받기로 한 함수에 null이 들어와도 타입 시스템은 모르죠. 그래서 null 체크는 문서와 관습, 그러니까 사람의 기억력에 맡겨졌고 그 기억이 어긋난 지점마다 NullPointerException과 크래시가 태어났습니다.

Swift의 해법은 "값이 없을 수 있음"을 타입에 새기는 겁니다. `String`은 반드시 문자열이 있고, `String?`은 없을 수도 있다. 이 둘은 아예 다른 타입이라 서로 그냥 대입도 안 됩니다. 없을 수도 있는 값을 쓰려면 반드시 "있는지 확인하는 절차"를 거쳐야 하고, 그 절차를 빼먹으면 컴파일이 안 돼요. nil 체크가 사람의 기억력에서 컴파일러의 업무로 이관된 겁니다.

## 정체 — Optional은 case 두 개짜리 enum

표준 라이브러리에서 Optional의 선언을 찾아보면 이렇게 생겼습니다(단순화한 형태입니다).

```swift
enum Optional<Wrapped> {
    case none
    case some(Wrapped)
}
```

이게 전부입니다. "값이 없음"을 뜻하는 `none`, "값이 있고 그 값은 이것"을 뜻하는 `some(Wrapped)`. 우리가 쓰는 문법은 전부 이 enum의 설탕(syntactic sugar)이에요.

- `String?`은 `Optional<String>`의 축약 표기입니다.
- `nil`은 `Optional.none`의 별명입니다.
- `var name: String? = "Kim"`은 실제로 `Optional.some("Kim")`을 담는 겁니다.

그래서 옵셔널은 "상자"에 비유됩니다. `String?`은 문자열이 아니라, 문자열이 들었을 수도 비었을 수도 있는 상자예요. 상자에 대고 `.count`를 부를 수 없는 건 당연합니다. 상자는 문자열이 아니니까요. 언래핑(unwrapping)이라 부르는 모든 행위는 결국 "상자를 열어 내용물을 꺼내는 일"이고, enum으로 말하면 `some` case의 연관값을 패턴 매칭으로 꺼내는 일입니다.

실제로 if let은 switch 패턴 매칭의 축약입니다.

```swift
let name: String? = fetchName()

// 원형: enum 패턴 매칭
switch name {
case .some(let value): print(value.count)
case .none: print("이름 없음")
}

// 축약: if let
if let value = name {
    print(value.count)
}
```

둘은 같은 코드입니다. 옵셔널 문법이 어렵게 느껴질 때 enum 원형으로 되돌려 생각하면 대부분 풀립니다.

## 언래핑 도구 상자 — 다섯 가지와 각각의 자리

옵셔널을 여는 방법이 여러 개라 헷갈리기 쉬운데, 각각 쓰이는 자리가 다릅니다. 실무 기준으로 정리할게요.

**1. if let — 있을 때만 잠깐 쓸 때.** 값이 있는 경우의 처리가 그 블록 안에서 끝날 때 씁니다. Swift 5.7부터는 `if let name = name`을 `if let name`으로 줄일 수 있습니다(SE-0345).

**2. guard let — 없으면 일찍 나갈 때.** 함수 초입에서 전제 조건을 검사하고, 통과하면 함수 끝까지 언래핑된 값을 쓰는 패턴입니다. 성공 경로가 들여쓰기 없이 쭉 흐르는 게 장점이라, 실무 함수 코드에서는 if let보다 자주 보입니다.

```swift
func register(email: String?) {
    guard let email else {
        print("이메일이 필요합니다")
        return
    }
    // 여기서부터 email은 String, 함수 끝까지 유효
    sendVerification(to: email)
}
```

**3. nil 병합 연산자 ?? — 기본값이 있을 때.** "없으면 이 값으로"가 한 줄로 끝납니다. `let title = inputTitle ?? "제목 없음"`.

**4. 옵셔널 체이닝 ?. — 없으면 그냥 통과시킬 때.** `user?.profile?.imageURL`처럼 중간에 nil이 나오면 전체가 nil로 단락됩니다. 결과 타입도 옵셔널이 된다는 점만 기억하면 됩니다.

**5. 강제 언래핑 ! — 없으면 죽어야 할 때.** 상자를 확인 없이 여는 문법이고, 비어 있으면 즉시 크래시입니다. "위험하니 절대 금지"로 알려져 있지만 더 정확한 기준은 이겁니다. nil이 나오는 게 프로그래머의 명백한 버그라서, 조용히 넘어가느니 크래시로 즉시 드러나는 게 나은 자리에만 쓴다. 예컨대 앱 번들에 반드시 포함된 리소스를 읽을 때가 그렇습니다. 반대로 네트워크 응답, 사용자 입력, 딕셔너리 조회처럼 nil이 정상 시나리오인 곳에 `!`를 쓰는 건 시한폭탄이 맞고요.

여기에 보너스로 하나 더. 옵셔널이 enum이라는 걸 알면 map도 이해됩니다. `imageURL.map { download($0) }`처럼, 상자를 열지 않고 내용물에 함수를 적용하는 도구도 있습니다. 값이 없으면 아무 일도 안 일어나고요.

## 암시적 언래핑 옵셔널 — 물음표 대신 느낌표가 붙은 타입

선언부에 느낌표가 붙는 `String!`도 있습니다. 암시적 언래핑 옵셔널(IUO)인데, "옵셔널이지만 쓸 때마다 자동으로 강제 언래핑되는" 타입입니다.

존재 이유는 초기화 타이밍 문제입니다. 대표적으로 스토리보드의 @IBOutlet이 그래요. 뷰컨트롤러가 생성되는 시점엔 아웃렛이 아직 연결 전이라 nil이지만, 화면이 뜬 뒤에는 반드시 값이 있습니다. 매번 언래핑하기엔 번거롭고 그렇다고 옵셔널이 아니라고 할 수도 없는 어중간한 상황을 위한 타협이죠.

실무 기준은 간단합니다. 프레임워크가 요구하는 자리(IBOutlet 등) 밖에서는 새로 만들지 않는 게 좋습니다. 초기화 순서 문제는 대부분 lazy나 의존성 주입으로 더 안전하게 풀 수 있어요.

## 옵셔널을 대하는 설계 감각

문법을 아는 것과 잘 쓰는 건 다른 문제라, 설계 차원의 기준 세 가지를 덧붙입니다.

**옵셔널을 만들지 않는 게 최선입니다.** 옵셔널은 "없을 수 있음"이 진짜일 때만 가치가 있습니다. 항상 값이 있는데 습관적으로 `?`를 붙이면, 모든 사용처에 무의미한 언래핑 의식만 퍼집니다. 프로퍼티를 선언할 때 "이게 nil인 순간이 실제로 존재하나"를 한 번 묻는 것만으로 코드가 꽤 달라져요.

**옵셔널은 경계에서 벗겨서 안쪽으로 들이지 않습니다.** 네트워크, 사용자 입력, 딕셔너리 조회 같은 바깥 경계에서는 옵셔널이 불가피합니다. 좋은 구조는 경계 함수에서 guard let으로 정리하고, 도메인 로직 안쪽에는 확정된 값만 흘려보내는 형태입니다. 안쪽 함수 시그니처가 옵셔널투성이라면 벗기는 위치가 너무 깊다는 신호예요.

**"없음"에 의미가 여러 개라면 옵셔널로 부족합니다.** nil이 "아직 로딩 전"인지 "로딩했는데 실패"인지 "원래 없음"인지 구분해야 한다면, 옵셔널 대신 의미를 담은 enum을 직접 만드는 게 맞습니다. 옵셔널이 enum이라는 걸 아는 순간, 같은 방식으로 내 상황에 맞는 enum을 설계할 수 있게 되는 거죠.

## 정리

- 옵셔널은 특별한 문법이 아니라 `case none`과 `case some(Wrapped)`를 가진 enum입니다. `String?`, `nil`, if let 전부 이 enum의 설탕입니다.
- 존재 이유는 "값이 없을 수 있음"을 타입으로 승격시켜, nil 체크를 컴파일러가 강제하게 만드는 것입니다.
- 언래핑은 자리가 정해져 있습니다. 잠깐 쓰면 if let, 조기 탈출은 guard let, 기본값은 ??, 통과시키려면 ?., 버그면 죽어야 할 자리만 !.
- 설계 감각: 불필요한 옵셔널을 만들지 않기, 경계에서 벗기기, "없음"의 의미가 여러 개면 전용 enum 만들기.

다음 편은 기초 시리즈 2편, 클로저입니다. 클로저가 왜 참조 타입인지, 캡처 리스트가 정확히 뭘 복사하는지, escaping이 왜 필요한지를 다룹니다.
